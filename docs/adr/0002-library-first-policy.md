# ADR-0002: 라이브러리 우선 채택 정책 + Minimum Cost 해석

- **Status**: Accepted
- **Date**: 2026-04-20
- **Scope**: 모든 `research.md` 단계의 외부 시스템 통신·유틸 선정

## Context

018 피처(Google Calendar 연동)의 research R2에서 "Google Calendar SDK 대신 `fetch` 자체 구현"을 제안하며 헌법 **Principle II — Minimum Cost**를 근거로 들었다. 사용자 리뷰에서 두 가지 해석 오류가 드러났다.

1. **Minimum Cost는 금전·인프라 비용 원칙**이다(별도 AI API 과금·유료 서비스 도입). npm 의존성은 그 원칙의 적용 범위가 아니다.
2. **직접 구현은 작성자 1명만 이해**한다. 라이브러리는 **문서·커뮤니티·표준 레퍼런스**가 함께 붙어 와서 팀원·미래 합류자의 러닝커브가 얕아진다. 1인 개발 + AI 보조 병렬 환경에서 유지 비용이 역으로 커진다.

동일 토론이 스펙 018에서 한 번 발생했으며, 외부 시스템 통합이 새로 들어올 때마다 반복될 가능성이 높다(#305 GCal, 차후 GCal→Trip 역방향, Slack/Discord 알림, Notion 연동 등). 판정 기준을 한 번에 못 박아 둔다.

## Decision

1. **라이브러리 사용이 기본값이다.** 다음 조건을 모두 만족하면 라이브러리 도입을 선택한다.
   - **정식 버전(>= 1.0)** 이다. 0.x 프로토타입은 지양.
   - **레퍼런스가 풍부**하다(공식 문서·Stack Overflow·블로그 등 검색 가능 자원).
   - **활발히 관리된다**(최근 12개월 내 릴리스 또는 보안 패치).
   - 가능하면 **공식 유지자**(해당 서비스 제공사, 표준 단체)가 관리한다.

2. **자체 구현을 선택하려면 증명 책임이 자체 구현 쪽에 있다.** 다음 중 하나 이상을 research.md에 명시한다.
   - 적절한 라이브러리가 존재하지 않음(조사 결과 포함).
   - 0.x 프로토 또는 관리 정지 상태만 존재.
   - 라이브러리가 요구 기능의 소수만 담고, 오히려 이를 감싸는 비용이 더 큼.
   - 보안·규제 상 외부 의존성 금지(해당 사례 없음).

3. **라이브러리 채택 근거를 1건당 1표로 기록한다.** research.md에 다음 열을 포함한 표를 둔다.

   | 항목 | 설명 |
   |---|---|
   | 패키지명 / 버전 | 예: `@googleapis/calendar` v14.2.0 |
   | 제공자 | Google / 커뮤니티 / 표준 단체 |
   | 최근 릴리스 시점 | `npm view … time.modified` 기준 |
   | 최초 등록 시점 | 오래된 라이브러리 vs 신생 비교용 |
   | 해결하는 문제 | 우리가 쓰는 기능 범위 |
   | 대안과의 비교 | 왜 이것을 골랐는가 |
   | 러닝커브 신호 | 공식 문서 존재, TypeScript 지원 등 |

4. **"Minimum Cost"(헌법 II) 해석**: 금전·인프라 비용 원칙이다. **오픈소스 npm 의존성 비용(번들 크기·학습 비용)에 확대 해석하지 않는다**. 번들 경량화는 필요할 때 별도 근거로 논의한다.

5. **지양 사례**(자동 기각).
   - 0.x 버전만 있는 라이브러리.
   - 최근 24개월 내 릴리스가 없는 라이브러리.
   - GitHub Archived 표시.
   - 라이선스가 상용·금지 카테고리(GPL 전파, SSPL 등 프로젝트 공개 계획과 상충).

## Consequences

- research.md 단계에서 외부 시스템 통신을 다루는 항목은 **라이브러리 평가 표**가 필수가 된다.
- 미래 합류자·AI 세션이 외부 API 코드에 들어왔을 때 공식 문서로 바로 맥락을 얻는다(혼자만의 구현체 해독 부담 제거).
- 자체 구현을 선택한다면 이유가 research에 남아 있어 3개월 뒤에도 의사결정 근거를 추적할 수 있다.
- PR 리뷰에서 "왜 라이브러리를 안 썼나?" 반복 토론이 사라진다.
- 라이선스·보안 체크는 여전히 팀(개발자 본인)이 책임진다. 기본값이 라이브러리라고 해서 검증 생략 금지.

## Alternatives Considered

- **"자체 구현 우선, 필요하면 라이브러리"** — 1인 개발 지속성에 부적합. 자체 코드가 누적될수록 유지 공수 기하급수. 기각.
- **"모든 선택 전 전수조사 필수"** — 의사결정 속도 저해. 기존 업계 1번 타자 라이브러리가 명확한 경우까지 과도한 절차. 기각.
- **"한 번에 폴더 단위로 의존성 자동 생성"** — 통제력 상실. 기각.
