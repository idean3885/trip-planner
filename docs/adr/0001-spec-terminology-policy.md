# ADR-0001: 스펙·문서 작성 용어 정책

- **Status**: Accepted
- **Date**: 2026-04-20
- **Scope**: `specs/**`, `docs/**`, `CHANGELOG.md`, 릴리스 노트, 이슈·PR 본문

## Context

trip-planner의 스펙은 WHAT/WHY를 담는 문서로, 구현 세부(플랜)와 분리되어 있다. 스펙의 독자는 작성자 본인뿐 아니라 비기술 이해관계자(기획·디자이너·추후 합류 협업자)도 포함한다.

피처 018(Google Calendar 연동, #305) 스펙 작성 중 다음 용어들이 축약되거나 영어 전문용어에 과도하게 의존한 것이 드러났다: "무회귀 유지", "옵트인", "export", "scope 동의/승격", "재로그인 루프", "primary/secondary", "ETag 상당". 이들은 도메인 배경 없이 읽는 독자에게 즉시 와닿지 않거나, 이해를 위해 추가 문맥을 요구했다.

동시에 MVP·CTA·ETag처럼 이미 실무에 정착한 용어까지 한국어로 풀어쓰면 오히려 군더더기가 생기고, 산업 표준(RFC 9110의 ETag 등)을 굳이 회피하게 된다.

용어 선택을 스펙마다 즉흥 판단하면 일관성이 무너지고, 각 PR 리뷰에서 반복 논쟁이 발생한다.

## Decision

1. **공학·개발자 전용 축약·은어는 한국어로 풀어쓴다.**
   - 예: `무회귀 유지` → "이번 변경으로 기존 경험이 깨지지 않는다"
   - 예: `옵트인` → "각자가 직접 실행했을 때에만"
   - 예: `재로그인 루프` → "로그인 → 동의 → 다시 로그인이 반복되는 현상"
   - 예: `scope 동의/승격` → "캘린더 쓰기에 필요한 권한 동의"
   - 예: `ETag 상당` 같은 "~상당" 수식은 쓰지 않고, 대상 용어 자체를 쓰거나 기능을 한국어로 서술한다.
2. **보편·기획·웹 표준 용어는 영어 그대로 쓴다.**
   - MVP, CTA, ETag, API, OAuth 등 실무·표준 어휘.
   - 판단 기준: "이 용어를 모르면 이 분야 실무 대화에 끼기 어려운가?" — 그렇다면 유지.
3. **주·부 관계를 나타낼 때는 `primary/secondary` 대신 `1순위/2순위`로 표기한다.**
   - 예: "구글 캘린더는 1순위 연결 경로, iCal은 2순위(기존 사용자용)".
4. **스펙 본문(`User Scenarios`, `Requirements`, `Success Criteria`, `Key Entities`)에는 구현 스택 이름(라이브러리·프레임워크명)을 노출하지 않는다.** `Assumptions` / `Dependencies` 섹션은 예외 — 여기서는 기술 의존성을 명시해야 한다.

## Consequences

- 스펙 독자층이 비기술 이해관계자까지 확장된다. 스펙 단독으로 맥락 이해가 가능해진다.
- 용어 선택 시 고민 지점이 줄어든다. 이 ADR이 기본값 역할을 한다.
- 후속 스펙(`018-gcal-integration` 이후 신규 피처)부터 적용한다. 기존 스펙(`013`~`017`)은 적용 유예 — 회귀적 소급 정리는 하지 않는다.
- PR 리뷰에서 "이 용어 왜 영어/한국어?" 같은 반복 논쟁이 사라진다.
- 용어 판단이 애매할 때는 이 ADR에 추가 예시를 보충하거나 새로운 ADR로 파생한다.

## Alternatives Considered

- **전부 영어 유지** — 개발자 친숙하지만 비기술 독자 배제. 스펙의 독자층 가정과 충돌. 기각.
- **전부 한국어 풀어쓰기** — MVP·CTA·ETag처럼 이미 보편화된 용어까지 풀면 군더더기가 생기고, 산업 표준 용어를 회피하게 됨. 기각.
- **별도 용어 사전(glossary) 관리** — 유지 비용 과다. 스펙·플랜·이슈가 각자 링크해야 함. 경량 ADR 한 장으로 충분. 기각.
- **스펙별 용어 재량** — 일관성 부재, 리뷰 반복 논쟁. 기각.
