# ADR-0008: Next.js 풀스택 단일 런타임 + Vercel 무료 티어

- **Status**: Accepted
- **Date**: 2026-06-03
- **Scope**: 앱 서버 런타임 선택, 배포 토폴로지, 데이터 접근 계층 구조

## Context

trip-planner는 1인 개발 + AI 보조 병렬 환경의 개인 포트폴리오 프로젝트다. 초기 v1은 마크다운 + Python MCP 구성이라 서버가 없었다. Phase 2(spec 004)에서 웹앱·DB·OAuth·동행자 협업으로 확장하면서 서버가 필요해졌다.

이 시점에 두 갈래가 있었다.

- **React SPA(서버 없음) + 별도 백엔드** — 프런트는 정적 호스팅, 데이터·인증은 독립 백엔드 서버가 담당.
- **Next.js 풀스택 단일 런타임** — 하나의 앱이 렌더링·API·인증을 모두 담당.

결정은 풀스택 단일 런타임으로 내려졌으나, 근거가 spec 004 plan에 흩어져 있고 최상위 결정으로 남아 있지 않다. 외부에 구조를 설명할 때(포트폴리오·면접) "왜 React만 쓰지 않고 서버를 도입했나", "왜 별도 백엔드 서버가 없나"가 반복 질문이 되므로 한 번에 명문화한다.

제약 조건:

- **운영 인력 1명.** 운영 표면(배포 대상·CI·모니터링·인증 인프라)이 늘수록 유지 비용이 직접 증가한다.
- **금전 비용 최소화**(헌법 II — Minimum Cost). Vercel 무료(Hobby) 티어 + Neon 무료 티어 안에서 동작해야 한다.
- **클라이언트가 둘**이다(웹 브라우저 + MCP/AI 에이전트). 두 경로가 같은 데이터 정본을 봐야 한다.

## Decision

1. **Next.js(App Router)를 풀스택 단일 런타임으로 쓴다.** 하나의 Next.js 앱이 (a) 서버 컴포넌트 SSR, (b) REST API Route, (c) 세션 가드 미들웨어, (d) Auth.js v5 OAuth 처리를 모두 담당한다. 별도 백엔드 서버(Express·NestJS 등)를 두지 않는다.

2. **DB 정본 접근은 Next.js 서버 한 곳으로 모은다.** 웹(서버 컴포넌트·API Route)과 MCP(Python)가 모두 이 서버를 경유해 Neon Postgres에 닿는다. MCP는 DB에 직접 붙지 않고 REST API(`/api/v2`)의 클라이언트로 동작한다.

3. **배포는 Vercel 무료 티어 + Git 브랜치 매핑으로 한다.** `main` → prod(trip.idean.me), `develop` → dev(dev.trip.idean.me). 별도 인프라 프로비저닝(컨테이너·로드밸런서·VPC)이 없다. 환경별 DB는 Neon 단일 호스트의 database 분리(#318)로 처리한다.

4. **데이터 접근 계층은 의도적으로 얇게 둔다.** 서비스/레포지토리 계층 없이 라우트·서버 컴포넌트가 Prisma를 직접 호출한다. 현재 규모에 맞춘 선택이며, 부채로 남기지 않도록 한계와 전환 신호를 함께 기록한다(아래 Consequences).

## Consequences

**얻는 것:**

- 배포 대상 1개, CI 파이프라인 1개, 인증 인프라 1개. 운영 표면이 최소다.
- 서버 컴포넌트가 Prisma를 직접 호출하므로 브라우저-서버 간 API 왕복·CORS·별도 토큰 교환이 없다. 페이지 데이터 경로가 짧다.
- 웹과 MCP가 같은 API/DB 정본을 공유하므로 데이터 일관성이 갈라지는 지점이 없다.
- Vercel·Neon 무료 티어 안에서 0원 운영이 된다.

**치르는 비용·한계:**

- 서비스 계층이 없어 비즈니스 로직 재사용이 어렵고, 테스트할 때 Prisma mock이 필요하다(`docs/ARCHITECTURE.md` "현재 한계" 표 참조).
- 서버 컴포넌트가 Prisma에 직접 결합돼, 훗날 API를 외부로 분리하려면 페이지 데이터 접근부를 함께 손봐야 한다.
- Vercel 무료 티어의 함수 실행·대역 한도에 묶인다. 한도를 넘으면 유료 전환 또는 셀프호스팅을 재검토해야 한다.

**전환 신호(이 결정을 다시 볼 때):**

- 동시 기여자가 2명 이상이 되거나, 같은 비즈니스 로직이 라우트 3곳 이상에서 반복될 때 → 서비스 계층 도입을 검토한다.
- 무료 티어 한도 초과가 반복될 때 → 배포 토폴로지를 재설계한다.

## Alternatives Considered

- **React SPA + 별도 백엔드(Express·NestJS)** — 정적 호스팅은 편하나 서버가 따로 필요해진다. 배포 대상 2개, CORS·세션·토큰 교환 인프라, OAuth 콜백 서버가 추가된다. 1인 운영 표면이 배가되어 기각.
- **Next.js + 독립 API 서버 분리** — API를 별도 서비스로 빼면 경계는 깨끗해지나 현재 규모에서는 운영·배포 비용이 이득보다 크다. 전환 신호 도달 시 재검토.
- **별도 BaaS 백엔드(Supabase Edge Functions 등)** — 외부 의존·대시보드·과금 표면이 하나 더 생긴다. 단일 런타임 원칙과 상충해 기각.
- **서비스/레포지토리 계층 즉시 도입** — 1인·현재 규모에서는 추상화 선투자 대비 이득이 작다. 라우트 직접 호출로 시작하고, 전환 신호에 따라 도입한다.
