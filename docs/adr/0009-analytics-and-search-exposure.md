# ADR 0009: 사용자 분석은 GA4, 검색 노출은 최소(동적 앱)

- **Status**: Accepted (2026-06-05, spec 057)
- **Context**: 이슈 [#767](https://github.com/idean3885/trip-planner/issues/767), 마일스톤 #58
- **Related**: ADR 0002 library-first-policy, ADR 0001 spec-terminology-policy

## Context

무료 오픈 후 "어떤 사용자가 어떻게 쓰는지"를 보는 분석 기반이 없었다. 동시에 검색 노출(GSC) 등록도 같이 검토했다.

두 가지 판단이 필요했다.

1. **사용 행태 분석 도구 선택** — 후보: GA4, PostHog, Amplitude, Vercel Web Analytics.
2. **검색 노출(GSC)의 비중** — 정적 콘텐츠 사이트라면 SEO·색인이 중요하지만, 이 앱은 가치의 대부분이 로그인 뒤 동적 데이터에 있다.

도구 비교(무료 기준, 2026-06 웹 확인):

- **GA4** — 표준 무료(360만 유료). 시장에서 가장 보편적. 세션 리플레이 없음, 제품 행동 분석 깊이는 약함.
- **PostHog** — 무료 100만 이벤트/월 + 세션 리플레이 + 기능 플래그. 행동 분석 강함. 단 한국 채용·협업에서 대세는 아님.
- **Amplitude** — 제품 분석 직군 표준. 무료 10K MTU(빡빡).
- **Vercel Web Analytics** — 익명 집계만, 개인 식별 불가.

## Decision

**(1) 분석 도구는 GA4를 단일 채택한다.**

- **대세(시장 표준) 우선.** 차별화를 위해 비주류(PostHog 등)를 고르지 않는다. 대세를 갖추지 않은 상태에서 비주류로 차별화하는 것은 협업·채용에서 "왜 대세를 안 썼나"라는 해명 부담이 된다. 도구 선택의 기준은 차별성이 아니라 표준 적합성이다.
- GA4는 협업에서 통용되고 학습 가치(이벤트 택소노미·전환·User-ID·BigQuery)가 크다.
- 통합은 자체 스니펫 대신 공식 `@next/third-parties`를 쓴다(ADR 0002 library-first).
- 깊은 세션 분석이 필요해지면 PostHog/Amplitude를 후속 검토한다(지금은 보류).

**(2) 검색 노출(GSC)은 최소만 둔다.**

- 앱 본체는 로그인 뒤 동적 데이터라 검색 색인 가치가 낮다. 공개 페이지(랜딩·소개·문서)만 sitemap·색인하고, 로그인 뒤 경로는 robots disallow + meta noindex로 제외한다.
- "정적 SEO"가 아니라 "동적 앱은 사용 행태 분석이 본질"이라는 판단. GSC는 거의 비용이 없어 열어두되 비중은 낮다.

세부:

- 측정 ID·검색 소유 확인값은 환경변수. 미설정 시 분석·검증 태그가 붙지 않고 앱은 정상 동작한다.
- 핵심 전환 이벤트는 발생 지점이 명확한 `trip_created`·`calendar_import`부터. 가입 정밀 집계는 서버 측 작업이 필요해 후속.
- `user_id`에는 내부 식별자만(PII 미전송). 동의 배너는 범위 밖(개인정보 고지로 시작).

## Consequences

- 협업·채용에서 통하는 표준 분석 경험을 쌓는다. 도구 인지도의 약점은 "왜 이 선택인가"를 본 ADR·블로그로 설명해 상쇄한다.
- 행동 분석 깊이(세션 리플레이 등)는 GA4로 한계가 있어, 필요 시 후속 도구를 얹는다.
- 검색 유입을 본격적으로 노리려면 후속에서 공개 콘텐츠·sitemap을 확장해야 한다.
