# ADR 0010: 테스트 커버리지 임계값 100% → 80/75 완화

- **Status**: Accepted (2026-06-05, 이슈 #778, 마일스톤 #60)
- **Context**: 이슈 [#778](https://github.com/idean3885/trip-planner/issues/778)
- **Related**: ADR 0002 library-first-policy

## Context

`vitest.config.ts` 의 커버리지 임계값이 statements/functions/lines 100%·branches 95% 였다. include 범위는 활동 핵심 파일(activity API 라우트 + `Activity*.tsx`)로 좁혀 둔, 의도가 분명한 설정이었다.

운영하며 두 가지가 드러났다.

1. **100% 의 마지막 1~2% 가 비싸다.** `onClick={(e) => e.stopPropagation()}` 한 줄, null 방어 분기, 도달 불가 패스 등 사소한 경로를 커버하려고 테스트를 쓰거나 `/* c8 ignore */` 주석을 다는 공수가 반복됐다. 새 필드 하나를 더할 때마다(예: `url`) 트리거됐다.
2. **100% 가 실제 안정성을 담보하지 못했다.** 커버리지 100% 상태에서도 동작 실패가 났고, 대부분 외부 연동(Google·Apple 캘린더)·시각/터치처럼 단위 테스트가 닿지 않는 영역이었다.

## Decision

커버리지 임계값을 **statements 80 / branches 75 / functions 80 / lines 80** 으로 낮춘다. include 범위(활동 핵심 파일)는 그대로 둔다.

- 테스트는 **TDD 명세**(무엇을 보장하는가)로서 작성한다. 숫자 100% 달성이 목적이 아니다.
- 시각·터치·외부 연동은 코드 리뷰와 **실기기/프리뷰 검증**으로 잡는다. 단위 테스트로 대체하지 않는다.

## Alternatives Considered

- **100% 유지 + `c8 ignore` 주석** — 현행. 사소한 경로마다 주석·테스트 공수가 누적돼 개발 속도를 떨어뜨린다. 기각.
- **90/85** — 여전히 거의 모든 경로를 요구해 1번 문제(마지막 1~2%)가 남는다. 기각.
- **임계값 제거(리포트만)** — 회귀 안전망이 약해진다. 핵심 파일에는 최소선을 두는 편이 낫다. 기각.

## Consequences

- 핵심 활동 파일의 실제 커버리지는 ~99% 라 80/75 를 여유 있게 통과한다. 즉 당장 커버리지가 떨어지는 게 아니라, **떨어뜨릴 자유가 생긴다**(사소한 경로 추적 면제).
- 회귀 방지의 무게중심이 "줄 단위 커버리지"에서 "의미 있는 명세 테스트 + 리뷰 + 실기기"로 이동한다.
- 되돌리려면 `vitest.config.ts` thresholds 를 다시 올리면 된다.
