# Feature Specification: 여행 종합정보 — 목록 금액·인원 + 개요

**Feature Branch**: `063-trip-overview`  
**Created**: 2026-06-25  
**Status**: Draft  
**Input**: User description: "여행 목록 카드에 사용 금액·인원 표시 + 여행 개요(설명·요약) 보기"

## Clarifications

1. **목록 카드 사용 금액 = 지출 합계(참고 원화 병기)** — 카드에 그 여행의 지출 합계를 통화별로 보이고, 가능하면 원화 근사 총액을 참고로 함께 보인다(기존 합산·환율 재사용). 지출이 없으면 금액 줄을 생략한다.
2. **인원 = 동행자 수** — 그 여행에 참여한 동행자(주인·호스트·게스트) 총원 수를 카드와 개요에 표기한다.
3. **개요는 상세 화면의 한 섹션** — 일정(캘린더)이 메인이라는 전제는 유지한다. 새 페이지를 만들지 않고, 여행 상세 화면 안에 "여행 개요"(기간·인원·총 사용 금액·설명)를 항상 보이는 한 자리로 둔다. 설명이 없으면 설명만 비우고 종합정보(기간·인원·총액)는 그대로 보인다.
4. **읽기 전용 표시** — 본 피처는 보여주기만 한다. 금액·인원·설명을 편집하는 새 입력은 추가하지 않는다(설명 편집 등은 기존 경로 그대로).

## Metatag Conventions *(normative, inherited from PR #204)*

본 피처의 tasks.md·plan.md는 네 종 메타태그를 통해 후속 자동 검증과 연결된다:

- `[artifact: <path>|<path>::<symbol>]` — 산출 파일 식별자(drift 감사 기준)
- `[why: <short-tag>]` — 추적 그룹 키(plan↔tasks 커버리지·이슈 합산)
- `[multi-step: N]` — plan bullet이 다단 작업일 때 최소 매핑 태스크 수(N ≥ 2)
- `[migration-type: schema-only | data-migration]` — 마이그레이션 산출물 구분

형식 검증은 `.specify/scripts/bash/validate-metatag-format.sh`가 수행한다. 의미 검증은 각 US의 validator가 담당한다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 여행 목록 카드의 사용 금액·인원 (Priority: P1)

여행자가 여행 목록을 볼 때, 각 카드에서 기간·일수·역할에 더해 그 여행에서 **얼마를 썼고 몇 명이 함께하는지**를 바로 본다. 여러 여행을 비교하거나 지난 여행을 돌아볼 때 카드만으로 규모가 잡힌다.

**Why this priority**: 목록은 가장 자주 보는 화면이고, 금액·인원은 여행을 식별·회상하는 핵심 단서다.

**Independent Test**: 지출과 동행자가 있는 여행이 목록에 있을 때, 카드에 지출 합계와 인원수가 보이는지로 단독 검증된다.

**Acceptance Scenarios**:

1. **Given** 지출이 있는 여행이 목록에 있을 때, **When** 목록을 보면, **Then** 카드에 그 여행의 지출 합계(통화별, 가능 시 원화 근사 참고)가 보인다.
2. **Given** 동행자가 여러 명인 여행에서, **When** 목록을 보면, **Then** 카드에 인원수가 보인다.
3. **Given** 지출이 하나도 없는 여행에서, **When** 목록을 보면, **Then** 금액 줄은 생략되고 나머지 정보는 그대로 보인다.

---

### User Story 2 - 여행 개요(종합정보) 보기 (Priority: P1)

여행자가 여행을 열면, 일정(캘린더) 중심 화면 안에서 그 여행이 **어떤 여행인지**를 한 자리에서 본다 — 기간, 인원, 총 사용 금액, 그리고 설명. 설명이 비어 있어도 종합정보는 보인다.

**Why this priority**: "이 여행이 무엇인지"를 잡아주는 맥락. 지금은 설명이 없으면 아무 요약도 안 보여 맥락이 약하다.

**Independent Test**: 여행 상세를 열었을 때, 일정과 별개로 기간·인원·총액(+설명)을 담은 개요가 보이는지로 검증된다.

**Acceptance Scenarios**:

1. **Given** 설명·지출·동행자가 있는 여행에서, **When** 상세를 열면, **Then** 기간·인원·총 사용 금액·설명을 담은 개요가 한 자리에 보인다.
2. **Given** 설명이 비어 있는 여행에서, **When** 상세를 열면, **Then** 설명 자리는 비고 기간·인원·총액은 그대로 보인다.
3. **Given** 어떤 여행이든, **When** 상세를 열면, **Then** 일정(캘린더)은 여전히 메인으로 보인다(개요가 일정을 대체하지 않는다).

---

### Edge Cases

- 지출이 여러 통화로 섞여 있으면? → 카드·개요 모두 통화별로 보이고, 원화 근사 총합을 참고로 합쳐 보인다(기존 합산 규칙 재사용, 미확보분 제외·참고 표기).
- 인원이 1명(혼자)인 여행? → "1명"으로 그대로 표기한다.
- 일정·지출이 0건인 새 여행? → 금액·일수는 비거나 0으로, 인원·기간(미정)은 그대로 보인다.
- 설명이 매우 길면? → 개요에서 과하지 않게 보여준다(접기/요약 등 표시 정책은 구현에서 결정).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 여행 목록의 각 카드는 그 여행의 지출 합계를 통화별로 보여야 하며, 가능하면 원화 근사 총액을 참고로 함께 보여야 한다.
- **FR-002**: 여행 목록의 각 카드는 그 여행의 참여 인원수를 보여야 한다.
- **FR-003**: 지출이 없는 여행 카드는 금액 표기를 생략하되 나머지 정보는 유지해야 한다.
- **FR-004**: 여행 상세 화면은 기간·인원·총 사용 금액·설명을 담은 "여행 개요"를 한 자리에 항상 보여야 한다.
- **FR-005**: 설명이 비어 있어도 개요의 종합정보(기간·인원·총액)는 보여야 한다.
- **FR-006**: 개요는 일정(캘린더) 메인을 대체하지 않아야 한다(보조 정보로 공존).
- **FR-007**: 총 사용 금액·원화 근사는 기존 지출·환율 합산 규칙(현화/원화 참고, 미확보분 제외)을 그대로 따라야 한다.
- **FR-008**: 인원수는 그 여행의 동행자(주인·호스트·게스트) 총원을 센 값이어야 한다.
- **FR-009**: 목록·개요의 금액·인원은 추가 입력 없이 기존 데이터에서 산출되어야 한다(읽기 전용).
- **FR-010**: 목록에서 여러 여행의 금액·인원 산출이 화면 표시를 눈에 띄게 지연시키지 않아야 한다.

### Key Entities *(include if feature involves data)*

- **여행 요약(Trip Summary)**: 한 여행의 표시용 집계 — 기간, 일수, 인원수, 지출 합계(통화별 + 원화 근사 참고). 기존 여행·일정·지출·동행자에서 파생되는 읽기 모델(새 저장 없음).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 여행자는 목록 화면만으로 각 여행의 사용 금액과 인원을 추가 클릭 없이 확인한다.
- **SC-002**: 여행자는 여행 상세를 열어 한 자리에서 기간·인원·총액·설명을 확인한다.
- **SC-003**: 설명이 없는 여행에서도 종합정보(기간·인원·총액)는 100% 표시된다.
- **SC-004**: 금액·인원 표기 추가가 목록·상세의 체감 로딩을 늘리지 않는다(기존과 동등 수준).
- **SC-005**: 일정(캘린더)은 상세에서 여전히 주 화면으로 유지된다.

## Assumptions

- 지출·환율 합산(현화/원화 참고)과 동행자(역할) 데이터는 이미 존재하며 그대로 재사용한다.
- 원화 근사는 참고용이며 정산 정확값이 아니다(기존 정책 승계).
- 한 사용자가 가진 여행 수는 소수(수~수십)다.

## Out of Scope

- 새로운 결제·정산 기능, 지출 입력 변경.
- 동행자 권한·초대 UI 변경(인원은 표시만).
- 별도의 여행 개요 전용 페이지(상세 안 섹션으로 충분).
- 설명 편집 경험 변경(기존 경로 유지).
