# Phase 0 Research: 모바일 트립 상세 인터랙션 개선

시각·터치 거동이 핵심이라 각 결정은 "방향 + 실기기 검증으로 확정" 형태로 둔다. 로컬(os=linux)에서 모바일 터치·스크롤을 그대로 재현할 수 없으므로, 구현 중 실기기 스크린샷/녹화로 미세 조정한다(`feedback_visual_bug_real_device`).

## 1. 캘린더 sticky 경계 스크롤 양방향 1차 정지 [why: scroll-snap-stage]

**현황**: 모바일은 페이지(window) 스크롤 위에 캘린더가 `sticky top-0`으로 얹혀 있고, 그 아래 일정 캐러셀이 일반 흐름으로 이어진다. 캘린더 sticky 진입/해제 경계에 멈춤 지점이 없어 한 번의 스크롤이 캘린더 영역과 일정을 한꺼번에 가로지른다.

**Decision**: CSS scroll-snap으로 경계 정지를 만든다. 스크롤 컨테이너에 `scroll-snap-type: y proximity`를 주고, (a) 페이지 최상단(캘린더가 펼쳐진 상태)과 (b) 일정 섹션 상단(캘린더가 고정된 상태) 두 지점에 snap 정렬점을 둔다. proximity로 두어 일정 목록 내부의 자유 스크롤을 방해하지 않는다.

**Rationale**: 스크롤 위치 제어를 JS 스크롤 이벤트로 직접 하면 관성·러버밴딩과 싸우게 되고 프레임이 끊긴다. scroll-snap은 브라우저 합성 단계에서 처리돼 부드럽고, proximity는 "가까울 때만 끌어당김"이라 일정 목록 중간 스크롤을 막지 않는다. 위 방향 해제(FR-002)는 일정 상단 snap 지점이 자연히 충족한다 — 일정이 최상단에 닿아야 그 위(캘린더 펼침) 지점으로 넘어간다.

**Alternatives considered**:
- 일정 목록을 독립 `overflow-y` 컨테이너로 분리: 캘린더 경계가 자연 정지가 되지만, 큰 인터랙션 영역에 스크롤 컨테이너를 두면 #649(touch-action으로 세로 스크롤 차단) 류의 회귀 위험이 크다. 보류.
- JS 스크롤 스냅(scrollTo 강제): 관성과 충돌·끊김. 거부.
- mandatory snap: 일정 목록 중간에서도 강제로 끌어당겨 읽기를 방해. 거부(proximity 채택).

**확정 기준(실기기)**: 아래로 천천히/빠르게 스크롤할 때 캘린더 경계에서 한 번 정지하고, 일정 목록 중간 스크롤은 끊기지 않는다. 위로 올릴 때 일정 최상단 도달 전 캘린더가 따라 올라가지 않는다.

## 2. 미로딩 날짜 스와이프 즉시 이동 [why: swipe-instant]

**현황**: `SwipeCarousel`은 항상 3슬라이드(-1/0/+1)를 렌더하고, embla `settle`(스냅 모션 종료) 이벤트에서 `onCommit` → `setSelectedDate`로 기준 날짜를 옮긴 뒤 `reInit`+`scrollTo(1, true)`로 가운데 복귀한다(#657·#661·#665). 다음 날짜 활동이 캐시에 없으면 `DayActivitiesPane`이 스켈레톤을 그려야 하나, 실측상 로딩 완료 전에는 스와이프가 가운데로 튕겨 돌아온다.

**Decision**: 스와이프 커밋(날짜 이동)을 데이터 도착과 완전히 분리한다. 슬라이드 렌더는 이미 캐시 참조(`activities ?? null` → 스켈레톤)로 데이터와 분리돼 있으므로, 커밋 경로가 데이터/리렌더에 막히지 않는지 진단해 끊는다. 구체적으로 ① 핍 슬라이드(±1)가 데이터 유무와 무관하게 항상 같은 구조(스켈레톤 포함)로 렌더되는지, ② 미로딩으로 인한 리렌더가 `settle` 직전 embla 상태를 흔들어 커밋이 무시되는지를 확인하고, 커밋 트리거를 데이터 의존 없는 경로로 고정한다.

**Rationale**: 사용자 조작(스와이프)은 데이터 가용성과 독립이어야 한다. 윈도우 로딩(#669)이 이미 미로딩 날짜를 스켈레톤으로 표현하도록 설계됐으므로, 막힘은 "표현"이 아니라 "커밋 트리거"에 있다.

**Alternatives considered**:
- 커밋을 `settle`→`select`로 되돌림: #661 끊김 회귀 위험. 우선 데이터 분리로 해결하고, 불가피할 때만 재검토.
- 다음 날짜를 미리 동기 prefetch 후 이동: 조작을 네트워크에 묶는 것이라 요구(즉시 이동)와 정면 충돌. 거부.

**확정 기준(실기기)**: 한 번도 안 연 먼 날짜로 연속 스와이프해도 매번 즉시 넘어가고, 미로딩 구간은 스켈레톤으로 보였다가 도착 시 채워진다.

## 3. 스와이프-캘린더 강조 동시 갱신 [why: swipe-sync-highlight]

**현황**: 캘린더 선택 강조는 `selectedDate`를 따른다. 커밋이 `settle`(모션 종료) 뒤라, 일정 슬라이드가 먼저 옮겨 보이고 캘린더 강조가 한 박자 늦게 갱신된다.

**Decision**: 캘린더 강조를 일정 이동과 같은 타이밍에 맞춘다. 강조 갱신은 경량(클래스 토글)이라 커밋 시점에 함께 반영하면 되고, 끊김 유발 요인(무거운 일정 재렌더)과는 성격이 다르다. 커밋 한 곳에서 일정 기준일과 캘린더 강조를 같은 상태 업데이트로 묶어, 둘이 서로 다른 날짜를 가리키는 중간 프레임을 없앤다.

**Rationale**: #661이 무거운 재렌더를 settle 뒤로 미룬 것은 옳다. 다만 캘린더 강조는 가볍고, 사용자가 보는 "일정=캘린더 같은 날짜" 일관성이 더 중요하다. 강조와 일정 기준일이 같은 상태에서 파생되면 타이밍이 갈라질 수 없다.

**Alternatives considered**:
- 강조를 select(드래그 중)에서 미리 갱신: 드래그를 되돌리면 강조가 잘못 남는다. 커밋 시점 동기화가 안전.

## 4. 일정 섹션 상단 헤더 제거 [why: header-removal]

**Decision**: `DayActivitiesPane`의 날짜 `CardTitle`(및 헤더 영역)을 제거한다. 선택 날짜는 캘린더 강조로 충분하고, 활동 개수(N)는 표시 가치가 낮다(spec Clarification 1). 빈 날짜의 "일정 없음 + 추가" 안내는 본문에 유지한다.

**Rationale**: 중복 정보 제거 + 화면 간결화. 동작 변경 없음.

## 5. '자세히' 버튼 디자인 토큰화 [why: detail-button-token]

**현황**: `variant="ghost"` + `text-muted-foreground`에 `Ellipsis`(⋯) 아이콘 + "자세히" 텍스트. 버튼이 흐릿한 글자처럼 보인다.

**Decision**: shadcn 버튼 토큰을 따르는 외형(예: `outline`/`secondary` 등 토큰 변형)으로 바꾸고 `Ellipsis` 아이콘을 제거한다. 클릭 시 여행 정보(동행자·외부 캘린더 동기화) Dialog를 여는 동작은 유지한다. 구체 변형·크기는 디자인 토큰 셋과 주변 버튼 일관성에 맞춰 구현 시 확정한다.

**Rationale**: 버튼 인지성 확보 + 군더더기 아이콘 제거. 디자인 토큰 일관성(spec FR-007).
