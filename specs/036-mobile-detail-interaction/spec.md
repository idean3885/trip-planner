# Feature Specification: 모바일 트립 상세 인터랙션 개선

**Feature Branch**: `036-mobile-detail-interaction`  
**Created**: 2026-05-31  
**Status**: Draft  
**Input**: 모바일(<1024px) 트립 상세 화면의 스크롤·스와이프·헤더 동작 개선 (이슈 #681·#682·#683·#684)

## Clarifications *(optional, add when drafting raises ambiguity)*

1. **일정 섹션 상단 헤더 처리** — 헤더를 두지 않는 방향으로 결정한다. 선택 날짜는 캘린더에 이미 강조 표시되므로 날짜 표기는 중복이고, 활동 개수(N)는 사용자에게 의미가 약하다. 헤더 자체를 제거해 일정 목록이 곧장 보이게 한다.
2. **스크롤 1차 정지의 의미** — 캘린더가 화면 상단에 고정(sticky)되는 경계를 스크롤이 한 번 멈추는 지점으로 삼는다. 캘린더 영역 보기 ↔ 일정 목록 보기 사이를 단계로 나눠, 한 번의 스크롤 제스처가 두 영역을 한꺼번에 가로지르지 않게 한다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 캘린더 고정 경계에서 스크롤이 한 번 멈춘다 (Priority: P1)

모바일에서 트립 상세를 볼 때, 사용자는 위쪽 캘린더에서 날짜를 고른 뒤 아래 일정 목록을 본다. 페이지를 아래로 내리면 캘린더가 화면 상단에 고정되는데, 이 고정 경계에서 스크롤이 한 번 멈춰 사용자가 "지금 캘린더가 고정됐다"를 인지하고 이어서 일정만 스크롤하게 한다. 반대로 위로 올릴 때는 일정이 맨 위에 닿은 다음에야 캘린더 고정이 풀려 위로 올라간다.

**Why this priority**: 캘린더 고정과 일정 스크롤이 한 제스처로 한꺼번에 흘러 사용자가 의도한 위치에 멈추기 어렵다. 화면의 두 영역을 단계로 나누는 가장 핵심적인 인터랙션이다.

**Independent Test**: 모바일 폭에서 일정이 여러 개인 날짜를 열고, 아래로 천천히 스크롤하면 캘린더 고정 경계에서 한 번 멈추는지, 위로 올릴 때 일정 최상단에 닿기 전에는 캘린더가 따라 올라가지 않는지로 확인한다.

**Acceptance Scenarios**:

1. **Given** 일정이 화면보다 긴 날짜를 보는 중, **When** 페이지를 아래로 스크롤한다, **Then** 캘린더가 상단에 고정되는 경계에서 스크롤이 한 번 멈추고, 이어진 스크롤은 일정 목록만 움직인다.
2. **Given** 일정 목록을 아래까지 내려 캘린더가 상단 고정된 상태, **When** 위로 스크롤한다, **Then** 일정이 맨 위에 닿을 때까지는 일정만 올라가고, 맨 위에 닿은 뒤에야 캘린더 고정이 풀려 위로 올라간다.

---

### User Story 2 - 로딩 중인 날짜로도 스와이프가 즉시 넘어간다 (Priority: P1)

선택 날짜의 일정을 좌우로 스와이프해 앞뒤 날짜로 이동할 때, 다음 날짜의 일정이 아직 받아지지 않았더라도 스와이프는 즉시 넘어가야 한다. 아직 받아지지 않은 날짜는 자리표시(스켈레톤)로 보이다가 일정이 도착하면 채워진다.

**Why this priority**: 현재는 일정이 도착하기 전에 스와이프가 가운데로 튕겨 되돌아오고, 로딩이 끝난 뒤에야 넘어간다. 스와이프 이동이 데이터 도착에 묶여 있어 조작이 막힌 것처럼 느껴진다.

**Independent Test**: 일정이 많은 트립에서 아직 열어본 적 없는 먼 날짜 방향으로 빠르게 연속 스와이프해, 매 스와이프가 즉시 넘어가고 미로딩 날짜는 스켈레톤으로 보이는지로 확인한다.

**Acceptance Scenarios**:

1. **Given** 다음 날짜의 일정이 아직 받아지지 않은 상태, **When** 다음 날짜로 스와이프한다, **Then** 즉시 다음 날짜로 넘어가며 일정 영역은 스켈레톤으로 표시된다.
2. **Given** 미로딩 날짜로 넘어가 스켈레톤이 보이는 상태, **When** 일정이 도착한다, **Then** 스켈레톤이 실제 일정으로 바뀐다.

---

### User Story 3 - 스와이프와 캘린더 날짜 강조가 같이 바뀐다 (Priority: P2)

일정을 좌우로 스와이프해 날짜를 옮기면, 위쪽 캘린더의 선택 날짜 강조도 같은 타이밍에 함께 바뀐다.

**Why this priority**: 현재는 일정이 먼저 넘어가고 캘린더 강조가 한 박자 늦게 따라와, 잠시 일정과 캘린더가 다른 날짜를 가리킨다. 어긋남이 거슬리지만 이동 자체는 되므로 P2다.

**Independent Test**: 일정을 한 칸 스와이프하고, 캘린더의 선택 강조가 일정과 동시에 새 날짜로 옮겨가는지로 확인한다.

**Acceptance Scenarios**:

1. **Given** 특정 날짜의 일정을 보는 중, **When** 옆 날짜로 스와이프해 정착한다, **Then** 일정과 캘린더 선택 강조가 같은 날짜를 같은 타이밍에 가리킨다.

---

### User Story 4 - 일정 섹션 상단의 중복 날짜 헤더를 없앤다 (Priority: P2)

일정 섹션 맨 위에 표시되던 날짜 헤더를 제거한다. 선택 날짜는 캘린더에 이미 강조돼 있어 같은 정보를 두 번 보일 필요가 없다.

**Why this priority**: 중복 정보 제거로 화면을 간결하게 한다. 동작 결함은 아니므로 P2다.

**Independent Test**: 날짜를 선택했을 때 일정 목록 위에 별도 날짜 헤더가 보이지 않고, 선택 날짜는 캘린더에서만 확인되는지로 검증한다.

**Acceptance Scenarios**:

1. **Given** 임의 날짜를 선택한 상태, **When** 일정 섹션을 본다, **Then** 일정 위에 날짜·활동 개수 헤더가 표시되지 않고 일정 목록이 바로 보인다.

---

### User Story 5 - '자세히' 버튼을 버튼답게 정비한다 (Priority: P3)

여행 정보(동행자·외부 캘린더 동기화)를 여는 '자세히' 버튼을, 디자인 토큰을 따르는 버튼 외형으로 바꾸고 앞의 점 세 개(⋯) 아이콘을 없앤다.

**Why this priority**: 현재는 흐릿한 글자처럼 보여 버튼인지 인지하기 어렵다. 외형 정비라 P3다.

**Independent Test**: 모바일 트립 상세 상단에서 '자세히'가 버튼 외형으로 보이고 점 세 개 아이콘 없이 표시되며, 눌렀을 때 여행 정보가 열리는지로 확인한다.

**Acceptance Scenarios**:

1. **Given** 모바일 트립 상세 화면, **When** 상단을 본다, **Then** '자세히'가 디자인 토큰을 따르는 버튼 외형으로 보이고 점 세 개 아이콘이 없다.
2. **Given** '자세히' 버튼, **When** 누른다, **Then** 동행자·외부 캘린더 동기화를 담은 여행 정보가 열린다.

---

### Edge Cases

- 여행의 첫 날·마지막 날에서 더 이전·이후로 스와이프하면 어떻게 되는가? (기간 밖 방향으로는 넘어가지 않거나 가운데로 복귀)
- 일정이 화면 높이보다 짧아 스크롤이 필요 없는 날짜에서 스크롤 1차 정지는 어떻게 동작하는가? (정지 경계가 사실상 즉시 충족되어 부자연스러운 멈춤이 없어야 함)
- 미로딩 날짜로 연속 스와이프 중 데이터가 늦게 도착하면 강조·내용이 엇갈리지 않는가?

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 모바일 트립 상세에서 아래로 스크롤 시, 캘린더가 화면 상단에 고정되는 경계에서 스크롤이 한 번 멈춰야 한다. 이어진 스크롤은 일정 목록만 움직인다.
- **FR-002**: 위로 스크롤 시, 일정 목록이 맨 위에 닿은 뒤에야 캘린더 고정이 풀려 위로 올라가야 한다. 일정 목록 중간을 보던 상태에서 캘린더가 곧장 따라 올라가지 않는다.
- **FR-003**: 일정 좌우 스와이프는 다음 날짜의 일정 로딩 여부와 무관하게 항상 즉시 넘어가야 한다.
- **FR-004**: 아직 일정이 받아지지 않은 날짜로 넘어가면 일정 영역을 자리표시(스켈레톤)로 보이고, 일정이 도착하면 실제 일정으로 채워야 한다.
- **FR-005**: 일정 스와이프로 날짜가 바뀌면 캘린더의 선택 날짜 강조도 같은 타이밍에 함께 갱신되어야 한다.
- **FR-006**: 일정 섹션 상단의 날짜·활동 개수 헤더를 제거해야 한다.
- **FR-007**: '자세히' 버튼은 디자인 토큰을 따르는 버튼 외형으로 표시하고, 점 세 개(⋯) 아이콘을 제거해야 한다. 버튼을 누르면 동행자·외부 캘린더 동기화를 담은 여행 정보가 열려야 한다.
- **FR-008**: 위 변경은 모바일(<1024px)에만 적용하고, 데스크탑(≥1024px) 2분할 레이아웃의 동작은 바꾸지 않는다.

### Key Entities

본 피처는 표시·인터랙션 동작만 다루며 영속 데이터 스키마 변경이 없다. (선택 날짜·일정 목록·로딩 상태는 기존 화면 상태를 그대로 사용)

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 아래로 스크롤할 때 캘린더 고정 경계에서 스크롤이 한 번 멈추고, 한 번의 스크롤 제스처가 캘린더와 일정을 한꺼번에 가로지르지 않는다.
- **SC-002**: 위로 스크롤할 때 일정이 최상단에 닿기 전에는 캘린더가 따라 올라가지 않는다.
- **SC-003**: 미로딩 날짜로 스와이프해도 항상 즉시 넘어가며, 데이터 도착을 기다리느라 가운데로 되돌아오는 경우가 없다.
- **SC-004**: 스와이프로 날짜가 바뀌는 순간 일정과 캘린더 강조가 같은 날짜를 가리킨다(둘이 서로 다른 날짜를 가리키는 중간 구간이 보이지 않는다).
- **SC-005**: 일정 섹션 위에 날짜·개수 헤더가 더는 보이지 않는다.
- **SC-006**: '자세히'가 버튼 외형으로 인지되고 점 세 개 아이콘 없이 표시되며, 눌러서 여행 정보를 연다.
- **SC-007**: 데스크탑 2분할 레이아웃의 스크롤·표시 동작에 변화가 없다.
