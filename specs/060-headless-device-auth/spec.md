## Clarifications *(optional, add when drafting raises ambiguity)*

1. **인터렉션 주체** — 인증 흐름의 개시·진행·토큰 수신·재시도는 **소비자(에이전트·CLI·런타임)가 자동으로 끌고간다**. 사람에게 수기 토큰 발급·복사·붙여넣기를 요구하지 않는다. 사람이 하는 단계는 "자기 기기에서 로그인 승인" 하나뿐이다(신원은 사람만 보유하므로 기술적으로 불가피).
2. **방식 = device code 흐름** — 브라우저와 토큰 수신자가 같은 기기일 필요가 없는 방식(OAuth 2.0 Device Authorization Grant)을 채택한다. 소비자가 코드를 발급받아 사람에게 "이 주소를 열고 승인하라"는 안내(또는 바로 열리는 링크)를 제시하고, 승인 완료를 스스로 감지해 토큰을 받는다. loopback(127.0.0.1 콜백) 의존을 제거한다.
3. **발급 토큰 정책 승계** — 이 경로로 발급되는 토큰은 기존 자동 발급 토큰과 동일하게 단기 만료(기본 30일)를 가진다. 만료 시 같은 흐름으로 재인증한다. 토큰 모델(폐기·목록·만료) 자체는 바꾸지 않는다.
4. **기존 경로 불변** — loopback 기반 자동 발급(설치 스크립트·런타임 통합·로그인 커맨드), 웹 세션 로그인, 수기 발급은 회귀 없이 유지한다. 본 경로는 헤드리스 환경을 위한 **추가** 경로다.
5. **승인 범위 = 본인 계정** — 승인 화면은 로그인한 사람 본인 계정에 대한 토큰을 발급한다. 대리 승인·타인 계정 발급은 범위 밖.
6. **상태 저장소 = 공유 가능한 단명 저장** — 인증 요청 상태(대기↔승인)는 승인 요청과 폴링 요청이 공유해야 하는 가변 상태다. 요청 간 메모리를 공유하지 않는 실행 환경이라 외부 공유 저장이 필요하다. 단명·자동만료 특성에 맞게 보관하되 **추가 비용·인프라 도입은 피한다**(헌법 II). 일반적 구현과 다른 절충이 되는 부분은 설계 문서에 사유를 **간단히 부연**한다. (구체 저장 수단은 plan에서.)
7. **명명 충돌 회피** — 신규 데이터 모델명은 기존 인증·계정 관련 모델명과 겹치지 않아야 한다. (구체 모델명은 plan에서 확정.)

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 브라우저 없는 환경에서 한 번의 승인으로 인증한다 (Priority: P1)

모바일에서 원격 환경으로 작업하는 세션·CI·SSH 등 **브라우저와 토큰 수신자가 같은 기기가 아닌** 환경의 소비자가 데이터를 다루려 한다. 소비자가 인증을 시작하면, 사람이 **자기 기기에서 안내된 주소를 열어 로그인·승인**만 하면 된다. 승인 직후 소비자는 토큰을 자동으로 받아 이후 호출이 바로 동작한다. 사람은 토큰을 보거나 복사하지 않는다.

**Why this priority**: 이 피처의 존재 이유 그 자체다. loopback 전제가 깨지는 환경에서 "수기 발급 폴백"으로 떨어지면 가치가 사라진다(수기 요구는 안 하느니만 못함). 이 흐름 하나로 헤드리스 자동 인증이라는 핵심 가치를 전달한다.

**Independent Test**: 토큰이 없는 헤드리스 소비자에서 인증을 1회 시작 → 다른 기기(폰)에서 안내 주소 열어 로그인·승인 → 소비자가 추가 입력 없이 임의 호출(GET)에 인증 성공하는지 확인. loopback 포트를 전혀 쓰지 않는다.

**Acceptance Scenarios**:

1. **Given** 저장된 토큰이 없는 헤드리스 소비자, **When** 인증을 시작하고 사람이 안내 주소에서 로그인·승인, **Then** 소비자가 토큰을 자동 수신·저장하고 안내가 성공으로 끝난다.
2. **Given** 인증 진행 중(승인 대기), **When** 사람이 아직 승인하지 않음, **Then** 소비자는 정해진 간격으로 조용히 대기하며 사람에게 같은 안내를 반복 강요하지 않는다.
3. **Given** 인증 시작, **When** 사람에게 제시되는 안내, **Then** 코드 입력이 필요 없는 **바로 열리는 링크**(또는 주소+코드)가 함께 제공되어 모바일에서 탭 한 번으로 승인 화면에 도달한다.

### User Story 2 - 만료·중단을 분명히 다룬다 (Priority: P2)

승인이 제때 끝나지 않거나(사용자 미승인·시간 초과), 사용자가 명시적으로 거부하거나, 발급 토큰이 단기 만료된 상황을 소비자가 분명하게 처리한다.

**Why this priority**: 헤드리스 흐름은 사람의 비동기 행동(다른 기기에서 승인)에 의존하므로 대기·실패 상태가 흔하다. 모호하면 무한 대기·토큰 오염·혼란이 생긴다. P1 흐름을 신뢰 가능하게 만드는 안전장치다.

**Independent Test**: (a) 승인하지 않고 시간 초과 → 소비자가 분명한 실패로 종료(저장 토큰 불변). (b) 사용자가 승인 화면에서 거부 → 소비자가 거부를 인지해 즉시 종료. (c) 만료된 토큰으로 호출 → 자동 갱신 소비자는 같은 흐름으로 재인증, 그 외는 재시작 안내.

**Acceptance Scenarios**:

1. **Given** 인증 시작 후 제한 시간 경과, **When** 사람이 끝내 승인 안 함, **Then** 소비자는 타임아웃으로 분명히 실패하고 기존 저장 토큰은 바뀌지 않는다.
2. **Given** 승인 화면, **When** 사람이 거부, **Then** 소비자는 더 기다리지 않고 거부로 종료한다.
3. **Given** 단기 만료된 토큰, **When** 자동 갱신 가능한 소비자가 호출, **Then** 같은 헤드리스 흐름으로 재인증 후 원 요청이 성공한다.

---

### User Story 3 - 기존 인증 경로가 회귀 없이 동작한다 (Priority: P2)

기존 설치 스크립트·런타임 통합·로그인 커맨드(loopback 기반)·웹 세션 로그인·수기 발급 사용자는 이 변경 이후에도 그대로 인증·호출한다.

**Why this priority**: 인증은 모든 기능의 관문이라 회귀가 치명적이다. 신규 경로 추가가 기존 경로를 깨면 안 된다.

**Independent Test**: 브라우저 있는 환경에서 기존 로그인 커맨드/설치 흐름으로 토큰 발급·호출이 그대로 성공. 수기 발급 토큰도 그대로 동작.

**Acceptance Scenarios**:

1. **Given** 브라우저 있는 환경, **When** 기존 자동 발급(loopback) 흐름 실행, **Then** 종전과 동일하게 토큰 발급·호출이 성공한다.
2. **Given** 수기 발급 토큰, **When** 임의 호출, **Then** 종전과 동일하게 인증된다.

### Edge Cases

- 사람이 승인 화면에서 **로그인되어 있지 않으면**: 먼저 로그인하도록 안내되고, 로그인 후 같은 승인 화면으로 이어진다.
- **승인 코드/요청이 만료된 뒤** 사람이 뒤늦게 주소를 열면: 만료를 분명히 알리고 소비자에게는 재시작을 안내한다(유효하지 않은 코드로 토큰이 발급되지 않는다).
- 소비자가 **너무 자주 폴링**하면: 서버가 더 느리게 폴링하도록 신호하고 소비자가 이를 따른다.
- **동시에 두 번 인증을 시작**하면: 각 요청은 독립적이며, 한쪽 승인이 다른 쪽 토큰을 발급하지 않는다(요청-승인 짝이 어긋나지 않는다).
- 같은 안내 링크/코드를 **여러 번 승인**해도 토큰이 중복 누적되지 않도록 1회 승인 = 1회 발급으로 제한한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 헤드리스 소비자가 **사람의 사전 입력 없이 인증을 개시**할 수 있어야 한다(loopback 수신 포트에 의존하지 않는다).
- **FR-002**: 시스템은 개시 시 사람에게 제시할 **승인 안내**(바로 열리는 링크, 그리고 주소+코드 형태)를 제공해야 한다. 모바일에서 탭 한 번으로 승인 화면에 도달할 수 있어야 한다.
- **FR-003**: 사람은 자기 기기에서 **로그인 후 승인 한 번**으로 인증을 완료할 수 있어야 한다. 토큰 값을 보거나 복사하지 않는다.
- **FR-004**: 소비자는 승인 완료를 **자동으로 감지**(대기·재시도 포함)하고 토큰을 자동 수신·보관해야 한다. 사람에게 토큰 처리를 위임하지 않는다.
- **FR-005**: 시스템은 승인 **대기/완료/거부/만료** 상태를 구분해 소비자에게 전달해야 하며, 소비자는 대기 중 과도한 안내 반복 없이 조용히 기다려야 한다.
- **FR-006**: 시스템은 소비자의 **폴링 속도를 조절**할 수 있어야 하고(너무 잦은 폴링 억제), 소비자는 그 신호를 따라야 한다.
- **FR-007**: 본 경로로 발급되는 토큰은 기존 자동 발급과 동일한 **단기 만료(기본 30일)**를 가져야 한다. 만료 시 동일 흐름으로 재인증할 수 있어야 한다.
- **FR-008**: 시스템은 **요청-승인 짝**을 안전하게 묶어, 한 승인이 다른 인증 요청의 토큰을 발급하지 않도록 보장해야 한다. 만료/소진된 요청으로는 토큰이 발급되지 않아야 한다.
- **FR-009**: 시스템은 기존 인증 경로(loopback 자동 발급·웹 세션 로그인·수기 발급)를 **회귀 없이 유지**해야 한다.
- **FR-010**: 승인은 **로그인한 본인 계정**에 대해서만 토큰을 발급해야 한다.
- **FR-011**: 실패(타임아웃·거부·만료·중단) 시 **기존 저장 토큰은 변경되지 않아야** 한다.

### Key Entities *(include if feature involves data)*

- **인증 요청(Device 요청)**: 소비자가 개시한 1회 인증 시도. 사람에게 보일 짧은 코드, 승인 주소, 내부 식별자, 만료 시각, 폴링 간격, 현재 상태(대기·승인·거부·만료)를 가진다. 승인 시 한 사람의 계정과 연결된다.
- **발급 토큰(개인 액세스 토큰)**: 승인 완료로 발급되는 기존 토큰. 소유자·만료 시각(단기)·폐기 가능 속성을 기존 모델 그대로 가진다.
- **사용자 계정**: 승인 주체. 로그인된 본인에 한해 토큰이 발급된다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 브라우저와 토큰 수신자가 다른 기기인 환경에서, 사람의 조작은 **로그인 승인 1회(탭 한 번 수준)** 만으로 인증이 완료된다(수기 토큰 복사·붙여넣기 0회).
- **SC-002**: 사람이 승인을 마치면 소비자는 **수 초 내** 자동으로 토큰을 받아 다음 호출이 성공한다.
- **SC-003**: 승인이 안 된 채 제한 시간이 지나면 소비자는 **무한 대기 없이** 분명한 실패로 종료하고 기존 토큰을 보존한다(100% 결정적 종료).
- **SC-004**: 기존 인증 경로(설치·런타임·로그인 커맨드·세션·수기 발급) 사용자는 본 변경 이후 **회귀 0건**으로 동일하게 인증한다.
- **SC-005**: 한 번의 승인은 정확히 한 개의 토큰만 발급한다(중복·오발급 0건).
