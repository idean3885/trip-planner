## User Scenarios & Testing *(mandatory)*

### User Story 1 - 자연어로 일정 수정 (Priority: P1)

비개발자 동행자가 AI 에이전트에게 "리스본 3일차에 벨렘탑 추가해줘"라고 말하면, AI가 관광지를 검색하고 여행 일정에 반영한다. 반영된 결과는 웹사이트에서 즉시 확인할 수 있다.

**Why this priority**: 프로젝트의 핵심 가치. 기존에는 마크다운 파일을 직접 편집하고 git push해야 반영되었으나, 이제 API를 통해 데이터베이스에 즉시 반영되어 비개발자도 실시간으로 일정을 변경할 수 있다. AX-First 원칙의 핵심 구현.

**Independent Test**: AI 에이전트에게 일정 추가를 요청하고, 웹사이트에서 즉시 반영 여부를 확인하여 독립 테스트 가능.

**Acceptance Scenarios**:

1. **Given** 로그인된 사용자가 AI 에이전트와 대화 중일 때, **When** "3일차에 벨렘탑 추가해줘"라고 요청하면, **Then** AI가 관광지를 검색하고 해당 일자에 활동을 추가하며 웹에서 즉시 확인 가능하다.
2. **Given** AI 에이전트가 숙소를 검색한 후, **When** 사용자가 "2번 호텔로 1일차 숙소 변경해줘"라고 요청하면, **Then** 해당 일자의 숙소 정보가 업데이트되고 웹에 즉시 반영된다.
3. **Given** 사용자가 일정 조회를 요청할 때, **When** AI가 전체 여행 일정을 조회하면, **Then** 모든 일자의 제목, 내용, 날짜가 정확히 표시된다.

---

### User Story 2 - 외부 클라이언트 인증 (Priority: P2)

개발자가 웹사이트에서 개인 접근 토큰(PAT)을 생성하고, 이를 AI 에이전트에 등록하면 이후 자동으로 인증이 처리된다. 비개발자는 설치 과정에서 토큰을 한 번 입력하면 이후 신경 쓸 필요가 없다.

**Why this priority**: P1의 전제 조건. AI 에이전트가 API를 호출하려면 인증 수단이 필요하다. 웹 브라우저는 세션 기반 인증을 사용하지만 외부 클라이언트에는 별도 인증 메커니즘이 필요하다.

**Independent Test**: PAT를 생성하고, 해당 토큰으로 API 호출이 성공하는지 확인하여 독립 테스트 가능.

**Acceptance Scenarios**:

1. **Given** 로그인한 사용자가 설정 페이지에 접근할 때, **When** 토큰 생성을 요청하면, **Then** 고유한 토큰이 발급되고 한 번만 표시된다.
2. **Given** 유효한 토큰을 가진 외부 클라이언트가 API를 호출할 때, **When** 토큰을 인증 헤더에 포함하면, **Then** 해당 토큰 소유자의 권한으로 요청이 처리된다.
3. **Given** 만료되거나 삭제된 토큰으로 API를 호출할 때, **When** 요청이 도착하면, **Then** 인증 실패 응답이 반환된다.
4. **Given** 사용자가 설정 페이지에서 토큰 목록을 조회할 때, **When** 삭제 버튼을 누르면, **Then** 해당 토큰이 즉시 무효화된다.

---

### User Story 3 - 1줄 설치로 전체 기능 사용 (Priority: P3)

비개발자가 터미널에서 설치 명령 1줄을 실행하면, 검색 기능과 일정 관리 기능이 모두 포함된 AI 에이전트 도구가 설치된다. 설치 과정에서 필요한 인증 정보만 입력하면 나머지는 자동 처리된다.

**Why this priority**: 비개발자 캡슐화 원칙. 설치가 복잡하면 사용자가 이탈한다. 기존에는 검색 도구와 피드백 도구가 분리되어 있었으나, 이제 하나의 통합 도구로 제공한다.

**Independent Test**: 새로운 맥북에서 설치 명령을 실행하고, AI 에이전트에서 검색과 일정 관리가 모두 작동하는지 확인하여 독립 테스트 가능.

**Acceptance Scenarios**:

1. **Given** AI 에이전트 도구가 설치되지 않은 환경에서, **When** 설치 명령 1줄을 실행하면, **Then** 모든 의존성이 설치되고 AI 에이전트에 도구가 자동 등록된다.
2. **Given** 기존 버전(v1)이 설치된 환경에서, **When** 설치 명령을 재실행하면, **Then** v2로 업그레이드되고 폐기된 기능(피드백 도구)이 자동 제거된다.
3. **Given** 설치 과정에서 API 키와 토큰 입력이 필요할 때, **When** 사용자가 값을 입력하면, **Then** 안전한 저장소에 자동 보관되어 재입력이 불필요하다.

### User Story 4 - API 문서 제공 (Priority: P4)

개발자가 여행 플래너의 API 문서를 조회하면, 모든 엔드포인트의 사용법, 요청/응답 형식, 인증 요구사항을 확인할 수 있다. AI 에이전트도 이 문서를 참조하여 정확한 API 호출을 수행한다.

**Why this priority**: 포트폴리오 어필과 AX 정확성에 기여. API 문서가 기계 판독 가능하면 AI 에이전트가 스스로 올바른 호출을 구성할 수 있다.

**Independent Test**: 문서 페이지에 접근하여 모든 엔드포인트가 정확히 기술되어 있는지 확인하여 독립 테스트 가능.

**Acceptance Scenarios**:

1. **Given** 개발자가 API 문서 페이지에 접근할 때, **When** 엔드포인트 목록을 조회하면, **Then** 모든 엔드포인트의 경로, 메서드, 요청/응답 형식, 인증 요구사항이 표시된다.
2. **Given** AI 에이전트가 API를 호출하기 전에, **When** 문서를 참조하면, **Then** 기계 판독 가능한 형식으로 정확한 스키마 정보를 얻을 수 있다.

---

### User Story 5 - 코드 구조의 명확한 분리 (Priority: P5)

개발자가 프로젝트 저장소를 탐색할 때, 웹 애플리케이션 코드와 AI 에이전트 도구 코드가 명확히 분리되어 각각의 역할이 즉시 파악된다. 배포 대상과 로컬 실행 대상이 디렉토리 구조만으로 구분된다.

**Why this priority**: 유지보수성과 프로젝트 이해도. 현재 웹앱 코드와 AI 도구 코드가 혼재되어 있어 구조의 의도가 불분명하다.

**Independent Test**: 프로젝트 루트의 디렉토리 구조만 보고 웹앱 코드와 AI 도구 코드를 구분할 수 있는지 확인하여 독립 테스트 가능.

**Acceptance Scenarios**:

1. **Given** 개발자가 프로젝트 루트를 탐색할 때, **When** 디렉토리 구조를 확인하면, **Then** 배포 대상(웹앱)과 로컬 실행 대상(AI 도구)이 최상위 디렉토리로 분리되어 있다.
2. **Given** 중복된 코드가 존재할 때, **When** 정리가 완료되면, **Then** 동일한 기능을 수행하는 코드가 하나만 존재한다.

---

### Edge Cases

- 토큰 만료 후 AI 에이전트가 API를 호출하면 어떻게 되는가? → 인증 실패 응답과 함께 토큰 재생성 안내 메시지가 제공된다.
- AI 에이전트와 웹 브라우저에서 동시에 같은 일정을 수정하면? → 마지막 저장이 우선하며, 웹에서 최신 상태를 즉시 확인할 수 있다.
- 설치 중 네트워크가 끊기면? → 재실행 시 이전 진행 상태를 감지하여 이어서 설치한다.
- 검색 API(외부)는 동작하지만 일정 관리 API(자체)에 접근 불가한 경우? → 검색 기능은 독립적으로 동작하며, 일정 관리 실패는 명확한 오류 메시지로 안내한다.
- 비개발자가 웹에서 토큰을 생성하지 않고 설치를 시도하면? → 설치 스크립트가 토큰 생성 방법을 안내하고, 입력을 대기한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 하나의 통합된 AI 에이전트 도구로 검색(숙소, 항공, 관광지)과 일정 관리(조회, 생성, 수정, 삭제)를 모두 제공해야 한다.
- **FR-002**: 시스템은 웹 브라우저 세션 외에 외부 클라이언트(AI 에이전트)를 위한 토큰 기반 인증을 지원해야 한다.
- **FR-003**: 사용자는 웹사이트 설정 페이지에서 개인 접근 토큰을 생성, 조회, 삭제할 수 있어야 한다.
- **FR-004**: 토큰으로 인증된 요청은 해당 토큰 소유자의 권한(역할)으로 처리되어야 한다.
- **FR-005**: AI 에이전트가 수행한 일정 변경은 웹사이트에서 즉시 확인 가능해야 한다.
- **FR-006**: 모든 API 엔드포인트는 기계 판독 가능한 표준 형식으로 문서화되어야 한다.
- **FR-007**: 설치 명령 1줄로 검색과 일정 관리 기능이 모두 설치되어야 한다.
- **FR-008**: 기존 버전(v1) 사용자가 설치를 재실행하면 v2로 업그레이드되고, 폐기된 기능이 자동 정리되어야 한다.
- **FR-009**: 웹 애플리케이션 코드와 AI 에이전트 도구 코드는 저장소 내에서 물리적으로 분리되어야 한다.
- **FR-010**: 폐기된 외부 플랫폼 기반 피드백 채널은 완전히 제거되어야 한다.
- **FR-011**: 중복된 AI 에이전트 도구 코드는 하나로 통합되어야 한다.

### Key Entities

- **개인 접근 토큰(PAT)**: 외부 클라이언트가 API에 인증하기 위한 수단. 사용자별로 여러 개 생성 가능하며, 이름/만료일/마지막 사용 시각을 가진다.
- **통합 AI 도구**: 검색(외부 서비스)과 일정 관리(자체 API)를 하나의 도구 세트로 제공하는 AI 에이전트용 도구.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 비개발자가 AI 에이전트에게 일정 변경을 요청하면, 30초 이내에 웹사이트에서 변경 결과를 확인할 수 있다.
- **SC-002**: 설치 명령 실행부터 AI 에이전트에서 첫 번째 검색/일정 조회까지 5분 이내에 완료된다.
- **SC-003**: 폐기 대상 코드(피드백 채널, 중복 도구) 제거 후 저장소에 해당 코드가 0건 존재한다.
- **SC-004**: 모든 공개 API 엔드포인트가 문서화되어 있고, 문서에 없는 엔드포인트가 0건이다.
- **SC-005**: v1에서 v2로 업그레이드 시 기존 검색 기능이 100% 동작하며 데이터 손실이 없다.
- **SC-006**: 프로젝트 디렉토리 구조만으로 웹앱 코드와 AI 도구 코드를 3초 이내에 구분할 수 있다.

## Assumptions

- 기존 웹 애플리케이션의 소셜 로그인 인증과 역할 기반 권한(OWNER/HOST/GUEST)은 그대로 유지된다.
- 토큰 인증은 기존 세션 인증과 병행하며, 동일한 권한 체계를 공유한다.
- 외부 검색 서비스의 인증 방식(API 키)은 변경되지 않는다.
- 마지막 저장 우선(last-write-wins) 정책으로 동시 편집 충돌을 처리한다.
- 설치 대상 환경은 macOS이며, OS 기본 자격 증명 저장소를 사용한다.
- 일정 내 활동의 구조화 데이터 모델은 이번 범위에 포함되지 않으며, 후속 마이너 버전에서 진행한다.

## Scope Boundaries

### In Scope
- PAT 기반 인증 인프라 (생성, 검증, 삭제)
- 기존 검색 도구 + 일정 관리 도구의 단일 AI 도구 통합
- 외부 플랫폼 기반 피드백 채널 제거
- 중복된 검색 도구 제거
- API 문서화
- 설치 스크립트 전면 개편
- 저장소 디렉토리 구조 정리

### Out of Scope
- 일정 내 활동의 구조화 데이터 모델 (v2.1+)
- 모바일 최적화 UI (v2.1+)
- CLI 도구 (API 안정화 후 필요 시)
- 실시간 동시 편집
- 여행 데이터 마이그레이션 (기존 정적 데이터는 레거시 보존)

## Constitution Compliance

| 원칙 | 준수 여부 | 근거 |
