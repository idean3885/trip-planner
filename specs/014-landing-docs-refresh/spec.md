# Feature Specification: 공개 랜딩 페이지 & 문서 체계 개편

**Feature Branch**: `014-landing-docs-refresh`
**Created**: 2026-04-20
**Status**: Draft
**Input**: User description: 비로그인 상태 방문자에게 프로젝트를 소개하는 공개 랜딩 페이지(/)를 신설하고, README·docs/ 문서 체계를 외부 방문자 친화적으로 재정돈한다. 목적은 이 프로젝트를 이력서/포트폴리오에서 어필하는 것. 트랙 3종: (1, 필수) 랜딩 페이지, (2, 필수) README·docs 정비, (3, 선택) 루트 구조 정리. 제약: 반응형 단일 디자인, shadcn 디자인 시스템 일관성, 인증 가드가 공개 진입점을 막지 않을 것.

## Clarifications *(decisions sealed during drafting)*

1. **언어 범위 — 한국어 단일** — 이 프로젝트의 1차 청중은 국내 채용·동료 개발자. 영어 버전은 별도 이슈로 이관한다. 랜딩·개편된 문서 모두 한국어로 작성한다.
2. **로그인 사용자의 루트(`/`) 진입 처리 — 대시보드로 자동 전환** — 랜딩은 "비로그인 방문자용 공개 마케팅 면"으로 정의한다. 이미 로그인한 사용자가 `/`에 도달하면 기존 여행 목록 화면으로 이동시켜 가치 있는 첫 화면을 보장한다.
3. **데모 영역 수준 — 정적 스크린샷 + 캡션이 기본선** — 라이브 샘플 여행 데이터나 인터랙티브 데모는 범위 밖. 스크린샷은 직전 디자인 시스템 마이그레이션이 반영된 최신 UI 기준으로 확보한다.
4. **트랙 3(루트 구조 정리) 범위 — 명백한 레거시 제거만** — 개인 레거시(`02_honeymoon_plan.md`)와 Jekyll 잔재(`_config.yml`·`index.md`) 현행 사용 여부 판정 및 제거/이관까지만 포함한다. 디렉토리 중복 해소(예: `mcp` vs `mcp-servers`, `src` vs `web`)는 별도 이슈로 v2.6.1 이후로 미룬다.
5. **대상 독자 그룹 — 3층으로 확정** — (a) 외부 방문자: 채용 담당자·동료 개발자·포트폴리오 열람자, (b) 기여자·개발자: 코드를 읽거나 수정할 사람, (c) 운영·감사: 릴리즈·감사·데이터 관리자. docs/ 엔트리는 이 3층으로 그룹핑한다.

## Metatag Conventions *(normative, inherited from PR #204)*

본 피처의 tasks.md·plan.md는 네 종 메타태그를 통해 후속 자동 검증과 연결된다:

- `[artifact: <path>|<path>::<symbol>]` — 산출 파일 식별자(drift 감사 기준)
- `[why: <short-tag>]` — 추적 그룹 키(plan↔tasks 커버리지·이슈 합산)
- `[multi-step: N]` — plan bullet이 다단 작업일 때 최소 매핑 태스크 수(N ≥ 2)
- `[migration-type: schema-only | data-migration]` — 마이그레이션 산출물 구분

형식 검증은 `.specify/scripts/bash/validate-metatag-format.sh`가 수행한다. 의미 검증은 각 US의 validator가 담당한다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 외부 방문자가 랜딩만 보고 프로젝트를 이해한다 (Priority: P1)

이력서·포트폴리오 링크를 타고 들어온 사람이 로그인하지 않은 상태에서도 "이 프로젝트가 무엇을 하는 서비스이고, 어떤 가치를 제공하며, 어떤 기술로 만들어졌는지"를 10초 내에 파악한다. 스크롤을 조금 더 내리면 주요 기능, 데모 스크린샷, 기술 스택, 소스 저장소 링크, 로그인 CTA가 이어진다.

**Why this priority**: 이 피처의 존재 이유 자체. 이 스토리가 실패하면 랜딩을 만들 이유가 없다. 포트폴리오 어필의 유일한 1차 접점이다.

**Independent Test**: 새 시크릿 창에서 `/`에 접속해 로그인 없이 랜딩 전체(정체성·가치·기능·기술·데모·저장소·CTA)를 스크롤로 훑고, 로그인 CTA 클릭 시 인증 플로우로 진입하는지 확인. 10초 이내에 첫 화면(히어로)에서 프로젝트 성격을 서술할 수 있으면 통과.

**Acceptance Scenarios**:

1. **Given** 비로그인 방문자가, **When** `/`에 접속하면, **Then** 인증 화면이 아닌 랜딩 콘텐츠가 렌더된다.
2. **Given** 랜딩 히어로 영역에서, **When** 방문자가 첫 화면을 읽으면, **Then** "무엇을 하는 서비스인지"가 한 문장으로 선언되어 있다.
3. **Given** 랜딩을 끝까지 스크롤했을 때, **When** 방문자가 본문을 훑으면, **Then** 핵심 가치·주요 기능·기술 스택·데모·저장소 링크·로그인 CTA가 모두 한 페이지 안에 존재한다.
4. **Given** 랜딩 페이지에서, **When** 저장소 링크를 클릭하면, **Then** 공개 GitHub 저장소로 이동한다.
5. **Given** 랜딩 페이지에서, **When** 로그인 CTA를 클릭하면, **Then** 기존 웹앱 인증 플로우가 시작된다.

### User Story 2 - README 하나로 프로젝트 탐색 동선을 파악한다 (Priority: P1)

외부 방문자가 GitHub 저장소에 먼저 도달한 경우, README 단 한 번만 읽고도 "서비스를 써보려면 랜딩(또는 웹앱)으로, 코드를 보려면 저장소·개발 문서로, 기여·운영 절차는 해당 문서로" 가는 세 동선이 명확히 보인다. 현재 README는 `trip.idean.me`를 4번 반복 노출하고, docs 진입은 한 줄 링크만 있어 독자별 경로가 불분명하다.

**Why this priority**: README는 GitHub 유입 시 최초 판단 지점이다. 랜딩만큼 중요한 공개 얼굴이며, 랜딩과 역할 분담이 안 되면 "어디로 가야 할지 모르는 독자"가 이탈한다.

**Independent Test**: README를 새로 읽는 사람에게 보여주고 "웹앱을 바로 쓰려면 어디로 가는지, 개발에 참여하려면 어디를 보는지, 프로젝트 개요만 알고 싶으면 어디를 보는지" 세 질문에 각 1클릭 이내 답을 찾게 한다. 3건 모두 즉답이면 통과.

**Acceptance Scenarios**:

1. **Given** 새 방문자가 README를 펼쳤을 때, **When** 첫 1화면을 읽으면, **Then** 프로젝트 정체성 1문장·대표 접근 링크(웹앱 or 랜딩) 1개가 중복 없이 제시된다.
2. **Given** README 본문에서, **When** 방문자가 "웹앱을 써보고 싶다"는 의도를 가지면, **Then** 명시된 단일 진입 링크가 있다.
3. **Given** README 본문에서, **When** 방문자가 "코드를 보고 싶다 / 개발에 참여하고 싶다"는 의도를 가지면, **Then** 개발자 문서로 연결되는 명시된 섹션·링크가 있다.
4. **Given** README 전체에서, **When** `trip.idean.me` 링크 노출 횟수를 세면, **Then** 2회 이하다(히어로 1회 + 링크 표 1회).

### User Story 3 - docs/ 에서 역할별 진입점을 통해 빠르게 목표 문서에 도달한다 (Priority: P2)

개발자·운영자·감사자 각각이 `docs/`에 들어왔을 때 자기 역할의 섹션을 우선 보고, 필요한 문서를 1~2클릭 안에 연다. 현재 `docs/` 하위에는 10종(ARCHITECTURE·DEVELOPMENT·DOMAIN·ENVIRONMENTS·ERD·WORKFLOW·design-handoff·spec.md + audits/evidence/research) 이 평면으로 나열되어 있어, 누가 어디부터 봐야 하는지 불분명하다.

**Why this priority**: US1·US2가 끝난 뒤 방문자가 실제로 깊이 있게 프로젝트를 살펴볼 때 쓰는 면. 랜딩이 "매력"이라면 docs는 "진정성"을 입증하는 면이라 포트폴리오 맥락에서 중요하다. 다만 랜딩·README가 전제 조건이라 P2.

**Independent Test**: 신규 방문자에게 "개발 환경 세팅은 어느 문서?", "ERD는 어느 문서?", "업무 프로세스는 어느 문서?"를 묻고 docs/ 엔트리에서 답하게 한다. 모두 1클릭 이내 통과. 또한 임의 문서 하나를 열어 상단에서 "이 문서는 누구를 위한 것인가" 문장이 있는지 확인.

**Acceptance Scenarios**:

1. **Given** 방문자가 `docs/README.md` 같은 엔트리 문서를 열었을 때, **When** 본문을 보면, **Then** 독자 그룹 3층(외부 방문자 / 기여자·개발자 / 운영·감사)별 섹션으로 목차가 구성되어 있다.
2. **Given** 임의 문서(예: `ARCHITECTURE.md`, `ERD.md`)를 열었을 때, **When** 상단을 보면, **Then** "대상 독자" 또는 그에 상응하는 요약 문장이 있다.
3. **Given** docs 전체에서, **When** 중복·역할이 겹치는 문서 쌍이 있으면, **Then** 통합 또는 한쪽에 포인터만 남기는 식으로 정리되어 있다.

### User Story 4 - 루트 레거시 파일이 사라져 저장소 첫인상이 깔끔하다 (Priority: P3)

저장소 루트에 `02_honeymoon_plan.md` 같은 과거 개인 여행 마크다운이 남아 있어 신규 방문자가 의아해한다. 또한 `_config.yml`·`index.md` 등 Jekyll 잔재가 현행 배포와 무관해 보이는데도 루트에 존재한다. 이번 스코프에서 "제거해도 안전함이 확인된 레거시"만 정리한다.

**Why this priority**: 포트폴리오 첫인상 개선 효과는 있으나, 랜딩·README 정비가 훨씬 직접적인 어필 수단이다. 또한 잘못 삭제하면 운영 영향이 있어 신중한 확인이 필요하므로 후순위.

**Independent Test**: 정리 후 저장소 루트를 본 사람이 "과거 여행 로그" 류의 개인 콘텐츠를 보지 않는다. 정리 대상으로 삼은 파일 각각에 대해 "현재 어떤 배포·빌드 경로에서도 참조되지 않음"이 confirm된 기록이 남아 있다.

**Acceptance Scenarios**:

1. **Given** 루트 정리 대상 파일 목록이 주어졌을 때, **When** 각 파일을 제거 또는 이관하면, **Then** 빌드·테스트·배포가 이전과 동일하게 통과한다.
2. **Given** 루트를 방문자가 처음 열었을 때, **When** 파일 목록을 보면, **Then** 개인 여행 로그 류 파일이 노출되지 않는다.

### Edge Cases

- 로그인 상태 사용자가 랜딩을 **일부러** 다시 보고 싶은 경우: 본 피처 범위 밖. 대시보드로 자동 이동되며, 마케팅 면 재열람 수요는 드물다고 판단.
- 랜딩 스크린샷에 포함된 UI가 향후 디자인 시스템 마이그레이션 과정에서 변경되는 경우: 스크린샷은 "재촬영이 쉬운 산출물"로 관리하며, 레이아웃 개편 시 본 피처와 무관하게 갱신할 수 있는 구조여야 한다.
- 자바스크립트 비활성·검색엔진 크롤러: 랜딩의 핵심 소개 콘텐츠(정체성·주요 기능·저장소 링크)는 JS 없이도 읽혀야 한다(포트폴리오·검색 유입 가치 보호).
- docs/ 재구성으로 기존 내부 링크가 깨지는 경우: 링크 검사 후 전부 갱신한다. 외부(다른 저장소·블로그 등) 링크까지는 보장하지 않는다.
- README에서 `trip.idean.me` 노출을 줄이면서도 "웹앱을 바로 쓰고 싶은" 의도의 독자가 혼란을 겪는 경우: 히어로에서 대표 링크로 그 의도를 먼저 받는다.
- 루트 레거시 제거가 다른 도구의 현행 동작(예: GitHub Pages·기존 curl 설치 스크립트 경로 등)에 영향을 주는 경우: 제거 전 각 파일별로 참조 검색(grep)·확인 후에만 제거한다.

## Requirements *(mandatory)*

### Functional Requirements

#### 랜딩 페이지 (Track 1, 필수)

- **FR-LA-001**: 비로그인 상태에서 루트 경로에 접속했을 때 인증 리디렉트 없이 랜딩 페이지가 렌더되어야 한다.
- **FR-LA-002**: 랜딩은 프로젝트 정체성을 한 문장으로 단언하는 히어로 영역을 제공해야 한다.
- **FR-LA-003**: 랜딩은 3~5개의 핵심 가치(WHY)를 짧은 문장으로 제시해야 한다.
- **FR-LA-004**: 랜딩은 2~3개의 대표 기능(WHAT — 웹 일정 관리, AI 에이전트 연동 등)을 각 1~2문장으로 요약해야 한다.
- **FR-LA-005**: 랜딩은 주요 기술 스택을 시각적으로 제시해야 한다.
- **FR-LA-006**: 랜딩은 실제 제품 화면 스크린샷을 최소 1컷 이상 포함해야 한다.
- **FR-LA-007**: 랜딩은 공개 GitHub 저장소 링크를 제공해야 한다.
- **FR-LA-008**: 랜딩은 로그인(웹앱 사용 시작)으로 이어지는 주요 CTA를 제공해야 한다.
- **FR-LA-009**: 랜딩은 반응형 단일 디자인으로 모바일·데스크톱 양쪽에서 동일 레이아웃을 유지해야 한다.
- **FR-LA-010**: 랜딩은 현 프로젝트가 사용하는 디자인 시스템의 토큰·컴포넌트와 일관된 시각 언어를 따라야 한다.
- **FR-LA-011**: 로그인 상태로 루트 경로에 진입하면 기존 여행 목록 화면으로 이동되어야 한다.
- **FR-LA-012**: 기존 인증 가드가 루트 경로(랜딩) 접근을 차단하지 않도록 공개 허용 경로로 취급되어야 한다.
- **FR-LA-013**: 랜딩은 자바스크립트 비활성 환경에서도 핵심 소개 텍스트와 주요 링크를 읽을 수 있는 콘텐츠 구조를 유지해야 한다.

#### README 정비 (Track 2, 필수)

- **FR-RM-001**: README는 비개발자·개발자 모두를 염두에 둔 1차 진입점으로 재작성되어야 한다.
- **FR-RM-002**: README의 `trip.idean.me` 링크 노출은 2회 이하로 감소해야 한다(히어로 1회 + 링크 표 1회).
- **FR-RM-003**: README는 3층 독자 그룹(외부 방문자 / 기여자·개발자 / 운영·감사) 각각이 "어디로 가야 하는지"를 최대 1클릭으로 알 수 있는 섹션을 가져야 한다.
- **FR-RM-004**: README는 프로젝트 정체성 문장을 랜딩과 일관되게 유지해야 한다(두 면이 서로 다른 요약을 내놓지 않음).
