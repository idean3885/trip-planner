**Feature Branch**: `001-ax-travel-planning`
**Created**: 2026-03-22
**Status**: Draft
**Input**: User description: "AX 기반 여행 플래닝 — AI가 여행 요소를 검색·분석·추천하고, 완성된 일정을 동행자에게 모바일로 공유"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 여행 요소 검색·추천 (Priority: P1)

사용자가 여행 조건(도시, 날짜, 예산, 선호 등)을 알려주면, AI가 해당 영역(숙소, 항공편, 관광, 식사, 교통 등)을 검색하고 분석하여 후보를 추천 기준과 함께 제시한다. 사용자가 후보 중 선택하거나 조건을 보완하여 확정한다. 모든 영역에서 동일한 플로우로 동작한다.

**Why this priority**: 여행 플래닝의 핵심. 모든 일정 구성 요소가 이 플로우를 통해 확정된다

**Independent Test**: "마드리드 6/11~12 숙소 추천해줘" 또는 "바르셀로나 맛집 추천해줘"라고 요청하면 후보가 추천 기준과 함께 제시된다

**Acceptance Scenarios**:

1. **Given** 여행 조건(영역, 도시, 날짜 등)이 주어졌을 때, **When** 검색을 요청하면, **Then** 해당 영역의 검색 결과가 반환된다
2. **Given** 검색 결과가 반환되었을 때, **When** 특정 항목의 상세를 요청하면, **Then** 리뷰·가격·시설 등 상세 정보가 반환된다
3. **Given** 여러 후보가 있을 때, **When** 추천을 요청하면, **Then** 후보가 추천 기준과 함께 제시되고 사용자가 선택 또는 보완할 수 있다
4. **Given** 예약이 필요한 항목일 때, **When** 상세를 확인하면, **Then** 예약 필요 여부가 안내된다

---

### User Story 2 - 일정 생성 (Priority: P1)

확정된 여행 요소를 기반으로 AI가 일별 일정과 전체 요약을 생성한다. 기존에 예약 완료된 항목은 유지하고, 미확정 부분은 검색·추천으로 채운다. 전체 요약에는 여행 루트·기간·숙소 한눈에 보기와 일별 목차가 포함된다.

**Why this priority**: 동행자에게 전달할 핵심 산출물

**Independent Test**: 확정된 여행 정보를 전달하면 일별 일정과 전체 요약이 생성되고, 핵심 정보가 모두 포함된다

**Acceptance Scenarios**:

1. **Given** 확정된 여행 정보와 기존 예약 데이터가 있을 때, **When** 일정 생성을 요청하면, **Then** 일별 일정과 전체 요약이 생성된다
2. **Given** 기존에 예약된 항목이 있을 때, **When** 일정을 생성하면, **Then** 예약 완료 정보는 그대로 유지되고 미확정 부분만 새로 채워진다
3. **Given** 생성된 일정이 있을 때, **When** 내용을 확인하면, **Then** 동행자가 필요한 핵심 정보(숙소, 이동, 관광, 식사, 비용)가 모두 포함되어 있다
4. **Given** 전체 요약이 생성되었을 때, **When** 내용을 확인하면, **Then** 여행 루트·기간·숙소 요약과 일별 목차가 포함되어 원하는 날짜로 바로 찾아갈 수 있다

---

### User Story 3 - 일정 공유 (Priority: P1)

생성된 일정을 웹앱을 통해 동행자에게 공유한다. 동행자는 모바일에서 일정을 열람한다.

**Why this priority**: 비개발자 동행자에게 딜리버리하는 최종 단계

**Independent Test**: 공유된 일정을 모바일에서 열면 정상 표시된다

**Acceptance Scenarios**:

1. **Given** 일정이 웹앱에 배포되었을 때, **When** 동행자가 모바일에서 열면, **Then** 추가 조작(줌, 가로 스크롤) 없이 열람 가능하다
2. **Given** 일정이 수정되었을 때, **When** 개발자가 앱을 재배포하면, **Then** 동행자가 최신 일정을 확인할 수 있다

---

### Edge Cases

- 검색 결과가 0건일 때 검색 조건 변경을 안내한다
- 기존 예약 데이터와 검색 결과가 충돌할 때 기존 예약을 우선한다

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 여행 요소(숙소, 항공편, 관광, 식사, 교통 등)를 조건에 따라 검색할 수 있어야 한다
- **FR-002**: 검색된 항목의 상세 정보(리뷰, 가격, 시설 등)를 조회할 수 있어야 한다
- **FR-003**: 검색 결과를 분석하여 후보를 추천 기준과 함께 제시해야 한다
- **FR-004**: 예약이 필요한 항목의 예약 필요 여부를 안내해야 한다
- **FR-009**: 가격은 한화(KRW)를 기본으로 표시해야 한다
- **FR-010**: 숙소 상세 조회 시 다중 플랫폼(Booking.com, Agoda, Hotels.com 등) 예약 링크를 제공해야 한다
- **FR-011**: 비개발자도 1줄 명령어로 설치하고 Claude Desktop에서 바로 사용할 수 있어야 한다
- **FR-005**: 확정된 여행 정보를 기반으로 일별 일정과 전체 요약을 생성해야 한다
- **FR-006**: 기존 예약 완료 데이터는 유지하고 미확정 부분만 새로 채워야 한다
- **FR-007**: 모든 산출물은 웹앱을 통해 모바일에서 추가 조작 없이 열람 가능해야 한다
- **FR-008**: 일정 수정 후 재배포하면 동행자가 최신 일정을 확인할 수 있어야 한다
- **FR-012**: 여러 여행을 독립적으로 관리하고, 홈 화면에서 여행 목록을 조회할 수 있어야 한다

### Key Entities

- **Trip**: 여행 단위 — 이름, 기간, 도시 목록, 인원
- **TravelItem**: 여행 요소 — 영역(숙소/항공/관광/식사/교통), 이름, 도시, 가격, 리뷰, 예약 상태
- **DailySchedule**: 일별 일정 — 날짜, 도시, 숙소, 이동, 관광, 식사, 비용

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 여행 1건(15일, 5개 도시) 플래닝을 완료할 수 있다
- **SC-002**: 생성된 일별 일정에 동행자가 필요한 핵심 정보(숙소, 이동, 관광, 식사, 비용)가 모두 포함된다
- **SC-003**: 동행자가 웹앱을 통해 모바일에서 일정을 추가 조작 없이 열람할 수 있다
- **SC-004**: 일정 수정 후 앱 재배포하면 동행자가 최신 일정을 확인할 수 있다
- **SC-005**: 실제 여행 데이터(포르투갈&스페인 6/7~6/21)로 전체 플래닝 사이클을 완료할 수 있다
