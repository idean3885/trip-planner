# ADR-001: FE-Only Stateless 아키텍처 선택

**Status**: Accepted
**Date**: 2026-03-29
**Deciders**: idean3885
**Context**: 1인 개발 여행 플래너, 개인정보 미포함

## Context

여행 일정 검색·조회·공유를 위한 웹 애플리케이션을 구축한다. 개발자 1인이 전체 스택을 담당하며, 사용 데이터는 여행 일정(도시, 숙소명, 관광지, 예산 등)으로 **개인정보를 일체 포함하지 않는다.**

기술 제약:
- AppPaaS 배포 (FE: React/Vue/Svelte 지원, BE/DB도 별도 제공)
- Node.js 20 (AppPaaS 최대 버전)
- FE 코드는 AI(Claude)가 전량 생성 (AX 기반 개발)

## Decision

### 1. FE 프레임워크: React (Next.js)

**선택 근거:**

| 후보 | AX 코드 생성 | 확장성 | 생태계 | 컨버팅 비용 |
|------|-------------|--------|--------|------------|
| React (Next.js) | 우수 (훈련 데이터 최다) | BE 추가 용이 | 최대 | 없음 |
| SvelteKit | 우수 (문법 단순) | BE 추가 가능 | 작음 | React 전환 시 전면 재작성 |
| Vue (Nuxt) | 보통 | BE 추가 가능 | 중간 | React 전환 시 전면 재작성 |

- AX 관점에서 SvelteKit이 문법 단순성으로 소폭 우위이나, **확장 시 컨버팅 비용이 결정적**
- BE 개발자로서 향후 AppPaaS BE + DB 추가, OAuth2 연동, 슈퍼앱 통합 가능성 존재
- React → 다른 프레임워크 전환 필요 없음. 반대 방향은 전면 재작성

**기각된 대안:**

- **Astro**: AX 최적이나 AppPaaS 미지원 (React/Vue/Svelte만 제공)
- **SvelteKit**: AX 단독 우위이나, 확장 시 React 전환 비용 > 현재 React 보일러플레이트 비용 (AI가 생성하므로 보일러플레이트는 비용 아님)

### 2. 아키텍처: FE-Only, Stateless

**현재 구조:**
```
[사용자] → [Next.js FE (AppPaaS)] → [정적 데이터 (JSON/마크다운, 빌드 번들)]
```

- BE 서버 없음, DB 없음
- 모든 데이터는 빌드 시점에 번들링되거나 정적 파일로 제공
- 디바이스 변경 시에도 동일 데이터 접근 가능 (서버 상태 없음)

**확장 방향 (구현하지 않음, 방향성만 기록):**
```
[사용자] → [Next.js FE] → [AppPaaS BE (Spring/Node)] → [AppPaaS DB]
```

- OAuth2 인증, 사용자별 데이터 분리, 슈퍼앱 연동 시 BE 도입
- AppPaaS 인프라로 BE/DB 모두 지원되므로 SaaS 벤더 종속 없음

### 3. 보안 수준: 최소 (의식적 판단)

#### 위협 분석

| 위협 | 영향도 | 발생 가능성 | 대응 |
|------|--------|------------|------|
| 여행 일정 데이터 노출 | **낮음** — 개인정보 없음. 도시명, 숙소명, 관광지는 공개 정보 | 중간 | 수용 |
| 데이터 변조 | **없음** — Stateless, 서버 상태 없음. 빌드된 정적 데이터만 존재 | 해당없음 | 해당없음 |
| API 키 노출 | **중간** — RapidAPI 키 (유료) | 낮음 (서버 사이드 or 빌드 시점 사용) | 환경변수 분리, .env gitignore |
| XSS/CSRF | **낮음** — 입력 폼 없음, 읽기 전용 | 낮음 | Next.js 기본 보호 |

#### 판단 근거

1. **보호할 자산이 없다**: 저장되는 데이터는 여행 일정(공개 정보의 조합)이며, 개인정보(이름, 연락처, 결제정보 등)를 일체 포함하지 않는다
2. **서버 상태가 없다**: DB가 없으므로 변조 대상이 없다. 클라이언트 조작은 본인 브라우저에만 영향
3. **사용자가 1명이다**: 인증/인가로 보호할 멀티테넌트 경계가 없다
4. **비용 대비 효과**: 인증 시스템 구축 비용 > 보호 대상의 가치

#### 보안 에스컬레이션 트리거

다음 조건 중 하나라도 충족되면 보안 수준을 재검토한다:

| 트리거 | 조치 |
|--------|------|
| 개인정보 저장 시작 (이름, 연락처, 여권번호 등) | **즉시** 인증 + BE 분리 + 암호화 |
| DB 도입 시 | 개인정보 제거 확인, FE 위임 범위 검토 |
| 다중 사용자 지원 시 | 인증 + 사용자별 데이터 격리 (RLS or BE) |
| 결제 기능 추가 시 | PCI DSS 준수, BE 필수 |
| 외부 사용자 공개 시 | 입력 검증, Rate Limiting, OWASP Top 10 대응 |

> **핵심**: 보안을 하지 않는 것이 아니라, 위협 분석 결과 현재 보호할 자산과 공격 표면이 없으므로 보안 투자를 유보하는 것이다. 에스컬레이션 트리거를 명시함으로써 보안이 필요해지는 시점을 사전에 정의한다.

### 4. 제외 항목

| 항목 | 제외 사유 |
|------|----------|
| PDF 추출 | 웹 뷰로 대체. 모바일 브라우저에서 직접 조회 |
| iCal 생성 | 현 스코프 밖. 캘린더 연동은 확장 시 검토 |
| MCP 서버 (검색 도구) | 토큰 낭비. CLI 스크립트 방식으로 전환 |
| BE / DB | 1인 개발, FE-only로 충분. 확장 시 AppPaaS BE 도입 |
| 인증 (OAuth2 등) | 단일 사용자, 개인정보 없음. 확장 시 도입 |

## Decision Log

의사결정 과정을 시간순으로 기록한다. 블로그 포스팅 및 회고 소재로 활용.

### 1. MCP → CLI 스크립트 전환

**기존**: Claude Desktop MCP 서버로 검색 도구 제공 (FastMCP + Booking.com API)
**문제**: MCP 호출마다 도구 스키마 + 요청/응답이 컨텍스트에 적재 → 과도한 토큰 소비
**결정**: CLI 스크립트 방식으로 전환. 기존 `api_client.py` 재사용, Bash에서 직접 실행
**효과**: MCP 프로토콜 오버헤드 제거, 필요한 결과만 선택적으로 읽기 가능

### 2. FE 프레임워크 선택 여정

**1차 후보 — Astro 5.x**:
- AX 관점 최적 (HTML 기반, AI 오류 표면 최소, 마크다운 네이티브, JS 제로)
- Node 20 지원 (`^20.3.0`)
- **탈락 사유**: AppPaaS가 React/Vue/Svelte만 지원. Astro는 배포 불가

**2차 후보 — SvelteKit**:
- AX 코드 생성 우수 (HTML에 가까운 문법)
- 번들 최소 (컴파일 방식)
- **탈락 사유**: 확장 시 React 전환이 필요하면 전면 재작성. 1인 개발에서 프레임워크 전환 비용은 치명적

**최종 선택 — React (Next.js)**:
- AI 훈련 데이터 최다 → 코드 생성 품질 안정적
- BE 확장 시 API routes 추가만으로 전환 가능
- React 보일러플레이트가 많다는 단점은 AI가 전량 생성하므로 비용 아님
- AppPaaS 주류 지원, 인력풀 최대

### 3. 보안 수준 결정 과정

**초기 검토**: FE-only DB 접근 (Supabase RLS, Firebase Security Rules) 조사
- CVE-2025-29927 (Next.js 미들웨어 인증 우회, CVSS 9.1) 확인
- Supabase RLS, Firebase Rules는 인프라 레벨 강제로 안전

**전환점**: "1인 개발이고 개인정보 없는 소규모 서비스에 이 수준이 과도한가?"
- 보호할 자산 분석 → 여행 일정 = 공개 정보의 조합, 개인정보 제로
- DB 자체가 없는 Stateless 구조 → 변조 대상 없음
- 인증 시스템 구축 비용 > 보호 대상의 가치

**결정**: 보안 최소화. 단, **에스컬레이션 트리거를 사전 정의**하여 "안 한 게 아니라 의식적으로 유보했다"를 증명.

**면접 관점**: "위협 분석을 수행했고, 보호할 자산이 없으므로 유보한다. 개인정보가 들어오면 즉시 인증 + BE 분리한다"가 핵심 논리. 보안을 모르는 것이 아니라, 과잉 설계를 피한 것.

### 4. GitHub Pages → AppPaaS 전환

**기존 spec**: GitHub Pages (Jekyll) + 정적 마크다운 공유
**문제**: 정적 사이트는 검색 결과 연동, 인터랙션 불가. 웹 뷰로 딜리버리하려면 동적 기능 필요
**전환점**: AppPaaS로 Svelte/React 배포 가능하다는 사실 확인
**결정**: AppPaaS 웹앱으로 전환. 검색 + 일정 조회 + 모바일 최적화를 하나의 앱으로

### 5. 마일스톤 재편

**기존**: tasks.md 6개 Phase와 GitHub 마일스톤 4개가 별개 체계로 운영
**문제**: Phase ↔ 마일스톤 불일치, 오픈 이슈에 마일스톤 미할당
**결정**: Phase = 마일스톤 1:1 대응으로 재편
- Phase 1+2 합쳐서 M1 (완료)
- Phase 3 US1 → M2 (검색 CLI)
- Phase 4+5 합쳐서 M3 (일정 생성 + 웹 뷰 딜리버리)
- Phase 6 → M4 (통합 검증)

## Consequences

**긍정적:**
- 개발·배포 단순화 (FE 하나만 관리)
- 디바이스 무관 접근 (Stateless)
- AI 생성 코드 검증 범위 최소화

**부정적:**
- 동적 데이터 갱신 불가 (빌드 필요)
- 오프라인 저장/동기화 불가
- 확장 시 BE 도입 필요 (설계 의도)

**리스크:**
- Next.js 메이저 업데이트 시 AI 생성 코드 호환성 확인 필요
- Node 20 EOL 시 AppPaaS 업그레이드 필요 (Astro 전환은 그때 재검토)
