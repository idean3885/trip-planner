# Feature Specification: Google Calendar 연동 (웹 접근자를 위한 주 캘린더 연결)

**Feature Branch**: `018-gcal-integration`
**Created**: 2026-04-20
**Status**: Released (v2.8.0 — 2026-04-21)
**Input**: User description: #305 Google Calendar 연동. 웹으로 처음 들어온 사용자에게 캘린더 연결 경로를 제공한다. 기존 iCal(명령줄 설치 + 플러그인) 경로는 개발자/레포 접근자용 보조 경로로 남겨 둔다. 구글 로그인이 이미 기본 인증이므로 권한 범위만 확장해 자연스럽게 연결한다. 공유 여행은 멤버 각자가 직접 "내 캘린더에 올리기"를 실행했을 때에 한해 본인 캘린더에 이벤트가 만들어진다. Closes #305, Resolves #150. 마일스톤: v2.8.0.

## Clarifications *(초안 작성 중 모호했던 지점을 결정으로 봉합)*

1. **실행 방식(언제 캘린더에 반영되는가)** — v1에서는 사용자가 "구글 캘린더에 올리기" 또는 "다시 반영하기" 버튼을 눌렀을 때에만 캘린더에 반영한다. 여행 일정을 수정할 때마다 시스템이 자동으로 구글 캘린더를 따라 고치는 방식(실시간 자동 동기화)은 이번 범위에서 제외한다(차기 검토 대상).
   - **근거**: 자동 반영은 수정·삭제의 전파 규칙이 복잡해지고, 사용자의 주 캘린더가 의도와 다르게 변경되는 사고 위험이 크다. MVP에서는 "내가 누를 때만 변한다"는 예측 가능성을 우선한다.
2. **캘린더에 올릴 활동의 범위** — 여행에 등록된 **모든 활동**을 캘린더에 올린다. 예약 상태(`REQUIRED`/`RECOMMENDED`/`ON_SITE`/`NOT_NEEDED`)로 걸러내지 않는다.
   - **근거**: iCal 경로는 예약·티켓처럼 "확정된 일정에만" 알람을 붙이려는 용도에 맞춰 제한적으로 올리지만, 구글 캘린더는 사용자가 "오늘 이 여행에서 무엇을 하지?"를 한눈에 확인하는 **여행 일정표**로 쓰이므로 관광·식사 등 모든 일정이 함께 보여야 한다. iCal과 구글 캘린더는 서로 다른 성격의 정본이다.
3. **삭제·반영 취소 규칙** — 여행 내 활동이 삭제되거나, 사용자가 연결을 끊으면 해당 활동에 대응하는 구글 캘린더 이벤트도 함께 삭제한다. 단, 사용자가 **구글 캘린더에서 직접 수정한 이벤트**는 덮어쓰거나 지우지 않고 그대로 두며, 그 개수를 화면에 "건너뛴 이벤트 N개"로 알려 준다.
   - **근거**: 여행 데이터의 원본은 trip-planner이므로, 여행에서 무엇이 사라지면 캘린더에서도 사라지는 흐름이 예측 가능하다. 한편 사용자가 구글 캘린더에서 개인적으로 고친 내용은 사용자의 선택이므로 함부로 되돌리지 않아 신뢰를 지킨다.
4. **이벤트 제목과 설명 구성** — 제목은 `[여행명] 카테고리기호 활동제목` 형식을 쓰고, 설명란에는 예약 상태, 위치(지도 링크), 원본 여행 상세 페이지 링크를 담는다.
   - **근거**: 캘린더의 월간 뷰에서도 어느 여행의 어느 활동인지 즉시 알아볼 수 있어야 한다. 여행명을 맨 앞에 붙이면 여러 여행이 섞여 있을 때 구분이 쉬워진다.
5. **어느 캘린더에 쓸 것인가** — 기본값은 **"이 여행 전용 캘린더를 자동으로 새로 만들어" 그 안에 이벤트를 넣는 방식**이다. 원하는 사용자는 "내 기본 캘린더에 바로 넣기"로 전환할 수 있다.
   - **근거**: 여행 전용 캘린더를 쓰면 여행이 끝난 뒤 표시를 끄거나 통째로 지우기 쉽고, 개인 일정과 섞이지 않아 "내 캘린더가 어지럽혀질까?" 하는 사용자 불안을 줄인다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 웹으로 들어온 사용자가 자기 여행을 구글 캘린더에 올리기 (Priority: P1)

trip.idean.me에 구글 계정으로 로그인한 사용자가 여행 상세 페이지에서 버튼 하나로 본인 구글 캘린더에 여행 활동들을 이벤트로 만들 수 있다. 현재 로그인 상태는 그대로 유지하면서, 캘린더에 쓰기 위해 추가로 필요한 권한만 그 자리에서 한 번 동의받는다. 다시 로그인하거나 계정을 바꾸는 단계 없이 내 여행이 즉시 내 구글 캘린더에 보이게 된다.

**Why this priority**: 이 피처의 핵심 가치이자, 웹으로 들어온 사용자가 캘린더에 여행을 넣을 수 있는 유일한 경로다. 이 한 스토리만 배포해도 "웹에서 캘린더 연결 지점을 못 찾겠다"는 문제(#305, #150)를 동시에 해소한다.

**Independent Test**: 구글 로그인 상태인 사용자가 여행 상세 페이지에서 "구글 캘린더에 올리기" 버튼을 누르고 권한 동의를 마치면, 본인 구글 캘린더에 해당 여행의 활동들이 이벤트로 생성되는 것을 끝까지 확인한다.

**Acceptance Scenarios**:

1. **Given** 활동 3개가 있는 여행 상세 페이지를 보고 있는 구글 로그인 사용자가 있을 때, **When** 사용자가 "구글 캘린더에 올리기"를 누르고 권한 동의를 마치면, **Then** 해당 사용자의 구글 캘린더에 활동 3개가 각각 이벤트로 만들어지고, 화면의 표시가 "연결됨 · 마지막 반영 시각"으로 바뀐다.
2. **Given** 이미 한 번 캘린더에 올려 둔 여행에서 사용자가 활동을 새로 추가하거나 시간을 바꿨을 때, **When** "다시 반영하기" 버튼을 누르면, **Then** 구글 캘린더에 있던 기존 이벤트들이 최신 내용으로 갱신되고, **같은 활동이 두 번 생기는 일은 일어나지 않는다**.
3. **Given** 사용자가 이 여행의 연결을 끊기로 했을 때, **When** 연결 해제를 확정하면, **Then** 이 여행을 위해 trip-planner가 만든 이벤트들이 구글 캘린더에서 삭제되고, 잠시 후 다시 연결을 원하면 권한 동의 화면을 새로 거치지 않고 재연결할 수 있다(권한이 아직 유효한 기간 동안).

### User Story 2 - 공유 여행에서 멤버가 직접 선택해야 자기 캘린더에 올라간다 (Priority: P2)

여러 사람이 함께 만드는 공유 여행에서, 각 멤버는 자기 캘린더에 올릴지 말지를 **스스로** 결정한다. 한 멤버가 캘린더 연결을 실행한다고 해서 다른 멤버의 구글 캘린더에 저절로 이벤트가 생기지 않는다. 누가 캘린더에 올렸는지 서로 드러나지도 않는다.

**Why this priority**: 공유 여행은 trip-planner의 핵심 특성이다. 한 사람이 시작한 행동이 다른 사람의 개인 캘린더에 영향을 주면, 사용자는 "이 서비스가 내 계정에 무슨 짓을 할지 모르겠다"고 느끼게 된다. 한 번 깨진 이 신뢰는 되찾기 어렵기 때문에 처음부터 원칙을 확실히 해 둔다.

**Independent Test**: 한 여행에 멤버 A, B가 속한 상태에서 A만 자기 브라우저로 연결을 실행한다. 그 뒤 B의 구글 캘린더를 확인해 아무런 변화가 없음을 확인한다.

**Acceptance Scenarios**:

1. **Given** 멤버 A, B가 속한 여행에서, **When** A만 "구글 캘린더에 올리기"를 실행하면, **Then** A의 구글 캘린더에만 이벤트가 생기고 B의 구글 캘린더에는 어떠한 변경도 일어나지 않는다.
2. **Given** A가 이미 자기 캘린더에 올려 둔 여행에 B가 나중에 합류했을 때, **When** B가 여행 상세 페이지를 열어 보면, **Then** B가 보는 화면은 "아직 내 캘린더에는 올라가 있지 않음 · 지금 올리기" 상태이며, A가 올렸다는 사실은 B에게 드러나지 않는다.
3. **Given** 여행 주인이 멤버 B를 여행에서 내보낼 때, **When** 내보내기가 반영되면, **Then** B가 직접 연결 해제를 하지 않는 한 B의 구글 캘린더에 이미 만들어진 이벤트들은 저절로 사라지지 않으며, trip-planner는 이후로 B 캘린더에 어떠한 업데이트도 보내지 않는다.

### User Story 3 - 기존 iCal 사용자의 경험이 이번 변경으로 깨지지 않는다 (Priority: P3)

`install.sh`와 `che-ical-mcp` 플러그인으로 iCal(iCloud '여행' 캘린더) 연동을 쓰고 있던 사용자(주로 개발자/레포 접근자)는 이번 구글 캘린더 연동이 추가된다고 해서 기존 작업 방식이 바뀌거나 망가지지 않는다. 구글 캘린더 연동은 iCal 경로와 **함께** 제공되는 새 경로이며, iCal 경로를 없애거나 방해하지 않는다.

**Why this priority**: 새 기능을 넣는 과정에서 이미 쓰고 있던 기능이 망가지면 기존 사용자에게 가장 실망스러운 상황이 된다. 이 스토리는 "새 기능이 기존 경험을 깨뜨리지 않는다"는 것을 독립적으로 검증하기 위해 존재한다.

**Independent Test**: 이번 배포 전·후 각각, iCal 경로로 여행을 iCloud '여행' 캘린더에 올리는 기존 시나리오를 똑같이 실행하여 결과가 동일함을 확인한다.

**Acceptance Scenarios**:

1. **Given** iCal 경로를 이미 쓰고 있는 사용자가 있을 때, **When** 그동안 쓰던 방식대로 iCal에 여행을 올리면, **Then** 이전과 똑같이 iCloud '여행' 캘린더에 이벤트가 만들어진다.
2. **Given** 한 사용자가 iCal과 구글 캘린더를 둘 다 쓰기로 선택했을 때, **When** 두 경로를 각각 실행하면, **Then** 두 캘린더에 각각 이벤트가 만들어진다. 시스템은 같은 여행이 두 캘린더에 동시에 보이게 되는 상황을 자동으로 정리해 주지 않으며, 문서를 통해 "혼동을 피하려면 한 경로만 쓰는 걸 권한다"고 안내한다(후속 과제로 남김).

### Edge Cases

- 사용자가 권한 동의 창에서 "허용 안 함"을 누르면 화면은 "연결 실패 · 다시 시도" 상태로 돌아가고, 여행 자체의 다른 기능(활동 보기·수정 등)에는 아무 영향이 없다.
- 구글 캘린더 쪽 일시 오류로 일부 활동만 올라가지 못할 수 있다. 이런 경우에도 올라간 것은 그대로 두고, 실패한 활동 목록을 화면에 보여 주며 "다시 시도" 옵션을 제공한다. 여행 원본 데이터는 어떤 경우에도 손상되지 않는다.
- 이미 지나간 날짜의 여행(돌아온 여행)도 그대로 캘린더에 올라간다. 시스템이 과거/미래를 임의로 걸러 내지 않는다.
- 구글 계정에 여러 캘린더(개인/업무 등)가 있을 때는 "이 여행 전용으로 새 캘린더 만들기(기본)"와 "내 기본 캘린더에 바로 넣기" 중에서 사용자가 고른다.
- 여행 이름·활동 제목에 이모지, 오른쪽에서 왼쪽으로 쓰는 언어, 매우 긴 문자열이 들어 있어도 가능한 한 그대로 캘린더에 올린다. 캘린더 화면에서 일부가 잘려 보이는지는 구글 캘린더의 표시 정책을 따른다.
- 사용자가 구글 캘린더에서 이미 직접 고친 이벤트는 재반영 시 덮어쓰지 않는다. 대신 "건너뛴 이벤트 N개"로 사용자에게 알려 사용자가 스스로 정리할 기회를 준다.
- 사용자가 구글 계정을 없애거나 trip-planner의 캘린더 권한을 회수하면, 다음 번 반영 시도에서 "권한이 없어요" 상황을 감지하고 화면에 재연결 CTA를 띄운다. 여행 원본 데이터는 변하지 않는다.
- 공유 여행의 주인이 여행을 삭제했을 때, trip-planner는 각 멤버의 구글 캘린더에 더 이상 어떤 업데이트도 보내지 않는다. 다만 이미 멤버 각자의 캘린더에 올라간 이벤트는 그 멤버가 직접 "연결 해제"를 누르기 전에는 자동으로 사라지지 않는다(다른 사람 캘린더를 저절로 바꾸지 않는다는 원칙이 삭제보다 우선한다).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 구글 로그인된 사용자가 여행 상세 페이지에서 버튼 하나로 본인 구글 캘린더에 여행 활동을 이벤트로 만들 수 있게 해야 한다.
- **FR-002**: 시스템은 현재 구글 로그인 상태를 그대로 유지한 채로 캘린더 쓰기에 필요한 권한만 추가로 동의받을 수 있어야 한다. 이 과정에서 "로그인 화면 → 동의 화면 → 다시 로그인 화면"이 반복되는 현상이 생겨서는 안 되며, 사용자가 동의를 거부해도 원래의 로그인 상태가 손상되지 않아야 한다.
- **FR-003**: 시스템은 연결 시작 시점에 "이 여행 전용으로 새 캘린더 자동 생성(기본값)"과 "내 기본 캘린더에 바로 넣기" 중 하나를 사용자가 고르게 해야 한다.
- **FR-004**: 시스템은 여행에 등록된 **모든 활동**을 구글 캘린더 이벤트로 바꿔서 올린다. 예약 상태로 걸러내지 않는다.
- **FR-005**: 시스템은 같은 여행을 다시 반영할 때, 이미 올려 둔 이벤트는 갱신하고, **같은 활동이 두 개로 늘어나는 일은 일어나지 않아야** 한다.
- **FR-006**: 시스템은 여행에서 활동이 삭제되거나 사용자가 연결 해제를 요청하면, 대응하는 구글 캘린더 이벤트도 삭제한다. 사용자가 구글 캘린더에서 직접 고친 이벤트는 덮어쓰거나 지우지 않고 그대로 두며, 건너뛴 개수를 화면에 표시한다.
- **FR-007**: 공유 여행에서 시스템은 각 멤버가 **직접 자기 손으로 "내 캘린더에 올리기"를 실행했을 때에만** 그 멤버의 구글 캘린더에 이벤트를 만든다. 다른 멤버의 구글 캘린더를 저절로 건드리지 않는다.
- **FR-008**: 시스템은 현재 연결 상태(연결됨/아님, 쓰고 있는 캘린더 이름, 마지막 반영 시각, 실패가 있었는지)를 여행 상세 페이지에서 해당 사용자 본인에게 분명히 보여 준다.
- **FR-009**: 시스템은 네트워크 오류, 권한 회수, 구글 캘린더 쪽 오류가 발생해도 여행 원본 데이터를 망가뜨리지 않으며, 실패한 활동만 "다시 시도" 할 수 있는 상태로 남겨 둔다.
- **FR-010**: 시스템은 웹 화면의 "구글 캘린더에 올리기" 버튼(또는 그와 동등한 진입점)을, 여행 상세 페이지에서 사용자가 스크롤을 많이 내리지 않아도 바로 보이는 곳에 둔다(#150 해소).
- **FR-011**: 시스템은 기존 iCal 경로(`che-ical-mcp`를 통한 흐름)를 없애거나 바꾸지 않는다. 구글 캘린더 연동은 iCal 경로와 독립적으로 동시에 제공된다.
- **FR-012**: 시스템은 구글에 요청하는 권한을 "연결된 캘린더의 이벤트를 만들고·수정하고·삭제하는 권한"에 한정하고, 그 이상을 요청하지 않는다.
- **FR-013**: 시스템은 이벤트 제목을 `[여행명] 카테고리기호 활동제목` 형식으로 만들고, 설명란에는 예약 상태, 위치(지도 링크), 원본 여행 상세 페이지 링크를 넣어, 사용자가 이벤트에서 원본 여행으로 되돌아올 수 있게 한다.

### Key Entities *(데이터 관점에서 추적해야 하는 개념)*

- **연결 상태 (사용자 × 여행)**: 한 사용자가 한 여행에 대해 현재 캘린더에 올려 둔 상태인지, 어떤 캘린더를 쓰고 있는지, 마지막으로 반영한 시각과 실패가 있었는지. 사용자마다 자기 시점에서만 관리된다.
- **이벤트 매핑 (활동 × 구글 캘린더 이벤트)**: trip-planner의 개별 활동이 구글 캘린더의 어떤 이벤트에 대응하는지의 연결 기록. 다시 반영할 때 어떤 이벤트를 갱신·삭제해야 하는지의 기준이 된다. 사용자가 구글 캘린더에서 직접 고쳤는지를 감지하기 위해 구글 캘린더가 이벤트마다 발급하는 ETag(HTTP 표준의 리소스 버전 식별자)를 함께 보관한다.
- **캘린더 권한 상태 (사용자)**: 사용자가 캘린더 쓰기 권한에 동의했는지, 언제 승인·회수했는지의 이력. 로그인 상태와 별개로 관리하여, 권한을 나중에 회수해도 로그인 자체에는 영향을 주지 않는다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 웹으로 처음 들어온 사용자가 여행 상세 페이지에 들어온 뒤 **60초 안에** 본인 구글 캘린더에 여행 이벤트를 만드는 데 성공한다(권한 동의 시간을 포함한 측정).
- **SC-002**: 같은 여행을 다시 반영했을 때 사용자가 캘린더에서 눈으로 볼 수 있는 **중복된 이벤트가 0개**다.
- **SC-003**: 구글 캘린더 연결을 써 본 사용자 가운데 **80% 이상**이 "내 여행을 캘린더에서 쉽게 확인할 수 있었다"는 취지로 긍정 응답한다(사후 설문 또는 UX 인터뷰).
- **SC-004**: 이 기능이 배포된 이후 "웹에서 캘린더 연결 지점을 찾지 못했다" 유형의 피드백이 **0건**이다(최소 한 번의 릴리스 주기 동안 관찰, #150 해소 판정 기준).
- **SC-005**: 공유 여행에서 "나도 모르게 내 구글 캘린더가 바뀌었다"는 보고가 **0건**이다.
- **SC-006**: 구글 캘린더 연동을 쓰지 않고 iCal만 쓰는 기존 사용자에게서 **"예전엔 되던 것이 안 된다"는 유형의 문제 보고가 0건**이다(배포 전·후 동일 동작 확인).

## Assumptions

- 구글 로그인은 이미 기본 인증으로 자리잡았고, 세션에 구글 계정 식별자가 저장되어 있다. 캘린더 쓰기 권한은 별도 동의로 이어서 확보할 수 있다.
- 여행 데이터의 원본은 trip-planner의 데이터베이스이고, 구글 캘린더는 그 내용을 보여 주는 보조 수단이다. 내용이 어긋날 때는 데이터베이스가 옳다고 본다.
- 사용자가 구글 캘린더에서 이벤트를 직접 고쳤다면 그것은 사용자의 뜻이므로, 재반영 시 그 변경을 덮어쓰지 않는다.
- 구글 캘린더 쪽 사용량 한도는 이 피처 규모(여행당 수십 개 이벤트, 개인 사용)에서 충분하다.
- 구글이 아닌 계정(이메일/비밀번호 로그인 등)으로 들어온 사용자는 이번 범위에서는 이 기능을 쓸 수 없다. 대안 제공 여부는 차기 버전에서 결정한다.
- 여행 일정의 시작/종료 시각과 시간대는 #232, #325에서 정상화되어 있어, 구글 캘린더로 올릴 때도 정확한 시각이 유지된다.

## Dependencies

- 기존 구글 로그인 흐름(Auth.js 기반)의 안정성.
- 활동의 시작/종료 시각과 시간대 정확성(#232, #325 기반).
- 공유 여행의 주인/멤버 구조가 정상화된 상태(#188의 보정 이후).
