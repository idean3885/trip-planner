# Feature Specification: 구글 캘린더 권한 제약 감지·안내

**Feature Branch**: `021-gcal-access-guide`
**Created**: 2026-04-22
**Status**: Released (v2.9.2 — 2026-04-22)
**Input**: User description: "외부 캘린더 OAuth Testing 모드 제약으로 Test users 외 사용자의 캘린더 기능 접근 거부를 정확히 감지하고 '개발자 문의' 안내로 전환한다."

## Background

본 앱의 외부 캘린더 OAuth 앱은 **Testing 모드**로 운영되어, 개발자가 Google Cloud Console의 **Test users** 목록에 직접 등록한 계정(최대 100명)만 캘린더 scope 동의를 받을 수 있다. 무료 심사(brand verification) 전 단계이기 때문이며, Production 승급은 Restricted scope 심사 비용이 커서 당분간 보류된다.

결과적으로 **미등록 사용자는 캘린더 기능을 전혀 사용할 수 없다**:

- 주인이 "공유 캘린더 연결"을 눌러 OAuth 동의 페이지로 가면, Test users에 없으면 외부 OAuth 제공자가 access_denied로 차단.
- 호스트·게스트가 "내 구글 캘린더에 추가"를 누를 때도 동일한 동의가 필요하며, 역시 미등록이면 차단될 것으로 예상된다(정확한 거부 시점·응답은 구현 중 확인).

현재 UI는 이 거부를 일반 실패로 처리해 **"연동에 실패했습니다"** 수준의 문구만 노출하므로, 사용자는 "내 계정이 Test users에 없어서"라는 구체 원인을 알 수 없고 해결 경로(개발자에게 등록 요청)도 찾지 못한다. 이번 피처는 이 제약을 **1급 UX 상태**로 승격해, 미등록 사용자가 오류를 보는 대신 "개발자 등록이 필요한 기능"으로 이해하게 한다.

Google Calendar 기능 전체의 실 사용성은 약하다(실사용 가능한 사용자 ≤ 100명, 수동 등록 필요). 따라서 v2.9.x 시점에서는 **"심사 전 제약이 존재한다"는 사실을 정직하게 전달**하는 것이 최선이며, 앱 심사 자체는 별도 의사결정(비용/시간/범위) 트랙이다.

## Clarifications

1. **용어 정합** — 본 스펙의 역할 호칭은 [`docs/glossary.md`](../../docs/glossary.md) 정본(주인/호스트/게스트)을 따른다.
2. **"미등록"의 정의** — 외부 OAuth 제공자의 Test users 목록에 계정 이메일이 포함되지 않은 상태. 본 스펙에서 "미등록 사용자"는 이 상태의 사용자를 뜻하며, 앱 내 트립 역할과 무관하다.
3. **subscribe 경로의 거부 시점** — 비-Test user가 `calendarList.insert`까지 도달할 수 있는지(토큰 취득 단계에서 차단될 것으로 예상)는 구현 중 리서치 과제로 두고, 스펙은 **"어느 단계에서 거부되든 UI가 동일한 안내 상태로 귀결된다"** 는 원칙만 고정한다.

### Session 2026-04-22

- Q: 미등록 감지 시 안내에 어느 경로를 노출할까? → A: **공개 토론 채널의 Q&A 카테고리 프리필 링크** (`/discussions/new?category=q-a`). 이유: Test user 등록 요청은 질문·요청 성격이라 버그 트래킹과 분리가 낫고, public repo 크롤링 스팸 리스크가 없는 로그인 필수 경로이며, 답변 템플릿화가 자연스럽다.
- Q: 안내 UI 형태는? → A: **단일 버튼 CTA** 하나만 노출하고 외부 링크 아이콘을 덧댄다. 복수 채널(메일·README 링크 병렬)은 의사결정 부담을 늘려 채택하지 않는다.

## User Scenarios & Testing *(mandatory)*

### 용어

- **Testing 모드 제약**: 본 앱의 외부 캘린더 OAuth 앱이 심사 전 상태이므로 Test users에 등록된 계정만 캘린더 scope 동의가 가능한 상태.
- **미등록 사용자**: Test users 목록에 이메일이 없는 현재 로그인 사용자.
- **권한 제약 안내 카드**: 미등록 사용자에게 표시되는 "이 기능은 현재 개발자 등록 사용자에게만 제공됩니다" 상태 카드(토스트 단문이 아님).

### User Story 1 — 미등록 주인이 공유 캘린더 연결을 시도 (Priority: P1)

미등록 상태의 주인이 "공유 캘린더 연결" CTA를 눌렀을 때, 일반 실패 토스트 대신 **권한 제약 안내 카드**가 표시되어 "왜 실패했는지 + 무엇을 하면 되는지"를 한 번에 이해할 수 있다.

**Why this priority**: 캘린더 진입의 주 경로이며 실패 빈도가 가장 높다. 현재 사용자 혼란을 가장 많이 유발하는 지점.

**Independent Test**: Test users에 등록되지 않은 Google 계정을 주인으로 갖는 트립을 준비 → 해당 계정으로 "공유 캘린더 연결" 클릭 → 안내 카드 노출 + 일반 실패 토스트 미노출 확인.

**Acceptance Scenarios**:

1. **Given** 주인이 미등록 상태, **When** OAuth 동의 페이지에서 access_denied 후 복귀, **Then** 캘린더 섹션 다이얼로그가 권한 제약 안내 카드로 전환된다.
2. **Given** 주인이 미등록 상태, **When** 동의 없이 토큰이 이미 있는 특수 경로로 Calendar API가 403을 반환, **Then** 서버는 이 상태를 "REVOKED"가 아닌 **별도의 미등록 상태**로 응답한다.

---

### User Story 2 — 미등록 호스트·게스트의 캘린더 접근 (Priority: P2)

미등록 호스트·게스트가 캘린더 섹션 트리거 또는 "내 구글 캘린더에 추가"를 눌렀을 때, 주인과 동일한 형태의 권한 제약 안내 카드가 노출된다.

**Why this priority**: 호스트·게스트는 본인 캘린더에 추가할 때만 scope 동의가 필요하므로 도달 빈도는 주인보다 낮다. 하지만 일관된 UX 유지는 신뢰 유지에 필수.

**Independent Test**: 미등록 Google 계정을 호스트로 갖는 트립에서 캘린더 트리거 → 안내 카드 확인. (subscribe 시점 거부인지 트리거 이전 차단인지는 구현 중 결정되며, 카드 형태는 주인과 동일.)

**Acceptance Scenarios**:

1. **Given** 호스트가 미등록 상태, **When** "내 구글 캘린더에 추가"를 시도, **Then** 안내 카드가 노출되고 실제 외부 호출은 발생하지 않거나 발생 시 403이 안내로 변환된다.
2. **Given** 게스트가 미등록 상태, **When** 캘린더 트리거 클릭, **Then** 동일한 안내 카드만 노출(게스트가 취할 수 있는 액션 없음).

---

### User Story 3 — 사전 인지 가이드 (Priority: P2)

앱 랜딩 페이지 또는 설정/가이드 문서에 Testing 모드 제약이 명시되어, 사용자가 캘린더 기능을 쓸 수 있는 조건을 **기능 시도 전에** 이해할 수 있다.

**Why this priority**: 안내 카드는 사후 설명. 사전 인지가 있으면 불필요한 클릭과 혼란이 줄어든다. 문서 한 블록 추가로 달성.

**Independent Test**: 앱 랜딩 또는 설정 화면에서 "Google 캘린더 연동은 현재 개발자 등록 사용자에게만 제공됩니다" 수준의 문구를 발견할 수 있음.

**Acceptance Scenarios**:

1. **Given** 신규 사용자 진입, **When** 랜딩/설정 페이지 열람, **Then** Testing 모드 제약 안내 블록을 확인할 수 있다.
2. **Given** 기존 사용자, **When** README 또는 설치 가이드 문서 열람, **Then** 동일한 제약 설명을 찾을 수 있다.

---

### Edge Cases

- **이미 Test users에 등록된 사용자**: 기존 동작 그대로. 안내 카드는 나타나지 않는다.
- **Test users에서 제거된 사용자**: 다음 호출에서 403 → 안내 카드로 전환. 기존 REVOKED(외부 콘솔에서 사용자가 권한 회수)와 구분해 표기.
- **OAuth 동의 플로우에서 access_denied가 아닌 다른 오류**: 일반 실패로 처리(본 피처 범위 밖).
- **주인은 Test user이고 호스트가 미등록**: 주인의 공유 캘린더 연결은 성공, 호스트는 자기 "추가" 시도 시 안내 카드. 주인 UI에는 "일부 동행자가 개발자 등록이 필요합니다" 힌트는 **제공하지 않는다**(프라이버시·범위 제한).
- **심사 승급 완료 후**: 감지 로직은 유지하되 access_denied가 발생하지 않아 자연스럽게 노출되지 않음. 피처 자체 제거는 불필요.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 주인의 공유 캘린더 연결 또는 sync 시도에서 OAuth access_denied 또는 Calendar API 403이 발생하면 이를 **"미등록 상태"** 로 식별해야 하며, 기존 REVOKED(권한 회수)와 구분되는 별도 상태여야 한다.
- **FR-002**: 시스템은 호스트·게스트의 subscribe 또는 관련 조작에서 동일 원인의 거부가 발생하면 FR-001과 **동일한 미등록 상태**로 식별해야 한다.
- **FR-003**: 시스템은 미등록 상태가 감지되면 역할과 무관하게 **권한 제약 안내 카드**(일반 실패 토스트 아님)를 UI에 노출해야 하며, 카드 문구는 "이 기능은 현재 개발자 등록 사용자에게만 제공됩니다. 개발자에게 문의해 주세요." 수준의 의미를 전달해야 한다.
- **FR-004**: 시스템은 미등록 상태의 사용자가 기능을 반복 시도해도 **추가 외부 호출을 유발하지 않도록** 상태를 기억하거나 즉시 차단한다.
- **FR-005**: 시스템은 앱의 **랜딩 페이지·설정 화면·설치 가이드 중 최소 하나**에 Testing 모드 제약을 사전 고지하는 블록을 포함해야 한다.
- **FR-006**: 안내 카드는 역할(주인/호스트/게스트)에 상관없이 **동일한 구조**를 가진다(spec 020의 "설명 + 닫기" 다이얼로그 패턴을 준수).
- **FR-007**: 미등록 상태는 사용자 계정 수준의 조건이므로, 해당 사용자가 다른 여행에 접근할 때도 **일관되게** 감지되어야 한다.
- **FR-008**: 안내 카드의 문의 경로는 **리포의 공개 토론 채널(Q&A)로 프리필된 새 토론 생성 링크**로 제공한다. 이메일·다른 채널은 노출하지 않는다(Clarification Session 2026-04-22).
- **FR-009**: 안내 카드의 문의 액션은 **단일 버튼 CTA** 하나만 노출하고, 외부 링크임을 나타내는 시각적 표식(아이콘 또는 동등한 접근성 표시)을 수반한다(Clarification Session 2026-04-22).
- **FR-010**: 본 피처는 캘린더 기능의 동작 로직(연결·sync·subscribe 성공 경로)을 변경하지 않는다. **오류 분류와 UI 전환**만 추가한다.

### Key Entities

- **미등록 상태 신호**: "이 계정이 Testing 모드 제약으로 캘린더 기능을 사용할 수 없다"는 사실. 서버 응답(캘린더 상태 조회·연결·subscribe)에서 식별 가능한 형태로 전달되며, 기존 `GCalLastError`와 동급의 병렬 상태로 다룬다.
- **Testing 모드 제약 가이드 블록**: 사용자가 사전 인지할 수 있도록 랜딩/설정/문서에 배치되는 텍스트 블록.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Testing 모드 제약으로 인한 거부가 **일반 실패 토스트로 잘못 노출되는 비율 = 0%**(배포 후 dev 재현 1회 확인).
- **SC-002**: 미등록 상태 자동 테스트 ≥ 2건(서버 오류 분류 + UI 렌더 분기).
- **SC-003**: 미등록 사용자가 안내 카드를 본 뒤 "어떻게 해결해야 하는지" 문구만으로 이해 가능(dev 재현 중 정성 확인 1회).
- **SC-004**: 사전 고지 블록이 랜딩/설정/README **중 최소 한 곳에** 존재한다.
- **SC-005**: 본 피처로 인해 기존 캘린더 기능(연결·sync·subscribe)의 성공 경로 동작이 변경되지 않는다(회귀 0 — 기존 단위 테스트 통과).

## Assumptions

- 외부 캘린더 OAuth 앱 심사 승급은 본 피처 범위 밖 — 별도 의사결정 트랙.
- Test users 등록은 **수동 절차**(Google Cloud Console 편집)로 유지되며 앱 내 자동화는 도입하지 않는다.
- 안내 카드의 문구 톤은 [`docs/glossary.md`](../../docs/glossary.md) 정본 용어를 따른다(주인/호스트/게스트).
- 기존 스펙 018/019/020의 역할·엔드포인트 구조는 본 피처에서 변경하지 않는다. 오류 분류 한 종 추가 + UI 분기 한 종 추가.
- subscribe 경로에서 비-Test user가 `calendarList.insert`까지 도달하는지는 불확실하며, 도달하지 못하더라도 UI 귀결 상태는 동일해야 한다(FR-002).

## Non-Goals

- Google 앱 심사 수행(브랜드 확인·Restricted scope 심사).
- Test users 목록을 앱에서 편집·동기화하는 기능.
- iCloud/Apple Calendar 등 대체 캘린더 연동(spec 018 별도 트랙).
- Test users에 등록된 사용자 수 모니터링·경고.

## Future Notes

- **앱 심사 승급 시**: FR-001/FR-002 감지 로직은 유지(심사 후에도 403은 발생 가능). 사전 고지 블록 문구만 "심사 완료 — 일반 사용자 지원"으로 교체.
- **안내 카드 CTA 확장**: 현재는 문의 경로만. 장래에 "Test user 등록 요청 폼"(구글 폼/Discussions) 연동이 가능해지면 CTA를 추가.
