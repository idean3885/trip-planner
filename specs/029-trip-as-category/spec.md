### 결정 1 — 명목 기간 vs derived 기간 충돌 정책 ✅

사용자가 최초 입력한 기간 외부의 일정을 추가하면 여행 기간이 자동으로 확장됩니다. 사용자 별도 확인 alert는 노출하지 않습니다. 명목 기간은 첫 일정이 추가되기 전까지의 fallback 표시값으로만 의미를 가지며, 그 이후는 derived 값이 정본입니다.

### 결정 2 — 사이드 체크박스 초기 상태 ✅

여행 상세 진입 시 사이드 체크박스는 **현재 여행만 체크된 상태**가 default입니다. 사용자가 사이드에서 다른 여행을 체크하면 통합 캘린더 뷰로 확장됩니다. 여행 상세 URL의 context가 그 여행이므로 mental model 정합. 캘린더 ↔ 목록 토글은 없습니다(캘린더 영역 항상 노출).

### 결정 3 — MCP/API 호환성 정책 ✅

`create_trip` / `update_trip` 시그니처에서 `startDate` / `endDate` 파라미터를 **breaking 제거**합니다. MAJOR 범프(v3.0.0)로 처리합니다. 명목 기간은 첫 일정 추가 전까지 여행 메타데이터 fallback으로만 의미를 갖고, MCP·API 입력으로 받지 않습니다(필요 시 첫 일정을 함께 생성).

배경: 협력자가 현재 MCP 자동 부트스트랩 누락으로 도구를 쓰지 못하는 상태이며, 같은 마일스톤(v3.0.0)에서 부트스트랩까지 함께 정리하므로 breaking 영향이 묶입니다.

### 결정 4 — MCP 자동 부트스트랩 강화 ✅

협력자가 Claude(또는 다른 AI client)에 "trip-planner MCP 설치/업데이트해줘"라고 자연어로 요청하면 사용자 노터치로 설치·인증(PAT 자동 발급)·등록·동작 검증까지 완료되는 자동 부트스트랩이 필요합니다. 도메인 분리상 별도 spec(spec 030)으로 신설하되 **같은 마일스톤 v3.0.0에 동시 출시**합니다. spec 029(여행 모델 + 캘린더 뷰)와 spec 030(MCP 자동 부트스트랩)이 v3.0.0 묶음입니다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 여행 기간 동적 갱신 (Priority: P1)

여행 사용자는 여행을 만들 때 최초 기간을 정합니다. 이후 일정을 추가·삭제하면 여행 기간 표시가 자동으로 갱신됩니다. 사용자는 여행 헤더의 "2026.6.7. ~ 2026.6.21." 같은 표시가 항상 실제 일정 범위와 정합한다고 신뢰할 수 있습니다.

**Why this priority**: 현행 모델의 가장 큰 마찰. 사용자가 일정을 추가했는데 여행 헤더 날짜가 갱신 안 되면 두 정보 출처 사이에 신뢰가 깨집니다. 캘린더 뷰가 derived 기간에 의존하므로 P1로 처리해야 P2·P3가 의미를 갖습니다.

**Independent Test**: 사용자가 여행을 새로 만들고 최초 기간 외부의 날짜로 일정을 추가하면 여행 헤더 날짜가 그 일정을 포함하는 범위로 갱신됨을 확인합니다.

**Acceptance Scenarios**:

1. **Given** 여행 "신혼여행"이 2026-06-07 ~ 2026-06-21로 생성된 상태, **When** 사용자가 2026-06-22 일정을 추가, **Then** 여행 헤더가 2026-06-07 ~ 2026-06-22로 갱신됩니다.
2. **Given** 일정 5건이 등록된 여행, **When** 가장 마지막 날짜 일정을 삭제, **Then** 여행 헤더 종료일이 그 직전 일정 날짜로 갱신됩니다.
3. **Given** 일정 0건의 여행, **When** 첫 일정을 추가, **Then** 여행 헤더 날짜가 그 일정 단일 날짜로 표시됩니다.

---

### User Story 2 - 여행 상세 캘린더 + 일정 리스트 (Priority: P2)

사용자는 여행 상세에 진입하면 캘린더가 항상 노출됩니다. 캘린더 ↔ 목록 토글은 없습니다. 월별 미니 캘린더에서 여행 기간 강조 + 오늘 강조를 보고, 날짜 클릭 시 그 날의 일정이 desktop은 사이드 영역에, mobile은 하단 영역에 표시됩니다. 두 영역(캘린더·일정 리스트)은 동시에 노출되며 사용자가 닫을 수 없습니다.

**Why this priority**: 사용자 mental model이 "DAY N"보다 절대 날짜 우선이라는 핵심 요구를 직접 해결합니다. P1 derived 기간 위에 동작하므로 P2.

**Independent Test**: 여행 상세 진입 시 캘린더가 항상 노출되는지, 날짜 클릭이 사이드/하단 영역의 일정 콘텐츠를 갱신하는지 확인합니다.

**Acceptance Scenarios**:

1. **Given** 여행 상세 페이지를 desktop(≥1024px) viewport에서 진입, **When** 페이지가 로드, **Then** 캘린더가 한쪽 영역, 사이드 일정 리스트가 다른 쪽 영역으로 분할(split) 노출됩니다.
2. **Given** desktop 캘린더, **When** 사용자가 6월 9일을 클릭, **Then** 사이드 리스트가 6월 9일 일정 목록으로 갱신됩니다. 사이드 영역 자체는 사라지지 않습니다.
3. **Given** 여행 상세를 mobile(<1024px) viewport에서 진입, **When** 페이지가 로드, **Then** 캘린더 상단 + 하단 일정 리스트 stacked 레이아웃이 노출됩니다.
4. **Given** mobile 캘린더에서 6월 9일을 클릭, **When** 하단 리스트가 6월 9일 일정으로 swipe 전환, **Then** 좌 스와이프 제스처로 기본 리스트(여행 전체 DAY 목록)로 복귀할 수 있습니다. URL은 변하지 않습니다.

---

### User Story 3 - 여러 여행 통합 캘린더 + 체크박스 토글 (Priority: P3)

사용자는 캘린더 뷰에서 자기가 속한 여러 여행을 한 화면에 봅니다. 각 여행은 구글 캘린더처럼 체크박스로 표시/숨김을 켜고 끌 수 있습니다. 일정 카드에는 어느 여행 소속인지 라벨(`[신혼여행]` 등)이 노출됩니다.

**Why this priority**: 여행의 의미가 청첩장 모임 같은 단발 행사까지 확장됩니다. 통합 뷰가 있어야 "내 일정 전체"라는 사용자 mental model이 살아납니다. 단 P1·P2 없이 단독으로 의미가 약하므로 P3.

**Independent Test**: 사용자가 둘 이상의 여행에 속해 있을 때 통합 캘린더에서 두 여행의 일정이 같은 날에 함께 보이는지, 체크박스를 끄면 해당 여행의 일정만 사라지는지 확인합니다.

**Acceptance Scenarios**:

1. **Given** 사용자가 "신혼여행"(2026-06-07~6-21)·"동창회"(2026-06-15) 두 여행에 속한 상태, **When** 통합 캘린더 화면 진입, **Then** 6월 15일 칸에 두 여행의 일정이 함께 표시되고 각 카드에 여행 라벨이 노출됩니다.
2. **Given** 두 여행 체크박스가 모두 켜진 상태, **When** "동창회" 체크박스를 끔, **Then** 동창회 일정이 캘린더에서 즉시 사라집니다(다른 여행은 유지).

---

### Edge Cases

* 일정 0개의 여행을 캘린더에서 어떻게 표시할지 — 본 spec에서는 "일정 미정" 카드로 여행 목록에 노출하되 캘린더 그리드에는 표시하지 않습니다. derived 기간은 null로 둡니다.
* 사용자가 최초 입력 기간 외부의 일정을 추가하면 (US1 시나리오 1) — 여행 기간이 자동 확장됩니다(사용자 확인 alert 없음, 결정 1 ✅).
* 여행에 등록된 일정이 viewport timezone 경계를 넘는 경우 — Activity의 `startTimezone`을 그대로 따르며 캘린더 표시는 사용자 viewport(브라우저) 로컬 날짜 기준. (현행 timezone 정책 유지, 본 spec 변경 없음)
* mobile에서 사이드 영역 swipe 전환 중 뒤로가기 — 좌 스와이프 시 즉시 기본 리스트로 복귀. 브라우저 뒤로가기 버튼은 여행 상세 이전 페이지로 이동(현행 동작 유지).
* desktop split layout에서 사이드 영역 사용자가 닫을 수 있는지 — 본 spec에서는 사이드 영역 닫기 버튼 없음(항상 노출). 향후 별도 spec.

## Requirements *(mandatory)*

### Functional Requirements

#### 데이터 모델

* **FR-001**: 여행의 시작·종료 날짜는 등록된 일정 집합에서 derived 값으로 계산됩니다(최소 일정 날짜 = 시작, 최대 일정 날짜 = 종료).
* **FR-002**: 일정 0건의 여행의 derived 기간은 null이며, UI는 "일정 미정"으로 표시합니다.
* **FR-003**: 사용자가 여행을 최초 생성할 때 명목 기간(시작·종료 날짜)을 입력합니다. 이 값은 첫 일정이 추가되기 전까지 표시·캘린더 import 범위·공유 캘린더 푸시 범위에 쓰입니다.
* **FR-004**: 일정 추가·수정·삭제 시 여행 derived 기간이 즉시 갱신됩니다. 최초 입력한 명목 기간 외부의 일정이 추가되면 명목 기간을 derived 기간으로 자동 확장합니다(사용자 확인 alert 없음). 새로고침 없이 UI에 반영됩니다.

#### 여행 상세 화면

* **FR-005**: 여행 상세는 캘린더 영역이 항상 노출됩니다. 캘린더 ↔ 목록 토글은 없습니다. 일정 리스트 영역(사이드 또는 하단)도 항상 함께 노출되며 사용자가 닫을 수 없습니다.
* **FR-006**: (제거 — 토글 자체가 없으므로 마지막 뷰 복원 prefs는 필요 없습니다.)
* **FR-007**: 캘린더 영역은 월별 미니 캘린더 + 여행 기간 강조 + 오늘 강조를 표시합니다. 사용자는 월을 앞뒤로 이동할 수 있습니다.
* **FR-008**: 사용자가 캘린더에서 날짜를 클릭하면 그 날의 일정 목록이 사이드(desktop) 또는 하단(mobile) 영역에 표시됩니다.

#### 레이아웃 — viewport 정본 `docs/glossary.md` 참조

* **FR-009**: **desktop**(viewport ≥1024px)에서 캘린더 뷰는 split 레이아웃(캘린더 + 사이드 리스트)입니다. 사이드 영역은 사용자가 닫지 않는 한 항상 노출되며 본 spec에는 닫기 버튼이 없습니다.
* **FR-010**: **mobile**(viewport <1024px)에서 캘린더 뷰는 stacked 레이아웃(캘린더 상단 + 리스트 하단)입니다. 날짜 클릭 시 하단 리스트가 그 날의 일정으로 in-place swap됩니다(URL 변경 없음).
* **FR-011**: mobile에서 좌 스와이프 제스처는 in-place swap의 "뒤로 가기" 인터랙션입니다(기본 리스트로 복귀). 브라우저 history는 변경하지 않습니다.

#### 통합 캘린더 (P3)

* **FR-012**: 사용자는 자기가 속한 여러 여행을 한 캘린더 화면에서 동시에 볼 수 있습니다. 여행 상세 진입 시 default는 현재 여행만 체크된 상태이며, 사용자가 사이드 체크박스로 다른 여행을 추가 체크하면 통합 뷰로 확장됩니다.
* **FR-013**: 각 여행은 사이드에 체크박스로 노출되어 사용자가 표시·숨김을 토글할 수 있습니다. 체크박스 상태는 사용자 prefs로 보존됩니다.
* **FR-014**: 일정 카드는 어느 여행 소속인지 여행 라벨(예: `[신혼여행]`)을 표시합니다.

#### 외부 인터페이스

* **FR-015**: 외부 캘린더 import 기간(Apple/Google → ActivityDraft)은 derived 기간을 사용합니다. 일정 0건의 여행은 import 범위가 없어 import 시작 시 "일정 0건이라 기간이 정해지지 않았습니다" 안내로 막힙니다.
* **FR-016**: 여행 공유 캘린더 push 기간도 derived를 사용합니다(동일).
* **FR-017**: MCP `create_trip` / `update_trip` 시그니처에서 `startDate` / `endDate` 파라미터는 **제거**됩니다(v3.0.0 MAJOR 범프). 여행 생성 시 명목 기간이 필요하면 사용자가 첫 일정을 함께 생성해 derived 기간으로 부여합니다.
* **FR-018**: OpenAPI `Trip` 응답의 startDate·endDate는 derived 값으로 노출됩니다. 응답 스키마 자체는 호환됩니다(필드 위치·타입 유지, 값 출처만 변경).

### Key Entities *(include if feature involves data)*

* **Trip** — 일정 묶음의 최상위 카테고리. 정체성은 제목·주인·동행자입니다. 시작·종료 날짜는 derived 값(또는 일정 0건 시 명목 기간 / null).
* **Day** — 일정 1건. 여행에 속하며 절대 날짜를 가집니다. derived 기간 계산의 단위입니다.
* **Trip 명목 기간** — 최초 생성 시 입력값. 첫 일정 추가 전까지의 표시·import 범위 기본값.
* **사용자 뷰 prefs** — 사용자가 마지막으로 선택한 뷰(캘린더/목록), 여행 체크박스 표시 상태.

## Success Criteria *(mandatory)*

### Measurable Outcomes

* **SC-001**: 일정 추가·삭제 후 여행 헤더 날짜가 사용자가 새로고침 없이 즉시 갱신됨이 확인됩니다. (US1 기능 검증)
* **SC-002**: 여행 상세 진입 시 사용자가 "오늘 며칠?", "복귀날 언제?"를 캘린더 한 화면에서 확인할 수 있습니다(스크롤·페이지 이동 없이). (US2 사용성 검증)
* **SC-003**: 사용자가 둘 이상의 여행을 가진 경우, 한 캘린더 화면에서 일정 충돌이 즉시 시각화됩니다(같은 날 칸에 두 여행 카드가 동시 노출). (US3 기능 검증)
* **SC-004**: 새로운 여행을 만들고 일정 5개를 추가하는 사용자 시나리오를 평균 3분 이내에 완료할 수 있습니다.
* **SC-005**: mobile viewport에서 캘린더 ↔ 일정 swap 인터랙션이 1초 이내에 시각적으로 응답합니다.
* **SC-006**: derived 기간 정책 도입 후 여행 헤더 표시와 실제 일정 범위가 어긋난다는 사용자 보고가 0건입니다(릴리즈 후 1개월 관찰).

## Assumptions

* **마일스톤 v3.0.0 묶음** — spec 029(여행 모델 + 캘린더 뷰) + spec 030(MCP 자동 부트스트랩) 두 spec이 v3.0.0 MAJOR 범프에 동시 출시됩니다. spec 030은 별도 작성 예정.
* 여행 최초 명목 기간 입력 폼은 **웹 UI에서만** 유지(사용자가 첫 일정 추가 전 표시·import 범위 기본값으로 필요). MCP·API 입력으로는 받지 않습니다.
* 일정 0건의 여행은 여행 목록 카드에서 "일정 미정"으로 표시되며 캘린더 그리드의 어느 칸에도 노출되지 않습니다.
* 연간 캘린더(이미지 3, Calendar 2026 #858)는 본 spec 029 범위 밖. 별도 후속 spec에서 다룹니다. 본 spec은 월별 캘린더만 다룹니다.
* 청첩장 모임 같은 의미 확장은 명칭 "여행" 유지 + 사용자 본인이 자유 입력. UI에서 카테고리 태그·아이콘 분기는 별도 후속 spec에서 다룹니다.
* viewport 분기점은 `docs/glossary.md`의 desktop ≥1024px 정본을 따릅니다. md(768px) 분기는 본 spec에서 사용하지 않습니다.
* `connectCalendar`의 OWNER 권한 정책, 외부 캘린더 import 흐름(Apple/Google → ActivityDraft), shadcn/Tailwind v4 토큰 시스템은 본 spec에서 변경하지 않습니다.
