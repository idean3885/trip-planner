# Feature Specification: 활동 컬렉션 읽기 REST 표현 정비

**Feature Branch**: `052-activity-collection-read`
**Created**: 2026-06-02
**Status**: Draft
**Input**: 활동 컬렉션을 읽었을 때 개수만 반환되어 활동 표현과 식별자를 얻을 수 없다 — 멤버 리소스를 표현으로 돌려주도록 정비(BE 우선, MCP 차선).

## Clarifications

1. **결함 범위 — 트립 레벨은 개수만, 일자 레벨은 표현 제공** — 트립 단건 읽기는 일자별 **활동 개수**만 돌려주고 활동 표현이 없다. 일자 단위 읽기는 활동 표현 전체를 돌려주지만 **API 문서에 노출돼 있지 않아** 소비자가 발견할 수 없다. 따라서 결함은 "트립 레벨 활동 표현 부재" + "활동 표현 읽기의 미문서화(발견 불가)" 두 가지다.
2. **트립 단위 일괄 읽기 방식은 plan에서 결정** — 트립의 활동을 한 번에 읽게 할지(일괄), 아니면 일자별 읽기를 문서화해 발견 가능하게만 할지의 구체 수단은 plan에서 정한다. spec은 "발견 가능한 경로로 활동 표현을 읽을 수 있어야 한다"까지만 규정한다.
3. **범위 제외** — MCP 도구 변경은 이 피처 범위 밖(차선, 추후 재검토). BE에서 파괴적 작업을 미리 시뮬레이션하는 dry-run도 도입하지 않는다(미리보기는 FE UX 영역).

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 트립의 활동을 식별자와 함께 읽기 (Priority: P1)

API 소비자(직접 호출자·에이전트)가 한 여행에 속한 활동들을 **각 활동의 식별자와 내용 필드까지** 읽을 수 있어야 한다. 지금은 여행을 읽으면 일자별 활동 **개수**만 돌아와, 특정 활동을 가리켜 수정·삭제·정렬하는 후속 작업이 불가능하다.

**Why this priority**: 읽기에서 식별자를 못 얻으면 모든 쓰기·삭제 워크플로우가 우회로 빠진다(실제로 활동 식별자를 못 얻어 일자 전체 삭제로 우회한 사례 발생). 읽기 표현은 후속 작업의 선행 조건이라 최우선.

**Independent Test**: 활동이 있는 여행을 문서화된 읽기 경로로 호출해, 응답에 각 활동의 식별자와 필드가 담겨 오는지 확인한다. 그 식별자로 활동을 가리킬 수 있으면 가치가 입증된다.

**Acceptance Scenarios**:

1. **Given** 활동이 여러 개 있는 여행, **When** 소비자가 그 여행의 활동을 읽으면, **Then** 각 활동의 식별자와 내용 필드(분류·제목·시간범위·시간대·장소·메모·비용/통화·예약상태·정렬순서)가 표현으로 반환된다.
2. **Given** 활동이 0개인 일자, **When** 그 일자의 활동을 읽으면, **Then** 오류가 아니라 빈 목록이 반환된다.
3. **Given** 자신이 멤버가 아닌 여행, **When** 그 여행의 활동을 읽으려 하면, **Then** 접근이 거부된다.

---

### User Story 2 - 활동 읽기를 API 문서에서 발견 (Priority: P2)

API 소비자가 **소스 코드를 보지 않고도** 활동을 표현으로 돌려주는 읽기 경로를 API 문서에서 찾아 쓸 수 있어야 한다. 지금은 활동 표현을 주는 읽기가 문서에 없어, 존재 자체를 알 수 없다.

**Why this priority**: 문서에 없으면 소비자·에이전트는 그 경로를 신뢰하거나 발견할 수 없어, 실질적으로 없는 것과 같다. 표현을 주더라도 발견 불가면 P1의 가치가 닫힌다.

**Independent Test**: API 문서만 보고 활동 표현 읽기 경로와 응답 형태를 파악해 호출에 성공하는지 확인한다.

**Acceptance Scenarios**:

1. **Given** API 문서, **When** 활동을 읽는 경로를 찾으면, **Then** 요청 형식과 응답 스키마(활동 필드 포함)와 예시가 문서에 있다.
2. **Given** 문서의 응답 스키마, **When** 실제 응답과 대조하면, **Then** 활동 필드 구성이 문서와 일치한다.

---

### User Story 3 - 트립 일정을 한 번에 개관 (Priority: P3)

소비자가 한 여행의 일정(일자 + 각 일자의 활동)을 **발견 가능한 방식으로 한 번에** 개관할 수 있어야 한다. 지금은 여행을 읽은 뒤 일자마다 따로 호출해야 활동을 모을 수 있고, 그 일자별 읽기도 문서화돼 있지 않다.

**Why this priority**: 호출 수·왕복을 줄여 에이전트 토큰·실패율을 낮춘다. 가치 있지만 P1·P2가 갖춰지면 점진 개선으로 둘 수 있어 P3.

**Independent Test**: 한 여행의 전체 일정(일자+활동)을 모을 때 필요한 호출 수가 목표 이하인지 측정한다.

**Acceptance Scenarios**:

1. **Given** 여러 일자에 활동이 있는 여행, **When** 소비자가 일정을 개관하려 하면, **Then** 문서화된 경로만으로 활동을 모을 수 있다.

---

### Edge Cases

- 활동이 0개인 일자/여행 → 오류가 아니라 빈 표현.
- 존재하지 않는 여행·일자 식별자 → not-found.
- 멤버가 아닌 사용자의 접근 → 거부.
- 활동이 많은 여행 → 표현 읽기가 일관된 형태로 반환(정렬 순서 포함).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 권한 있는 소비자가 한 여행에 속한 활동들을 **완전한 리소스 표현**으로 얻게 해야 한다. 각 표현은 안정적 식별자와 내용 필드를 포함한다.
- **FR-002**: 활동 표현을 반환하는 읽기 경로는 **공개 API 문서에 노출**되어야 한다(요청 형식·응답 스키마·예시). 소비자가 소스나 미문서화 경로에 의존하지 않게 한다.
- **FR-003**: 시스템은 한 여행의 활동 내용을 **발견 가능한 경로로 읽을 수 있게** 해야 한다(개수만 노출하는 상태를 해소). 일괄 읽기/일자별 읽기 문서화 중 어떤 수단을 쓸지는 plan에서 정한다.
- **FR-004**: 활동 읽기는 후속 작업(식별·수정·삭제·정렬)에 필요한 필드를 포함해야 한다: 식별자, 분류, 제목, 시작/종료 시각, 시작/종료 시간대, 장소, 메모, 비용/통화, 예약상태, 정렬순서.
- **FR-005**: 빈 컬렉션은 오류가 아니라 빈 표현을 반환하고, 존재하지 않는 여행·일자는 not-found, 권한 없는 접근은 거부되어야 한다.
- **FR-006**: 본 피처는 MCP 도구 변경과 BE dry-run/시뮬레이션을 포함하지 않는다(범위 제외).

### Key Entities *(include if feature involves data)*

- **여행(Trip)**: 일정의 최상위 단위. 여러 일자를 가진다. 현재 읽기는 일자별 활동 개수만 노출한다.
- **일자(Day)**: 여행의 하루. 여러 활동을 가진다.
- **활동(Activity)**: 일자에 속한 일정 항목. 식별자 + 내용 필드(분류·제목·시간범위·시간대·장소·메모·비용/통화·예약상태·정렬순서)로 표현된다. 본 피처의 핵심 표현 대상.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 소비자는 **문서화된 경로만으로** 한 여행의 모든 활동을 식별자와 함께 얻을 수 있다.
- **SC-002**: 활동 표현을 반환하는 모든 읽기 경로가 API 문서에 응답 스키마 + 예시와 함께 노출된다(노출율 100%).
- **SC-003**: 한 여행의 전체 일정(일자+활동)을 모으는 데 필요한 문서화된 호출 수가 정해진 목표 이하다(목표 수치는 plan에서 확정).
- **SC-004**: 활동 목록을 얻기 위해 소스 코드를 읽거나 미문서화 경로에 의존할 필요가 없다.
