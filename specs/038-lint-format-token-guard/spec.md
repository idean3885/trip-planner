## Clarifications

1. **잔여 위반 처리 방침** — 신규 규칙 도입으로 발생하는 기존 코드 위반은 자동정비(자동 수정·일괄 변환)를 1순위로 처리한다. 자동화로 해소되지 않는 잔여는 경고 수준으로 강등하거나 파일 단위 예외로 보류해, 이번 작업이 대규모 수기 정비로 번지지 않게 한다.
2. **규칙 채택 범위** — 검증된(well-known·관리 활발) 규칙 세트 중 현재 프레임워크/런타임과 호환되는 것만 채택한다. 채택 전 라이선스·호환성 컴플라이언스를 사전 확인한다.
3. **동작 불변 제약** — 이번 작업은 설정·포맷·코드 정비에 한정한다. 애플리케이션의 런타임 동작·화면 결과는 바뀌지 않는다.
4. **색상 가드 범위** — 디자인 토큰으로 표현 가능한 색상에 한해 리터럴(hex/rgb 등) 하드코딩을 차단한다. 토큰 체계로 다룰 수 없는 예외(예: 외부 제공 자산 색)는 명시적 예외로 허용한다.

## Metatag Conventions _(normative, inherited from PR #204)_

본 피처의 tasks.md·plan.md는 네 종 메타태그로 후속 자동 검증과 연결된다:

- `[artifact: <path>|<path>::<symbol>]` — 산출 파일 식별자(drift 감사 기준)
- `[why: <short-tag>]` — 추적 그룹 키(plan↔tasks 커버리지)
- `[multi-step: N]` — plan bullet이 다단 작업일 때 최소 매핑 태스크 수(N ≥ 2)
- `[migration-type: schema-only | data-migration]` — 마이그레이션 산출물 구분 (본 피처 해당 없음)

## User Scenarios & Testing _(mandatory)_

본 피처의 사용자는 이 저장소에서 화면·기능 코드를 작성하는 **개발자와 AI 에이전트**다. 일관된 코드 규칙은 유지보수성을 높이고, 에이전트가 따라야 할 패턴을 한 가지로 좁혀 환각(잘못된 추정 코드)을 줄인다.

### User Story 1 - 신규 코드가 강화된 일관성 규칙을 자동으로 따른다 (Priority: P1)

개발자/에이전트가 새 화면 코드를 작성하면, 저장·검사 시점에 일관성 규칙(미사용 변수, 훅 사용 규칙, 접근성, import 순서 등)과 포맷 규약이 자동으로 검사·정렬된다. 규칙 위반은 검사 단계에서 드러난다.

**Why this priority**: 후속 마일스톤의 모든 화면 작업이 이 규칙 위에서 진행된다. 선행되지 않으면 새 코드가 제각각으로 쌓여 정비 비용이 누적된다.

**Independent Test**: 규칙을 일부러 위반한 코드를 작성하고 검사 명령을 실행하면 위반이 보고되는지, 포맷 명령 실행 시 규약대로 정렬되는지로 단독 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 강화된 규칙이 적용된 상태, **When** 미사용 import·잘못된 훅 사용·접근성 결함이 있는 코드를 검사하면, **Then** 각 위반이 명확한 규칙명과 함께 보고된다.
2. **Given** 포맷 규약이 도입된 상태, **When** 포맷 명령을 실행하면, **Then** 들여쓰기·따옴표·import 정렬이 규약대로 통일된다.

---

### User Story 2 - 디자인 토큰 외 임의 색상 사용이 차단된다 (Priority: P1)

개발자/에이전트가 토큰 대신 색상 리터럴(hex/rgb 등)을 직접 적으면 검사 단계에서 차단된다. 색상은 디자인 토큰으로만 표현된다.

**Why this priority**: 후속 디자인 정비(워싱·리브랜딩) 때 색을 한 곳(토큰)에서 바꾸면 전부 반영되어야 한다. 하드코딩 색이 남아 있으면 누락이 생긴다. 가드를 화면 작업 전에 깔아야 새 화면이 토큰만 쓴다.

**Independent Test**: 토큰 대신 hex 색을 적은 코드를 검사하면 차단되고, 토큰을 쓴 코드는 통과하는지로 단독 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 색상 가드가 적용된 상태, **When** 컴포넌트에 hex/rgb 색 리터럴을 직접 적으면, **Then** 검사가 해당 위치를 차단하고 토큰 사용을 안내한다.
2. **Given** 토큰 체계로 다룰 수 없는 명시적 예외, **When** 예외로 표시된 색을 사용하면, **Then** 차단 없이 통과한다.

---

### User Story 3 - 규칙 도입이 대규모 수기 정비로 번지지 않는다 (Priority: P2)

규칙을 새로 켜면 기존 코드에 다량의 위반이 드러난다. 이를 자동정비로 우선 해소하고, 자동으로 안 되는 잔여는 경고 강등/예외 보류로 처리해 작업이 한 번에 끝나도록 한다.

**Why this priority**: 일관성 인프라는 빠르게 깔고 화면 작업으로 넘어가야 한다. 정비가 비대해지면 마일스톤 전체가 지연된다.

**Independent Test**: 규칙 적용 후 자동정비를 실행하면 위반 수가 크게 줄고, 남은 잔여가 경고/예외로 분류되어 검사가 통과 상태가 되는지로 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 강화된 규칙 적용 직후, **When** 자동정비를 실행하면, **Then** 자동 해소 가능한 위반이 일괄 수정된다.
2. **Given** 자동정비 후 잔여 위반, **When** 검사를 실행하면, **Then** 잔여는 경고 또는 명시적 예외로 분류되어 검사가 실패하지 않는다.

### Edge Cases

- 자동정비가 코드 의미를 바꿀 위험이 있는 위반은 어떻게 처리하는가? → 자동 수정 대상에서 제외하고 경고/예외로 분류한다.
- 외부에서 받은 자산·서드파티 코드의 색·스타일은? → 가드 대상에서 제외(예외)로 둔다.
- 규칙 세트가 현재 프레임워크/런타임과 호환되지 않으면? → 채택하지 않는다(컴플라이언스 사전 확인 게이트).

## Requirements _(mandatory)_

### Functional Requirements

- **FR-001**: 코드 검사 도구는 기존 기본 규칙을 넘어 타입체크 기반 규칙·훅 사용 규칙·접근성·import 정렬을 포함해 일관성 위반을 보고해야 한다.
- **FR-002**: 저장소는 단일 포맷 규약을 도입하고, 명령 한 번으로 코드 포맷을 규약대로 정렬할 수 있어야 한다.
- **FR-003**: 색상은 디자인 토큰으로만 표현되어야 하며, 토큰 외 색상 리터럴(hex/rgb 등) 직접 사용 시 검사가 이를 차단해야 한다(명시적 예외 제외).
- **FR-004**: 신규 규칙 도입 후, 기존 코드 위반은 자동정비를 우선 적용하고 잔여는 경고 강등 또는 파일 단위 예외로 분류하여, 검사가 실패하지 않는 통과 상태에 도달해야 한다.
- **FR-005**: 이번 작업은 애플리케이션의 런타임 동작·화면 결과를 바꾸지 않아야 한다(기존 테스트 전부 통과로 확인).
- **FR-006**: 규칙 세트·도구를 채택하기 전에 라이선스·프레임워크 호환성을 사전 확인해야 한다.
- **FR-007**: 보류된 잔여 위반과 예외 처리 내역은 후속 처리가 가능하도록 기록되어야 한다.

### Key Entities

- **코드 검사 규칙 세트**: 일관성·접근성·색상 가드를 포함하는 규칙 묶음. 통과/경고/예외 상태를 가진다.
- **포맷 규약**: 들여쓰기·따옴표·import 정렬 등 코드 표기 규칙. 단일 출처로 관리된다.
- **잔여 위반 목록**: 자동정비로 해소되지 않아 경고·예외로 분류된 항목. 후속 처리 대상.

## Success Criteria _(mandatory)_

### Measurable Outcomes

- **SC-001**: 토큰 외 색상 리터럴을 새로 작성하면 검사가 100% 차단한다(명시적 예외 제외).
- **SC-002**: 강화된 규칙·포맷·색상 가드 적용 후, 저장소 전체 검사가 실패 없이 통과한다(잔여는 경고/예외로만 존재).
- **SC-003**: 기존 자동화 테스트가 전부 통과한다(런타임 동작 동등성 유지).
- **SC-004**: 규칙 도입으로 드러난 기존 위반의 대다수가 자동정비로 해소되고, 수기 정비 잔여는 명시적 예외 목록으로 한정된다.
