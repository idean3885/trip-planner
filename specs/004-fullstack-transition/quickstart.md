# Quickstart: 풀스택 전환 검증 시나리오

**Date**: 2026-04-09
**Feature**: [spec.md](spec.md)

## QS1: 소셜 로그인 (US1)

1. 브라우저에서 앱 접속
2. 로그인 화면 표시 확인
3. 소셜 로그인 버튼 클릭 → 인증 완료
4. 홈 화면에서 여행 목록 표시 확인
5. 로그아웃 → 로그인 화면 복귀 확인

**Pass 기준**: 비로그인 시 보호, 로그인 후 3초 이내 목록 표시, 로그아웃 정상

## QS2: 여행 일정 CRUD (US2)

1. 로그인 후 여행 생성 (제목, 기간, 도시)
2. 일별 일정 추가 (날짜, 도시, 텍스트)
3. 일별 일정 텍스트 수정 → 저장
4. 새로고침 후 수정 내용 유지 확인
5. 여행 개요 수정 (제목 변경) → 저장 확인
6. 일별 일정 삭제 확인

**Pass 기준**: 생성/조회/수정/삭제 전부 동작, 새로고침 후 데이터 유지

## QS3: 팀 초대 및 권한 (US3)

1. 여행 소유자가 동행자 이메일로 초대
2. 초대 링크 생성 확인 (대기중 상태)
3. 동행자가 초대 링크 접속 → 로그인 → 팀 합류
4. 동행자의 여행 목록에 해당 여행 표시 확인
5. 소유자가 동행자 권한을 VIEWER → EDITOR로 변경
6. 동행자(EDITOR)가 일정 편집 가능 확인
7. 소유자가 동행자를 제거 → 여행 목록에서 사라짐 확인

**Pass 기준**: 초대~합류 5분 이내, 권한 변경 즉시 반영, 제거 후 접근 불가

## QS4: 데이터 마이그레이션 (US4)

1. 기존 여행 데이터(14일, 6개 도시) 마이그레이션 실행
2. 웹앱에서 여행 목록 확인
3. 여행 선택 → 일별 일정 14일 전부 표시 확인
4. 각 일정의 내용이 기존과 동일한지 확인

**Pass 기준**: 14일 전체 마이그레이션, 기존 화면과 동일 표시

## QS5: 권한 보안 검증 (SC-005)

1. VIEWER 권한 사용자로 로그인
2. 일정 편집 UI가 비활성화 확인
3. 브라우저 개발자 도구로 편집 API 직접 호출 시도
4. 서버에서 403 Forbidden 반환 확인

**Pass 기준**: UI 비활성화 + 서버 검증 모두 동작

## Edge Case 시나리오

### EC1: 만료된 초대 링크
1. 초대 생성 후 7일 경과 (또는 테스트용 단축 만료)
2. 초대 링크 접속 → "초대가 만료되었습니다" 메시지 표시

### EC2: 소유자 자기 제거 시도
1. 소유자가 팀 관리에서 자기 자신을 제거 시도
2. "소유자는 제거할 수 없습니다" 안내 표시

### EC3: 동시 편집 (last-write-wins)
1. 두 편집자가 같은 일정을 동시에 열기
2. A가 먼저 저장, B가 나중에 저장
3. B의 내용이 최종 반영됨 확인
