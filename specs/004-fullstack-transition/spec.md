## Clarifications

### Session 2026-04-09

- Q: 일정 편집 단위는 하루 전체 텍스트인가, 시간대별 개별 항목인가? → A: 하루 전체를 하나의 텍스트 블록으로 편집한다
- Q: 여행 삭제 권한과 데이터 처리 방식은? → A: 오너만 삭제 가능, 즉시 삭제(hard delete), UI 확인 대화상자 1회, 알림 없음
- Q: 소유권 이전이 가능한가? → A: 오너 양도 가능. OWNER가 다른 HOST에게 양도하면 기존 OWNER는 HOST로 변경
- Q: 여행 개요(제목, 기간, 도시 등)를 편집할 수 있는가? → A: 가능. OWNER와 HOST가 개요를 수정할 수 있다
- Q: Phase 1 기능(MCP 검색, 캘린더, 피드백 채널)은 Phase 2에서 어떻게 되는가? → A: 별도 도구로 제공을 유지한다. 사용 여부는 사용자의 선택이다
- Q: 게스트(GUEST)가 일정을 편집할 수 있는가? → A: 불가. 게스트는 조회만 가능. 편집이 필요하면 호스트로 초대

### Session 2026-04-14

- Q: 초대 링크 전달 방식은? → A: 호스트가 링크를 복사하여 직접 전달 (카카오톡, 문자 등). 시스템 이메일 발송 없음
- Q: 호스트가 자발적으로 여행에서 탈퇴할 수 있는가? → A: 가능 (마지막 호스트가 아닌 경우). OWNER는 양도 후 탈퇴 가능
- Q: 게스트가 자발적으로 여행에서 탈퇴할 수 있는가? → A: 가능
- Q: 여행 삭제 시 확인 절차는? → A: UI 확인 대화상자 1회 ("정말 삭제하시겠습니까?")
- Q: 호스트를 게스트로 강등할 수 있는가? → A: OWNER만 가능 (마지막 호스트가 아닌 경우)

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 소셜 계정으로 로그인 (Priority: P1)

사용자가 여행 플래너에 접속하여 소셜 계정으로 로그인한다. 로그인 후 자신이 속한 여행 목록을 확인할 수 있다. 로그인하지 않은 사용자는 여행 목록을 볼 수 없다.

**Why this priority**: 인증이 모든 기능의 전제 조건이다. 로그인 없이는 팀 구분, 데이터 격리, 권한 관리가 불가능하다.

**Independent Test**: 소셜 로그인 버튼을 클릭하고 인증을 완료하면, 홈 화면에서 자신의 여행 목록이 표시되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 비로그인 사용자가 홈에 접속했을 때, **When** 페이지가 로드되면, **Then** 로그인 화면이 표시되고 여행 목록은 보이지 않는다
2. **Given** 로그인 화면에서 소셜 로그인 버튼을 클릭했을 때, **When** 인증을 완료하면, **Then** 홈 화면으로 이동하고 자신이 속한 여행 목록이 표시된다
3. **Given** 로그인된 사용자가 로그아웃을 클릭했을 때, **When** 로그아웃이 처리되면, **Then** 로그인 화면으로 돌아간다
4. **Given** 세션이 만료된 사용자가 페이지를 새로고침했을 때, **When** 페이지가 로드되면, **Then** 로그인 화면으로 리다이렉트된다

---

### User Story 2 - 여행 일정 조회 및 편집 (Priority: P1)

로그인한 사용자가 자신이 속한 여행의 일정을 조회한다. 호스트(주인 포함)는 일정을 직접 편집할 수 있고, 게스트는 조회만 가능하다. 편집한 내용은 즉시 저장되며 다른 멤버가 새로고침하면 최신 상태를 확인할 수 있다.

**Why this priority**: 여행 플래너의 핵심 기능이다. 기존 정적 조회를 동적 편집으로 전환하는 것이 Phase 2의 본질이다.

**Independent Test**: 로그인 후 여행을 선택하고 일별 일정을 확인한 뒤, 하루 단위 텍스트를 수정하고 저장하면 재접속 시에도 수정된 내용이 유지되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 로그인한 사용자가 여행을 선택했을 때, **When** 여행 페이지가 로드되면, **Then** 여행 개요와 일별 일정 목록이 표시된다
2. **Given** 호스트가 일정 항목을 수정했을 때, **When** 저장을 완료하면, **Then** 변경 사항이 즉시 반영되고 다른 멤버가 새로고침하면 최신 상태를 확인할 수 있다
3. **Given** 게스트가 일정을 보고 있을 때, **When** 편집을 시도하면, **Then** 편집 기능이 비활성화되어 있고 수정할 수 없다
4. **Given** 여행에 참여하지 않은 사용자가 URL로 직접 접근했을 때, **When** 페이지가 로드되면, **Then** 접근이 거부된다

---

### User Story 3 - 팀 초대 및 조인 (Priority: P2)

호스트가 동행자를 이메일로 초대하고, 초대받은 사용자가 링크를 통해 여행에 합류한다. 초대 시 호스트/게스트 역할을 지정할 수 있다.

**Why this priority**: 팀 기능은 "동행자가 직접 참여하는 여행 플래너"의 핵심이지만, 인증과 일정 편집이 먼저 갖춰져야 의미가 있다.

**Independent Test**: 호스트가 동행자 이메일로 초대 링크를 생성하고, 동행자가 해당 링크로 접속하여 여행에 합류한 뒤 일정을 확인할 수 있는지 검증한다.

**Acceptance Scenarios**:

1. **Given** 호스트가 동행자 관리 화면에서 이메일을 입력했을 때, **When** 초대를 전송하면, **Then** 초대 링크가 생성되고 초대 상태가 "대기중"으로 표시된다
2. **Given** 초대받은 사용자가 초대 링크를 클릭했을 때, **When** 로그인을 완료하면, **Then** 해당 여행의 멤버로 등록되고 여행 목록에 해당 여행이 표시된다
3. **Given** 호스트가 게스트를 호스트로 승격했을 때, **When** 역할 변경을 완료하면, **Then** 해당 멤버가 호스트 권한(편집/초대)을 갖게 된다
4. **Given** 호스트가 멤버를 제거했을 때, **When** 제거를 확인하면, **Then** 해당 멤버의 여행 목록에서 이 여행이 사라진다

---

### User Story 4 - 기존 데이터 마이그레이션 (Priority: P1)

기존 여행 데이터를 데이터베이스로 마이그레이션한다. 마이그레이션 후 기존과 동일한 일정 정보가 웹앱에서 조회된다.

**Why this priority**: 데이터베이스 전환 없이는 동적 편집과 팀 기능이 불가능하다. 기존 데이터를 보존하면서 전환해야 한다.

**Independent Test**: 마이그레이션을 실행한 뒤 웹앱에서 기존 여행 일정이 동일하게 표시되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 기존 여행 데이터가 있을 때, **When** 마이그레이션을 실행하면, **Then** 모든 여행, 일별 일정, 개요 정보가 데이터베이스에 저장된다
2. **Given** 마이그레이션이 완료된 후, **When** 웹앱에서 여행을 조회하면, **Then** 기존 화면과 동일한 정보가 표시된다

---

### Edge Cases

- 초대 링크가 만료된 경우, 명확한 안내 메시지를 표시한다. 이미 멤버인 사용자가 링크를 다시 사용하면 여행으로 바로 이동한다
- 주인이 탈퇴하려는 경우, 먼저 다른 호스트에게 양도해야 한다
- 마지막 호스트(=주인)가 자신을 제거하려는 경우, 호스트가 0명이 되므로 제거할 수 없음을 안내한다
- 동시에 같은 일정을 편집하는 경우, 마지막 저장이 우선한다 (last-write-wins)
- 소셜 계정이 없는 동행자가 초대 링크를 받은 경우, 계정 생성 안내를 표시한다
- 마이그레이션 중 데이터 파싱 실패 시, 실패 항목을 로그로 기록하고 나머지는 계속 처리한다

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 사용자는 소셜 계정으로 로그인/로그아웃할 수 있어야 한다
- **FR-002**: 비로그인 사용자는 로그인 화면만 볼 수 있어야 한다 (보호된 라우트)
- **FR-003**: 로그인한 사용자는 자신이 속한 여행 목록만 조회할 수 있어야 한다
- **FR-004**: 호스트(주인 포함)는 일별 일정을 하루 단위 텍스트로 수정하고 저장할 수 있어야 한다
- **FR-004a**: 호스트는 일별 일정을 추가/삭제할 수 있어야 한다
- **FR-004b**: 호스트는 여행 개요(제목, 기간, 도시 목록)를 수정할 수 있어야 한다
- **FR-005**: 게스트는 일정을 조회할 수 있지만 수정할 수 없어야 한다. 여행에 참여하지 않은 사용자는 해당 여행에 접근할 수 없어야 한다
- **FR-006**: 호스트는 초대 링크를 생성(호스트/게스트 역할 지정)하여 직접 전달할 수 있어야 한다. 링크는 만료 기간이 있으며, 이메일 입력 없이 링크만으로 합류 가능하다
- **FR-007**: 초대 링크를 받은 사용자는 로그인 후 자동으로 여행에 합류할 수 있어야 한다
- **FR-008**: 호스트는 게스트를 호스트로 승격할 수 있어야 한다. 주인만 호스트를 게스트로 강등할 수 있다 (마지막 호스트 보호)
- **FR-009**: 호스트는 게스트를 제거할 수 있어야 한다. 주인만 호스트를 제거할 수 있다. 멤버는 자발적으로 탈퇴할 수 있다 (주인은 양도 후 탈퇴)
- **FR-010**: 기존 여행 데이터를 데이터베이스로 마이그레이션할 수 있어야 한다
- **FR-011**: 마이그레이션 후 기존과 동일한 일정 정보가 웹앱에서 조회되어야 한다
- **FR-012**: 여행을 새로 생성할 수 있어야 한다 (제목, 기간, 도시 목록)
- **FR-013**: 모바일에서 추가 조작 없이 모든 기능을 사용할 수 있어야 한다 (기존 Mobile-First 원칙 유지)
- **FR-014**: 주인만 여행을 삭제할 수 있어야 한다 (즉시 삭제, 복구 불가, 확인 대화상자 1회)
- **FR-015**: 주인은 다른 호스트에게 주인 권한을 양도할 수 있어야 한다. 양도 시 기존 주인은 호스트로 변경된다

### Non-Functional Requirements

- **NFR-001 (시간 기준)**: 시스템의 기준 시간대는 한국 표준시(KST, UTC+9)이다. 서버 기록, 감사 로그, 정렬 기준 등 모든 내부 시간 처리는 KST를 기준으로 한다.
- **NFR-002 (현지 시각 표시)**: 사용자에게 표시되는 시각은 사용자의 현재 위치(현지 시각)에 맞게 변환하여 제공한다. 여행 중 해외에서 접속하는 경우, 현지 시간대 기준으로 일정 시각이 표시되어야 한다.
- **NFR-003 (달력 날짜)**: 여행 일정의 날짜(몇 월 며칠)는 시간대 변환의 영향을 받지 않는다. "6월 7일"은 어느 시간대에서 조회하더라도 "6월 7일"로 표시되어야 한다.
- **NFR-004 (시간 정밀도)**: 모든 시간 값은 밀리초(3자리) 정밀도로 저장한다.

### Key Entities

- **사용자 (User)**: 서비스 사용자. 소셜 계정으로 인증. 이름, 이메일, 프로필 이미지.
- **여행 (Trip)**: 여행 단위. 제목, 기간, 생성자/수정자 기록.
- **일정 (Day)**: 일별 일정. 여행에 속하며 날짜, 도시, 하루 단위 텍스트 내용.
- **멤버 (TripMember)**: 여행-사용자 참여 관계. 역할: 주인(OWNER) / 호스트(HOST) / 게스트(GUEST). 화면에서 주인과 호스트는 모두 "호스트"로 표시.
- **초대 링크**: 만료 기간이 포함된 토큰 기반 링크. 역할(호스트 또는 게스트) 지정. 별도 테이블 없이 토큰 자체에 정보를 담는다. 주인 권한은 양도로만 부여.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 사용자가 소셜 로그인 후 3초 이내에 여행 목록을 확인할 수 있다
- **SC-002**: 기존 여행 데이터(14일, 6개 도시)를 마이그레이션한 뒤, 모든 일정이 웹앱에서 동일하게 표시된다
- **SC-003**: 호스트가 동행자를 초대하고, 동행자가 합류하여 일정을 조회하기까지 5분 이내에 완료할 수 있다
- **SC-004**: 호스트가 일정을 수정하면, 다른 멤버가 새로고침 시 최신 상태를 확인할 수 있다
- **SC-005**: 게스트는 일정을 조회할 수 있지만 수정할 수 없다. 비멤버는 접근 자체가 거부된다

## Glossary

본 문서에서 사용하는 용어를 정의한다. 스펙, 플랜, 태스크, 코드 전반에서 이 용어를 일관되게 사용한다.

### 역할 (Role)

| 한글 | 영문 | 정의 |
|------|------|------|
| 주인 | OWNER | 여행을 생성한 사용자. 여행당 1명. 양도 가능. 모든 권한 보유 |
| 호스트 | HOST | 여행을 함께 기획하는 동행자. 일정 편집 + 초대 가능 |
| 게스트 | GUEST | 일정을 공유받는 동행자. 조회만 가능 |

- 화면에서 주인과 호스트는 모두 **"호스트"**로 표시한다. 주인 전용 기능(삭제, 강등, 양도)만 추가 노출.
- 게스트는 화면에서 **"게스트"**로 표시한다.
- 이 문서에서 "호스트"는 OWNER + HOST를 포괄하며, OWNER만 해당하는 경우 "주인"으로 명시한다.

### 주요 용어

| 한글 | 영문 | 정의 |
|------|------|------|
| 여행 | Trip | 여행 단위. 제목, 기간, 일정을 포함하는 최상위 단위 |
| 일정 | Day | 일별 일정. 여행에 속하며 날짜, 도시, 하루 단위 텍스트 내용을 포함 |
| 멤버 | TripMember | 여행에 참여하는 사용자. 역할(OWNER/HOST/GUEST) 포함 |
| 초대 | Invite | 동행자를 여행에 초대하는 행위. 만료 기간이 포함된 토큰 기반 링크 |
| 승격 | Promote | 게스트를 호스트로 역할 변경 |
| 강등 | Demote | 호스트를 게스트로 역할 변경. 주인만 가능 |
| 양도 | Transfer | 주인 권한을 다른 호스트에게 이전. 기존 주인은 호스트로 변경 |
| 탈퇴 | Leave | 멤버가 자발적으로 여행에서 나감 |
| 제거 | Remove | 호스트가 다른 멤버를 여행에서 내보냄 |
| 달력 날짜 | Calendar Date | 시간대 변환 영향을 받지 않는 날짜 (예: "6월 7일"은 어디서든 "6월 7일") |
| 시각 | Timestamp | 특정 시점. 시스템 기준은 KST, 사용자 화면은 현지 시각으로 변환 |

### 사용하지 않는 용어

| 용어 | 대신 사용 | 이유 |
|------|----------|------|
| 소유자 | 주인 (OWNER) | "소유자"는 법적 뉘앙스. "주인"이 여행 맥락에 적합 |
| 편집자 | 호스트 (HOST) | 2단계 역할(OWNER/EDITOR/VIEWER)에서 3단계(OWNER/HOST/GUEST)로 변경 |
| 조회자 | 게스트 (GUEST) | 동일 사유 |
| 팀원 | 멤버 (TripMember) | "팀"보다 "여행 멤버"가 도메인에 적합 |

## Assumptions

- 초기 소셜 로그인 프로바이더는 1개 이상 지원한다 (구체적 프로바이더는 플랜에서 결정)
- 동시 편집 충돌은 last-write-wins로 처리한다 (실시간 공동 편집은 범위 밖)
- 초대 링크의 유효 기간은 7일로 한다
- 기존 여행 데이터의 일정 내용은 텍스트 그대로 저장한다 (구조화 파싱은 점진적으로 진행)
- 무료 인프라 한도 내에서 운영 가능한 규모 (사용자 10명 이내)
- 배포 환경은 플랜에서 결정한다 (기존 정적 호스팅에서 전환)
- Phase 1 기능(MCP 검색, 캘린더 연동, 피드백 채널)은 별도 도구로 계속 제공한다. Phase 2 웹앱에 통합하지 않으며, 사용 여부는 사용자의 선택이다
