# Feature Specification: Google Calendar 공유 플로우 재설계

**Feature Branch**: `019-gcal-shared-flow`
**Created**: 2026-04-22
**Status**: Draft
**Input**: v2.8.0 연동이 멤버마다 본인 계정에 DEDICATED 캘린더를 개별 생성하여 여행 1개에 N배 중복 발생. v2.9.0에서 오너 1개 공유 캘린더에 멤버가 옵트인으로 구독하는 단일-소스 모델로 재설계.

**Related**: Epic [#349](https://github.com/idean3885/trip-planner/issues/349), 마일스톤 [#26 v2.9.0](https://github.com/idean3885/trip-planner/milestone/26), PoC research `docs/research/v290-gcal-share-poc.md`

## Clarifications *(v2.9.0 결정 포인트, PoC 결과 반영)*

1. **공유 캘린더 유형** — DEDICATED(전용) 전용. 외부 캘린더 API는 primary 유형 공유를 지원하지 않으므로 v2.8.0의 PRIMARY 옵션은 본 피처에서 제거한다.
2. **여행당 캘린더 1개** — 캘린더는 trip당 1개만 존재한다. 오너가 캘린더를 만들고(또는 기존 것을 채택), 다른 멤버는 추가 생성하지 않는다.
3. **권한 부여는 서버 자동** — 멤버 가입·역할 변경 시 서버가 오너의 권한으로 캘린더 ACL을 갱신한다. 멤버 개인의 추가 작업 불필요.
4. **개인 캘린더 추가는 멤버 수동** — 멤버가 본인 외부 캘린더 UI에 공유 캘린더를 띄우는 작업은 **명시적 옵트인**이다. 가입만으로 자동 추가하지 않는다 (안 쓸 자유 보장).
5. **알림 이메일은 외부 시스템 소관** — PoC에서 확인: 외부 API의 "알림 억제" 플래그는 개인 계정 간 공유에서 무시된다. v2.9.0은 외부 알림 이메일이 항상 발송될 수 있다고 가정하고, in-app 알림과 병존을 UI 문구로 안내한다.
6. **v2.8.0 데이터 마이그레이션** — 기존 오너 DEDICATED 캘린더는 v2.9.0 공유 캘린더로 승격 재사용한다. 같은 트립의 다른 멤버가 만든 DEDICATED 캘린더는 자동 연결 해제하고, 그 캘린더 자체는 멤버 본인 계정에 남겨둔다(사용자가 수정·유지한 데이터 손실 방지).
7. **역할 매핑** — 트립 오너 ↔ 캘린더 owner(secondary calendar), 호스트 ↔ writer, 게스트 ↔ reader. PoC에서 외부 API가 이 매핑을 지원함을 확인.
8. **scope 지속성 이슈 분리** — PoC 중 발견한 Account.scope 리셋 이상은 본 피처 범위 밖의 조사 이슈로 분리한다 (v2.9.0 릴리즈 차단 요소 아님).

## Metatag Conventions *(normative, inherited from PR #204)*

본 피처의 tasks.md·plan.md는 네 종 메타태그를 통해 후속 자동 검증과 연결된다:

- `[artifact: <path>|<path>::<symbol>]` — 산출 파일 식별자(drift 감사 기준)
- `[why: <short-tag>]` — 추적 그룹 키(plan↔tasks 커버리지·이슈 합산)
- `[multi-step: N]` — plan bullet이 다단 작업일 때 최소 매핑 태스크 수(N ≥ 2)
- `[migration-type: schema-only | data-migration]` — 마이그레이션 산출물 구분

형식 검증은 `.specify/scripts/bash/validate-metatag-format.sh`가 수행한다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — 오너가 여행에 공유 캘린더를 1개 붙인다 (Priority: P1)

여행 오너가 "외부 캘린더 연결" 버튼을 누르면 해당 여행 전용의 공유 캘린더가 오너 계정에 생성되고, 현재 트립 멤버 전원에게 역할에 맞는 권한이 자동 부여된다. 이후 멤버가 가입하면 서버가 자동으로 ACL을 갱신한다.

**Why this priority**: 공유 모델의 정본 상태를 만드는 단계. 이 하나만으로도 "여행마다 외부 캘린더 1개" 원칙이 성립하며, 사용자는 기존 v2.8.0의 멤버별 중복 캘린더 문제에서 벗어난다 (MVP로 가치 전달).

**Independent Test**: 멤버 2인 이상의 트립에서 오너가 연결 → 오너 계정에 1개의 공유 캘린더 생성 확인 + 다른 멤버 전원에게 올바른 권한 부여 확인 + 다른 멤버 계정에는 **자동 생성 없음** 확인으로 단독 검증 가능.

**Acceptance Scenarios**:

1. **Given** 여행에 오너·호스트 1명·게스트 1명이 있는 상태, **When** 오너가 "외부 캘린더 연결"을 실행, **Then** 오너 계정에 공유 캘린더 1개 생성 + 호스트에는 writer + 게스트에는 reader 권한 부여 + 호스트·게스트 계정에는 새 캘린더가 생성되지 않음.
2. **Given** 캘린더가 이미 연결된 여행, **When** 새 멤버가 초대를 수락해 합류, **Then** 추가 사용자 조작 없이 서버가 신규 멤버에게 역할에 맞는 권한을 부여.
3. **Given** 캘린더가 연결된 여행에서 멤버의 역할이 host↔guest로 변경, **When** 역할 변경이 저장됨, **Then** 해당 멤버의 캘린더 권한이 새 역할로 자동 조정됨.
4. **Given** 같은 여행에 "재연결"이 반복 실행, **When** 오너가 다시 "외부 캘린더 연결" 실행, **Then** 기존 연결을 그대로 유지하거나 idempotent 재적용만 일어나고 중복 권한 항목이 쌓이지 않음.

### User Story 2 — 멤버가 공유 캘린더를 본인 외부 캘린더에 추가/해제 (Priority: P1)

멤버(호스트·게스트)는 트립 페이지에서 "내 외부 캘린더에 추가" 버튼으로 공유 캘린더를 본인 외부 캘린더 UI에 띄울 수 있다. 버튼을 누르지 않으면 아무 변화도 일어나지 않으며, 언제든 "제거"로 되돌릴 수 있다. 외부 계정과 앱의 연결이 안 된 경우 필요한 권한만 요청하는 동의 흐름으로 안내한다.

**Why this priority**: US1의 캘린더는 권한은 부여됐지만 멤버 외부 UI에는 자동으로 뜨지 않는다(외부 API의 공식 정책). 이 스토리로 멤버 경험이 완성된다. 둘은 독립적으로 릴리즈 가능하나, US2 없이 US1만으로는 "외부 캘린더에 뜨지 않는다"는 혼란이 남으므로 동일 P1.

**Independent Test**: 오너가 이미 연결한 여행에서 멤버가 버튼을 누른 뒤 본인 외부 캘린더 UI에 해당 캘린더가 나타남을 확인. 다시 누르면 제거됨을 확인. 앱 인증이 없는 멤버는 동의 플로우를 거친 뒤 동일 결과에 도달.

**Acceptance Scenarios**:

1. **Given** 공유 캘린더에 reader 권한이 부여된 게스트 멤버, **When** 트립 페이지에서 "내 외부 캘린더에 추가"를 누름, **Then** 본인 외부 캘린더 UI에 해당 캘린더가 표시되고 이벤트 조회가 가능.
2. **Given** 외부 계정에 앱의 캘린더 권한이 없는 멤버, **When** "내 외부 캘린더에 추가"를 누름, **Then** 필요한 최소 권한 동의 화면으로 이동하고 동의 후 원래 위치로 복귀해 자동으로 추가 완료.
3. **Given** 내 외부 캘린더에 이미 추가된 멤버, **When** "제거"를 누름, **Then** 본인 외부 UI에서 캘린더가 사라지지만 ACL은 그대로 남아 다시 추가가 가능.
4. **Given** 멤버가 트립을 탈퇴, **When** 탈퇴가 저장됨, **Then** 해당 멤버의 ACL이 자동 회수되고, 이미 본인 외부 UI에 떠 있던 경우 읽기가 불가능해진다(앱은 멤버 외부 UI에서 제거 강제 시도하지 않음 — 사용자 정돈 몫).

### User Story 3 — v2.8.0에서 넘어온 기존 연결을 무중단으로 승격 (Priority: P2)

v2.8.0에서 이미 외부 캘린더를 연결한 여행은 릴리즈 후에도 기존 오너 캘린더를 그대로 재사용하고, 같은 트립의 다른 멤버가 과거에 만든 개별 DEDICATED 캘린더는 자동 연결 해제된다. 사용자가 직접 수정·기록한 이벤트는 보존된다.

**Why this priority**: 호환성·신뢰 차원. US1/US2가 MVP로 가치 있지만, 기존 사용자가 "어제 되던 연동이 깨졌다" 경험을 하지 않도록 필수. 단, 마이그레이션 중 일시적 연결 해제 상태는 허용됨 → P2.

**Independent Test**: v2.8.0 상태의 샘플 트립 1개(오너+호스트+게스트 전원 DEDICATED) 준비 → 마이그레이션 실행 → 오너 캘린더 재사용 확인 + 멤버 DEDICATED 캘린더는 trip-planner 앱 시점에서 unlink 상태 + 멤버 외부 계정에 남은 캘린더의 이벤트는 그대로인지 확인.

**Acceptance Scenarios**:

1. **Given** v2.8.0에서 오너만 DEDICATED 캘린더를 연결한 트립, **When** v2.9.0 마이그레이션 실행, **Then** 기존 오너 캘린더가 공유 캘린더로 승격되고 멤버 전원에게 ACL 자동 부여.
2. **Given** v2.8.0에서 오너+호스트가 각자 DEDICATED 캘린더를 연결한 트립, **When** 마이그레이션 실행, **Then** 오너 캘린더 재사용 + 호스트 DEDICATED는 앱 내 연결 해제 + 호스트 외부 캘린더에 남은 캘린더는 삭제되지 않음(본인이 직접 정리 가능).
3. **Given** 마이그레이션 도중 앱 이용, **When** 트립 조회, **Then** 트립 페이지 자체는 정상 동작하고 캘린더 연동 섹션만 "업그레이드 중" 상태로 표시되며 이용 후 자동 해제.

---

### Edge Cases

- **오너 외부 계정이 해제됨** — 오너의 외부 앱 권한이 해제된 상태에서 신규 멤버가 가입할 때, 자동 ACL 부여는 실패 기록(`lastError=REVOKED`)으로 남기고 다른 멤버의 조회/동작에는 영향 주지 않는다.
- **외부 API 쿼터 초과** — 멤버 수가 많아 초기 연결 시 ACL 부여가 부분 실패하면, 성공한 멤버만 확정하고 실패 목록을 응답에 포함한다(오너가 "다시 반영하기"로 재시도 가능).
- **멤버가 외부 UI에서 직접 캘린더를 제거** — 멤버 외부 UI에서만 숨긴 상태. 앱은 "내 외부 캘린더에 다시 추가" 버튼을 다시 보여준다(ACL은 유지된 상태).
- **오너 역할 이관** — 오너 변경 기능이 있는 경우, 공유 캘린더의 외부 API 상의 데이터 오너는 이관 불가(외부 API 제약). 트립 내 오너 역할만 이관되고 외부 캘린더 데이터는 원 오너에 남되, 새 오너가 앱에서 "재연결"로 본인 캘린더로 대체 생성 가능.
- **이메일 알림 중복 불편** — 외부 시스템의 이메일 + 앱 내 배너가 동시에 도착. UI 문구로 "외부에서도 이메일이 발송될 수 있습니다"를 짧게 안내해 혼선 최소화.
- **마이그레이션 중 실패** — 특정 트립의 마이그레이션이 실패한 경우, 그 트립만 v2.8.0 상태로 남기고 다른 트립에는 영향 주지 않는다. 실패한 트립은 다음 기회(오너 재연결 또는 재배포 시 재시도)에 다시 대상.
- **동일 이메일의 멤버 중복 초대** — 하나의 외부 계정 이메일로 두 명이 초대된 경우 ACL은 이메일 기준이라 결과적으로 같은 권한이 부여된다(앱 내 멤버십은 별개). 문제 없음.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 하나의 여행에 외부 공유 캘린더를 **여행당 1개**만 연결해야 한다. 같은 여행에 두 번째 캘린더 연결 시도는 기존 연결을 재사용하거나 교체를 명시해야 한다.
- **FR-002**: 시스템은 오너가 외부 캘린더 연결 조작을 한 시점에 **오너 계정**에 신규 DEDICATED 캘린더를 생성하거나, v2.8.0의 기존 오너 DEDICATED 캘린더를 재사용해야 한다.
- **FR-003**: 시스템은 외부 캘린더 연결 즉시 현재 여행 멤버 전원에게 역할별 권한(오너=owner / 호스트=writer / 게스트=reader)을 자동으로 부여해야 한다.
- **FR-004**: 시스템은 멤버가 여행에 신규 가입(초대 수락 포함)하는 시점에 해당 멤버에게 역할에 맞는 권한을 자동 부여해야 한다.
- **FR-005**: 시스템은 기존 멤버의 역할이 변경될 때 해당 멤버의 외부 캘린더 권한을 새 역할로 자동 갱신해야 한다.
- **FR-006**: 시스템은 멤버가 여행에서 탈퇴하거나 제거될 때 해당 멤버의 외부 캘린더 권한을 회수해야 한다.
- **FR-007**: 시스템은 멤버가 본인 외부 캘린더에 공유 캘린더를 **옵트인으로 추가**할 수 있는 명시적 액션을 트립 페이지에 제공해야 하며, 해당 액션 없이는 자동으로 멤버 외부 캘린더 목록에 추가하지 않는다.
- **FR-008**: 시스템은 멤버가 외부 캘린더에 추가한 공유 캘린더를 **제거**할 수 있어야 한다. 제거는 멤버 외부 UI에서만 숨기고 ACL과 앱 내 멤버십은 유지해야 한다.
- **FR-009**: 시스템은 멤버의 외부 계정이 앱에 권한을 부여하지 않은 상태에서 "내 외부 캘린더에 추가"를 시도하면 **필요한 최소 권한 동의 플로우**로 안내하고, 동의 후 원래 동작을 재개해야 한다.
- **FR-010**: 시스템은 외부 API의 권한 부여가 재호출 시 중복 항목을 생성하지 않도록 **idempotent**하게 동작해야 한다(같은 멤버·역할 재부여 호출은 side-effect 0).
- **FR-011**: 시스템은 외부 알림 이메일(우리가 억제 지시해도 외부가 보내는 이메일)을 전제로 하여, 앱 내 알림 문구에 외부 이메일 병존 사실을 명시해야 한다.
- **FR-012**: 시스템은 v2.8.0 사용자 데이터 마이그레이션 시 **기존 오너 DEDICATED 캘린더는 공유 캘린더로 재사용**하고, 같은 여행의 다른 멤버가 만든 DEDICATED 캘린더는 앱 내 연결을 자동 해제해야 한다. 멤버의 외부 계정에 남는 캘린더 자체는 삭제하지 않는다.
- **FR-013**: 시스템은 마이그레이션 도중 특정 여행의 마이그레이션이 실패하더라도, 다른 여행의 동작과 앱 전체 이용에 영향이 없어야 한다.
- **FR-014**: 시스템은 여행당 캘린더 상태(연결됨/미연결/오류/마이그레이션 중)를 오너·호스트·게스트 각자의 시점에서 명확히 표시해야 한다. 멤버 시점에서는 "본인이 외부 캘린더에 추가했는지" 여부도 함께 표시해야 한다.
- **FR-015**: 시스템은 연결·권한 부여·구독·해제 모든 동작에서 부분 실패를 허용해야 하며, 실패 항목을 응답에 포함해 오너가 재시도할 수 있어야 한다.
- **FR-016**: 시스템은 외부 캘린더 데이터 오너(외부 API 상의 생성자)를 이관할 수 없다는 외부 제약을 수용하여, 앱 내 오너 이관 기능이 있더라도 외부 캘린더 데이터 오너는 변경하지 않는다. 대신 새 앱 오너가 필요 시 자기 계정에 새 공유 캘린더를 생성하는 재연결 경로를 제공해야 한다.
- **FR-017**: 시스템은 트립 페이지에서 오너/호스트/게스트 각 시점별 UI 문구를 역할에 맞게 노출해야 한다(예: 오너는 "연결/재연결/해제", 멤버는 "내 외부 캘린더에 추가/제거").

### Key Entities *(include if feature involves data)*

- **Trip** — 여행. 오너·호스트·게스트 멤버십 보유. 본 피처 범위에서 구조 변경 없음.
- **TripMember** — 트립 멤버십. 역할(owner/host/guest) 보유. 본 피처는 이 역할을 외부 캘린더 권한으로 매핑한다.
- **TripCalendarLink (신규)** — 트립당 1개, 외부 캘린더와의 연결. 여행당 **단일** 오너가 기록된 공유 캘린더 식별자를 보유하며, 마지막 동기화 시각·오류 상태·건너뛴 항목 수 등을 포함한다. (기존 v2.8.0의 per-member `GCalLink`를 대체한다 — 본 피처의 파괴적 스키마 변경 포인트)
- **MemberCalendarSubscription (신규)** — 멤버 시점의 "본인 외부 캘린더 목록에 해당 공유 캘린더를 추가했는지" 상태. 추가/미추가/제거 요청/오류 상태를 표현. 멤버 경험의 ON/OFF만을 다루며, 실제 subscribe 동작은 외부 API에 위임.
- **CalendarEventMapping** — 여행의 활동과 외부 캘린더 이벤트 간 매핑. 본 피처에서는 per-link 기준이 per-trip 기준으로 축약될 뿐, 개념은 유지.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: v2.9.0 릴리즈 이후 새로 연결된 여행 100%에서 외부 캘린더가 여행당 1개만 존재한다(샘플 집합 기준).
- **SC-002**: 오너가 "외부 캘린더 연결"을 누른 뒤 **5초 이내** 트립 페이지의 캘린더 섹션이 "연결됨" 상태로 바뀌고, 모든 현재 멤버에게 권한이 부여된다(부분 실패 허용 — SC-005로 재시도 경로).
- **SC-003**: 멤버가 "내 외부 캘린더에 추가"를 누르고 **10초 이내** 본인 외부 캘린더 UI에 표시되는 비율 95% 이상(네트워크 왕복 + 외부 반영 시간 포함).
- **SC-004**: v2.8.0 사용자 중 기존 오너 DEDICATED 캘린더가 있던 트립의 **95% 이상**이 마이그레이션 후에도 연결 상태 유지(이벤트 유실 0).
- **SC-005**: 권한 부여·구독 실패 항목은 오너/멤버 화면에서 이유와 함께 노출되며, 같은 액션 재시도로 성공률 90% 이상 회복.
- **SC-006**: 본 피처 배포 후 30일간 동일 사용자로부터 "내 캘린더에 같은 여행이 두 번 생긴다"는 CS가 접수되지 않음(현 v2.8.0 대비 감소 목표: 이런 유형 CS 100% 제거).
- **SC-007**: v2.8.0 멤버별 DEDICATED 캘린더의 중복 저장 용량이 마이그레이션 완료 후 신규 트립 생성에서 0이 된다(측정: 트립당 외부 캘린더 개수 = 1).

## Out of Scope *(for clarity)*

- 다른 외부 캘린더 provider(예: Apple CalDAV) 추가 — 별도 v2.10+ 마일스톤
- 외부 캘린더 provider 추상화 리팩터 — 별도 마일스톤
- 외부 API 이메일 문구 커스터마이징 — 외부 시스템이 통제
- 트립 내 오너 이관 기능 신설 — 있으면 수용, 본 피처는 이관 기능을 새로 만들지 않음
- scope 리셋 이상 조사 — 별도 이슈
