# Feature Specification: 여행 상세 캘린더 중심 단일 화면 재설계

**Feature Branch**: `032-calendar-centric-detail`  
**Created**: 2026-05-30  
**Status**: Draft  
**Input**: 여행 상세(/trips/[id])를 캘린더 중심 단일 화면으로 재설계한다. 날짜를 선택하면 페이지 이동 없이 같은 화면에서 그 날짜의 일정을 보고 추가·수정·삭제한다. 데스크탑은 좌측 캘린더(확대)+동기화 카드 / 우측 동행자+선택 일정 2분할, 모바일은 sticky 캘린더(월↔주 스와이프 압축)+선택 일정. DayList와 day 상세 페이지(/trips/[id]/day/[dayId])를 제거한다. spec 031의 후속 재설계. 연관 이슈 #622.

## Clarifications

1. **진입 시 초기 선택 날짜** — 여행 상세 진입 시 여행 기간 안에 오늘이 포함되면 오늘을, 포함되지 않으면 여행 첫날을 선택 상태로 표시합니다. WHY: 빈 패널로 진입하면 사용자가 추가 클릭을 해야 일정을 보므로, 가장 볼 가능성이 높은 날짜를 선택해 둡니다. 일반 캘린더 앱의 관행과 같습니다.
2. **day 상세 페이지 제거 처리** — `/trips/[id]/day/[dayId]` 라우트를 제거하고, 해당 경로로 직접 접근하면 여행 상세(`/trips/[id]`)로 이동시킵니다. WHY: 일정 조회·편집이 모두 상세 화면 안으로 들어오므로 별도 페이지가 중복됩니다. 기존 링크·북마크가 깨지지 않도록 상세로 보냅니다.
3. **모바일에서 동기화·동행자 위치** — 모바일(viewport < 1024px)에서는 외부 캘린더 동기화와 동행자를 본문에 펼치지 않고 `자세히` 진입점 한 곳으로 한 단계 뒤에 둡니다. WHY: 모바일의 주인공은 선택한 날짜의 일정 수행입니다. 부가 정보가 캘린더와 일정 사이 공간을 차지하지 않게 합니다.
4. **모바일 캘린더 압축 동작** — 모바일 캘린더는 상단에 고정(sticky)되고, 디폴트는 월 단위 표시이며 위로 스와이프하면 선택한 날짜가 속한 한 주만 남기고 압축됩니다. 아래로 스와이프하면 월 표시로 복귀합니다. WHY: 일정 목록을 길게 볼 때 캘린더가 화면을 적게 차지하도록 하는 일반 캘린더 앱의 표준 동작입니다.
5. **일정 편집 범위** — 선택한 날짜의 일정 패널에서 일정 추가·수정·삭제를 모두 처리합니다. 별도 페이지로 이동하지 않습니다. WHY: 단일 화면 완결이 본 재설계의 핵심 가치입니다.

> 본 피처는 spec 031의 두 결정을 의도적으로 번복합니다. 031 FR-005(셀 클릭 시 day 페이지로 라우팅)는 본 피처에서 인라인 선택으로 대체되고, 031 FR-009(선택 날짜 일정 패널을 본문에 노출 금지)는 본 피처에서 우측·하단 패널로 명시적으로 다시 도입됩니다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — 날짜를 선택해 같은 화면에서 일정을 본다 (Priority: P1)

여행 상세에 진입한 사용자는 캘린더에서 날짜를 클릭하면 페이지가 바뀌지 않고 같은 화면 안에서 그 날짜의 일정을 봅니다. 데스크탑에서는 우측 패널이, 모바일에서는 캘린더 아래 영역이 선택한 날짜의 일정으로 갱신됩니다. 진입 직후에도 빈 화면이 아니라 기본 선택 날짜의 일정이 이미 보입니다.

**Why this priority**: 날짜 클릭마다 별도 페이지로 이동하던 흐름이 본 재설계로 사라집니다. 같은 화면에서 날짜를 바꿔 가며 일정을 비교하는 것이 사용자가 가장 자주 하는 동작입니다.

**Independent Test**: /trips/[id]에서 캘린더의 서로 다른 날짜를 클릭했을 때 URL이 바뀌지 않고 일정 패널만 해당 날짜로 갱신되는지 확인하면 검증된다.

**Acceptance Scenarios**:

1. **Given** 여행 기간 안에 일정이 등록된 날짜가 있는 여행에 진입 **When** 화면이 렌더된다 **Then** 초기 선택 날짜(여행 기간 내 오늘, 없으면 첫날)의 일정이 패널에 이미 표시된다.
2. **Given** 상세 화면이 떠 있음 **When** 캘린더에서 다른 날짜 셀을 클릭 **Then** URL은 그대로이고 일정 패널이 클릭한 날짜의 일정으로 갱신된다.
3. **Given** 일정이 0건인 날짜를 선택 **When** 패널이 갱신된다 **Then** "등록된 일정이 없습니다" 안내와 함께 일정 추가 동선이 보인다.

---

### User Story 2 — 선택한 날짜의 일정을 그 자리에서 추가·수정·삭제한다 (Priority: P1)

사용자는 선택한 날짜의 일정 패널에서 일정을 추가하고, 기존 일정 항목을 눌러 수정하거나 삭제합니다. 별도 페이지(day 상세)로 이동하지 않고 모든 편집이 같은 화면에서 끝납니다.

**Why this priority**: 조회만 인라인이고 편집은 다른 페이지로 가면 단일 화면 완결이 깨집니다. 추가·수정·삭제까지 한 화면에서 끝나야 재설계의 가치가 완성됩니다.

**Independent Test**: 선택 날짜 패널에서 일정을 추가하면 패널 목록에 즉시 반영되고, 항목을 수정·삭제해도 페이지 이동 없이 목록이 갱신되는지 확인하면 검증된다.

**Acceptance Scenarios**:

1. **Given** 날짜가 선택된 상태 **When** 일정 추가 동선으로 새 일정을 저장 **Then** 페이지 이동 없이 선택 날짜 패널 목록에 새 일정이 나타난다.
2. **Given** 일정 항목이 있는 패널 **When** 항목을 눌러 수정하고 저장 **Then** 같은 화면에서 항목 내용이 갱신된다.
3. **Given** 일정 항목이 있는 패널 **When** 항목을 삭제 **Then** 같은 화면에서 항목이 목록에서 사라진다.
4. **Given** 기존 `/trips/[id]/day/[dayId]` 경로 **When** 직접 접근 **Then** 여행 상세(`/trips/[id]`)로 이동한다.

---

### User Story 3 — 데스크탑에서 캘린더와 일정을 좌우로 함께 본다 (Priority: P1)

데스크탑·풀스크린(viewport ≥ 1024px) 사용자는 좌측에 컬럼을 꽉 채운 큰 캘린더와 그 아래 외부 캘린더 동기화 카드를, 우측에 동행자와 그 아래 선택 날짜 일정을 함께 봅니다. day 목록은 화면 어디에도 나타나지 않습니다.

**Why this priority**: 현재 캘린더가 컬럼의 일부만 차지해 여백이 과도하고, 캘린더와 day 목록이 같은 정보를 중복 노출합니다. 캘린더 확대 + day 목록 제거 + 우측 일정 배치로 화면 밀도를 정리합니다.

**Independent Test**: viewport ≥ 1024px의 /trips/[id]에서 좌측 캘린더가 컬럼 폭을 거의 채우고, 그 아래 동기화 카드, 우측 상단 동행자, 우측 하단 선택 일정이 보이며, day 목록 요소가 0개인지 확인하면 검증된다.

**Acceptance Scenarios**:

1. **Given** viewport ≥ 1024px **When** 상세 화면이 렌더된다 **Then** 좌측 컬럼은 캘린더(상단) + 동기화 카드(하단), 우측 컬럼은 동행자(상단) + 선택 날짜 일정(하단)으로 배치된다.
2. **Given** 같은 조건 **When** 좌측 캘린더 폭을 좌측 컬럼 폭과 비교한다 **Then** 캘린더가 좌측 컬럼 폭을 거의 채우고 과도한 좌우 여백이 없다.
3. **Given** 같은 조건 **When** 화면 전체를 탐색한다 **Then** day 목록(DayList) 요소가 노출되지 않는다.

---

### User Story 4 — 모바일에서 캘린더를 고정한 채 일정에 집중한다 (Priority: P2)

모바일(viewport < 1024px) 사용자는 상단에 고정된 캘린더 아래에서 선택한 날짜의 일정에 집중합니다. 일정을 길게 볼 때 위로 스와이프하면 캘린더가 선택 주 한 줄로 압축돼 일정 영역이 넓어지고, 아래로 스와이프하면 월 표시로 돌아옵니다. 동기화와 동행자는 `자세히` 한 곳에 모여 본문을 차지하지 않습니다.

**Why this priority**: 모바일의 핵심 동작은 정해진 일정을 따라가는 것입니다. 데스크탑 레이아웃(P1) 정리 이후 도입해도 단독 가치가 있어 P2로 둡니다.

**Independent Test**: viewport < 1024px의 /trips/[id]에서 캘린더가 상단에 고정되고, 위로 스와이프 시 선택 주만 남으며, 동기화·동행자가 본문에 펼쳐지지 않고 `자세히`로 들어가는지 확인하면 검증된다.

**Acceptance Scenarios**:

1. **Given** viewport < 1024px **When** 상세 화면이 렌더된다 **Then** 캘린더가 상단에 고정되고 그 아래 선택 날짜 일정이 표시되며 동기화·동행자는 `자세히` 진입점으로만 노출된다.
2. **Given** 모바일 월 표시 상태 **When** 위로 스와이프 **Then** 캘린더가 선택 날짜가 속한 한 주만 남기고 압축되어 일정 영역이 넓어진다.
3. **Given** 모바일 압축(주) 상태 **When** 아래로 스와이프 **Then** 캘린더가 월 표시로 복귀한다.

---

### Edge Cases

- 일정이 0건인 여행에 진입 → 초기 선택 날짜(여행 기간 내 오늘 또는 첫날) 패널이 빈 상태 안내 + 일정 추가 동선을 보인다. 캘린더는 정상 표시된다.
- 여행 기간 밖 날짜를 캘린더에서 선택 → 패널이 그 날짜로 갱신되되 빈 상태 안내를 보인다(여행 기간 강조는 기간 내 날짜에만).
- 선택 날짜의 일정을 모두 삭제 → 패널이 빈 상태 안내로 전환되고 선택 상태는 유지된다.
- 모바일 압축(주) 상태에서 다른 주의 날짜로 이동 → 압축 상태를 유지하며 선택 주가 새 날짜의 주로 바뀐다.
- 동기화 카드·동행자 높이 합이 우측 컬럼에서 캘린더 높이보다 작을 때 → 빈 공간을 허용하고 비대칭 회귀로 보지 않는다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 사용자가 캘린더에서 날짜 셀을 클릭하면 시스템은 페이지를 이동하지 않고 선택 상태만 갱신하여 해당 날짜의 일정을 패널에 표시해야 한다.
- **FR-002**: 시스템은 여행 상세 진입 시 여행 기간 안에 오늘이 포함되면 오늘을, 포함되지 않으면 여행 첫날을 초기 선택 날짜로 표시해야 한다.
- **FR-003**: 사용자는 선택한 날짜의 일정 패널에서 일정을 추가·수정·삭제할 수 있어야 하며, 각 동작 후 페이지 이동 없이 패널 목록이 갱신되어야 한다.
- **FR-004**: 시스템은 day 상세 페이지(`/trips/[id]/day/[dayId]`) 라우트를 제거하고, 해당 경로로의 접근을 여행 상세(`/trips/[id]`)로 이동시켜야 한다.
- **FR-005**: 시스템은 모든 viewport에서 day 목록(DayList) 요소를 노출해서는 안 된다.
- **FR-006**: 시스템은 viewport ≥ 1024px에서 좌측 컬럼에 캘린더(상단)와 외부 캘린더 동기화 카드(하단)를, 우측 컬럼에 동행자(상단)와 선택 날짜 일정(하단)을 배치해야 한다.
- **FR-007**: 시스템은 viewport ≥ 1024px에서 좌측 캘린더가 좌측 컬럼 폭을 거의 채우도록 표시하여 과도한 좌우 여백을 두어서는 안 된다.
- **FR-008**: 시스템은 viewport < 1024px에서 캘린더를 상단에 고정(sticky)하고 그 아래에 선택 날짜 일정을 표시해야 한다.
- **FR-009**: 시스템은 viewport < 1024px에서 캘린더를 디폴트 월 표시로 두고, 위로 스와이프 시 선택 날짜가 속한 한 주만 남기고 압축, 아래로 스와이프 시 월 표시로 복귀시켜야 한다.
- **FR-010**: 시스템은 viewport < 1024px에서 외부 캘린더 동기화와 동행자를 본문에 펼치지 않고, 캘린더 상단 sticky 바의 `자세히` 버튼 한 곳으로 노출해야 한다.
- **FR-011**: 시스템은 일정이 0건인 선택 날짜에 대해 빈 상태 안내와 일정 추가 동선을 표시해야 한다.

### Key Entities

본 피처는 데이터 모델 변경 없이 기존 Trip / Day / Activity / TripCalendarLink / TripMember 엔티티를 화면에서 다르게 배치·편집한다. 신규 엔티티·스키마 변경 없음. 일정 추가·수정·삭제는 기존 Activity·Day 조작 경로를 재사용한다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 캘린더에서 서로 다른 날짜를 클릭해도 URL 경로가 바뀌지 않고 일정 패널만 갱신된다 (페이지 전이 0회).
- **SC-002**: 선택 날짜 패널에서 일정 추가·수정·삭제가 각각 페이지 이동 없이 1개 화면에서 완료된다.
- **SC-003**: viewport ≥ 1024px에서 좌측 캘린더 컬럼의 캘린더 폭이 좌측 컬럼 콘텐츠 폭의 90% 이상을 차지한다.
- **SC-004**: 모든 viewport에서 day 목록(DayList) 요소가 0개 노출된다.
- **SC-005**: `/trips/[id]/day/[dayId]` 경로 접근이 여행 상세로 100% 이동한다 (깨진 링크 0건).
- **SC-006**: viewport < 1024px에서 캘린더가 상단 고정되고 위로 스와이프 시 선택 주 1줄로 압축된다.
