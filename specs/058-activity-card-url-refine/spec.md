## Clarifications *(optional, add when drafting raises ambiguity)*

1. **하단 여백 축소 정도** — 활동이 없거나 적은 날의 하단 빈 영역을 한 화면 전체(직전 버전)에서 줄이되, 캘린더 월↔주 접힘이 스크롤에 의존하므로 접힘이 동작할 최소한의 스크롤 여지(콘텐츠가 화면을 약간 넘김)는 유지한다. 정확한 높이 값은 plan에서 결정한다.
2. **메모 클램프 줄 수** — 활동 카드 목록에서 메모는 최대 3줄까지 보이고 넘치면 말줄임표로 줄인다. 전문은 기존 활동 상세 보기에서 그대로 확인한다.
3. **URL 표시·입력 위치** — URL은 메모와 분리된 단일 항목이다. 입력 폼에서는 장소·메모와 구분된 입력 칸으로, 활동 카드·상세에서는 클릭 가능한 링크로 노출한다. 값이 없으면 어디에도 표시하지 않는다.
4. **빈 상태 vs 로딩 구분** — 활동 0건(빈 상태)은 안내 카드로, 활동을 아직 받지 못한 로딩 상태는 기존 스켈레톤으로 계속 구분한다. 둘을 같은 표시로 합치지 않는다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 선택 날짜 일정을 정돈된 화면으로 본다 (Priority: P1)

여행 상세에서 날짜를 선택하면, 그 날의 활동이 군더더기 보더 없이 카드만으로 보인다. 활동이 없는 날은 비어 있다는 안내가 카드로 분명히 보이고, 화면 아래로 과한 빈 공간이 남지 않는다.

**Why this priority**: 직전 버전에서 컨테이너 보더가 화면 높이만큼 늘어나 큰 빈 박스로 보이고, 활동 0건일 때 안내가 없어 비어 있는지 로딩 중인지 모호했다. 매일 보는 핵심 화면의 정돈도가 가장 큰 가치다.

**Independent Test**: 활동이 없는 날과 한두 건만 있는 날을 모바일에서 열어, (1) 컨테이너 보더로 인한 빈 박스가 없고 (2) 활동 0건 안내 카드가 보이며 (3) 하단 빈 여백이 직전 버전보다 줄되 캘린더가 스크롤로 주간 접힘되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 활동이 1건 이상인 날, **When** 그 날을 선택, **Then** 활동이 카드로만 보이고 그 카드들을 감싸는 별도 컨테이너 보더가 없다.
2. **Given** 활동이 0건인 날, **When** 그 날을 선택, **Then** "등록된 활동이 없습니다" 안내가 카드로 보이고, 편집 권한이 있으면 추가 동작 진입점이 함께 보인다.
3. **Given** 활동이 없거나 적은 날, **When** 모바일에서 아래로 스크롤, **Then** 캘린더가 주간으로 접히고, 직전 버전 대비 하단 빈 여백이 줄어 있다.
4. **Given** 활동을 아직 받지 못한 로딩 상태, **When** 그 날을 선택, **Then** 빈 상태 안내가 아니라 로딩 표시가 보인다.

### User Story 2 - 긴 메모를 목록에서 짧게 본다 (Priority: P2)

활동 카드에서 메모가 길어도 목록이 한눈에 들어오도록, 메모는 처음 몇 줄만 보이고 나머지는 말줄임표로 줄인다. 전문은 활동을 눌러 상세에서 본다.

**Why this priority**: 예약 메모처럼 긴 줄글이 카드를 가득 채워 목록 훑기가 불편하다. 가독성 개선이며 신규 기능은 아니라 P2.

**Independent Test**: 여러 줄짜리 긴 메모를 가진 활동 카드를 열어 목록에서 최대 3줄 + 말줄임표로 보이는지, 활동을 누른 상세에서는 전문이 보이는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 4줄 이상 메모를 가진 활동, **When** 목록 카드를 봄, **Then** 메모가 최대 3줄까지만 보이고 넘침은 말줄임표로 표시된다.
2. **Given** 같은 활동, **When** 카드를 눌러 상세를 엶, **Then** 메모 전문이 잘림 없이 보인다.
3. **Given** 2줄 이하 짧은 메모, **When** 목록 카드를 봄, **Then** 말줄임표 없이 전체가 보인다.

### User Story 3 - 활동에 URL을 따로 기록한다 (Priority: P1)

활동에 예약·티켓·문서 링크를 메모와 섞지 않고 별도 URL 항목으로 기록한다. 애플 캘린더에서 가져온 URL도 이 항목에 담긴다.

**Why this priority**: 애플 캘린더 연동(가져오기·기록)에서 URL은 별도 필드라 메모에 섞으면 정보가 흐트러진다. 도메인 데이터 구조를 바로잡는 신규 항목이라 P1.

**Independent Test**: 활동에 URL을 입력·저장한 뒤 카드·상세에서 링크로 노출되는지, API·도구 표현에 URL 항목이 일관되게 나타나는지, 기존 메모가 그대로 보존되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 활동 추가/편집 폼, **When** URL 입력 칸에 링크를 입력하고 저장, **Then** 그 활동의 URL이 저장되고 카드·상세에서 클릭 가능한 링크로 보인다.
2. **Given** URL이 비어 있는 활동, **When** 카드·상세를 봄, **Then** URL 자리가 표시되지 않는다(빈 항목 노출 없음).
3. **Given** 기존에 메모만 있던 활동, **When** 이 변경 이후 그 활동을 봄, **Then** 메모는 그대로이고 URL은 비어 있다.
4. **Given** API·도구로 활동을 조회·생성·수정, **When** URL을 포함해 요청·응답, **Then** URL 항목이 표현에 일관되게 포함된다.

---

### User Story 4 - 좁은 화면에서 폼 입력이 겹치지 않는다 (Priority: P2)

폭이 좁은 휴대폰(약 320px)에서도 활동 추가/편집 폼의 시작·종료 일시 입력이 서로 겹치지 않고 또렷이 보인다.

**Why this priority**: 아이폰 13 미니 등 320px 기기에서 시작·종료를 가로로 나란히 둬 종료 입력이 시작 위로 겹쳐 입력이 어렵다. 일부 기기 한정이라 P2.

**Independent Test**: 320px 폭에서 활동 폼을 열어 시작·종료 일시 입력이 겹치지 않고 각각 조작 가능한지 확인한다.

**Acceptance Scenarios**:

1. **Given** 320px 폭 화면, **When** 활동 추가/편집 폼을 엶, **Then** 시작·종료 일시 입력이 겹치지 않고 각각 온전히 보인다.
2. **Given** 넓은 화면, **When** 같은 폼을 엶, **Then** 기존 배치가 유지되어 회귀가 없다.

---

### User Story 5 - 가져오기 화면에서 provider를 분명히 구분한다 (Priority: P3)

외부 캘린더 가져오기 화면에서 애플 캘린더와 구글 캘린더가 각각 분명한 섹션으로 나뉘어, 어느 provider 영역인지 한눈에 구분된다.

**Why this priority**: 현재는 옆의 작은 설명만으로 구분돼 어느 provider인지 약하다. 정보 구조 개선이라 P3.

**Independent Test**: 가져오기 화면을 열어 애플·구글이 각각 제목이 분명한 별도 섹션으로 나뉘어 보이는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 가져오기 화면, **When** 화면을 봄, **Then** 애플 캘린더와 구글 캘린더가 각각 제목이 붙은 별도 섹션으로 구분된다.
2. **Given** 한쪽 provider만 연결된 상태, **When** 화면을 봄, **Then** 각 섹션이 자기 provider의 연결·미연결 상태를 자기 영역 안에서 보여준다.

### Edge Cases

- 메모가 정확히 클램프 경계(3줄)일 때 말줄임표 없이 보인다.
- URL 값이 형식상 링크가 아닐 때도 저장은 되며 표시 시 안전하게 다룬다(임의 스크립트 실행 없음).
- 매우 긴 단일 URL이 카드 폭을 넘을 때 줄바꿈으로 가로 넘침을 만들지 않는다.
- 320px보다 더 좁은 폭에서도 폼 입력 겹침이 없다.
- 하단 여백 축소 후에도 활동이 많은 날의 스크롤·캘린더 접힘이 정상 동작한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 여행 상세 하단 일정 영역은 활동을 카드로만 표시하고, 그 카드들을 감싸는 별도의 컨테이너 보더·배경을 두지 않는다.
- **FR-002**: 선택한 날짜에 활동이 0건이면 "등록된 활동이 없습니다" 취지의 빈 상태 안내를 카드로 표시한다. 편집 권한이 있으면 활동/일정 추가 진입점을 함께 제공한다.
- **FR-003**: 활동을 아직 받지 못한 로딩 상태는 빈 상태 안내와 구분되는 로딩 표시로 유지한다.
- **FR-004**: 활동이 없거나 적은 날의 하단 빈 여백을 직전 버전보다 줄이되, 모바일에서 캘린더 월↔주 접힘을 트리거할 최소 스크롤 여지는 유지한다.
- **FR-005**: 활동 카드 목록에서 메모는 최대 3줄까지 표시하고 넘치는 부분은 말줄임표로 줄인다.
- **FR-006**: 활동 상세 보기에서는 메모 전문을 잘림 없이 표시한다.
- **FR-007**: 활동은 메모와 분리된 단일 URL 속성을 가진다. 기존 메모 데이터는 변경하지 않고 URL 속성을 새로 추가하며, 기존 활동의 URL 초기값은 비어 있다.
- **FR-008**: 활동 추가/편집 폼은 메모·장소와 구분된 URL 입력 칸을 제공한다.
- **FR-009**: 활동 카드·상세는 URL이 있을 때만 클릭 가능한 링크로 노출하고, 없으면 표시하지 않는다.
- **FR-010**: 활동 조회·생성·수정의 API·도구 표현에 URL 속성이 일관되게 포함된다.
- **FR-011**: 약 320px 폭 화면에서 활동 추가/편집 폼의 시작·종료 일시 입력이 서로 겹치지 않고 각각 조작 가능하다. 넓은 화면의 기존 배치는 회귀 없이 유지한다.
- **FR-012**: 외부 캘린더 가져오기 화면은 애플 캘린더와 구글 캘린더를 각각 제목이 붙은 별도 섹션으로 구분하고, 각 섹션이 자기 provider의 연결·미연결 상태를 자기 영역 안에서 보여준다.

### Key Entities *(include if feature involves data)*

- **Activity(활동)**: 여행 일자에 속한 일정 항목. 기존 속성(제목·유형·시작/종료 일시·장소·비용·통화·예약 상태·메모 등)에 더해 **URL**(예약·티켓·문서 링크, 선택값) 속성을 새로 가진다. 메모와 URL은 독립 속성이다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 활동이 0건인 날을 열면 사용자가 추가 동작 없이 "비어 있음"을 즉시 인지한다(빈 박스·무표시 0건).
- **SC-002**: 활동이 1건인 날의 하단 빈 여백이 직전 버전보다 줄어들면서도, 모바일에서 한 번의 아래 스크롤로 캘린더가 주간으로 접힌다.
- **SC-003**: 4줄 이상 메모를 가진 활동 카드의 메모 표시 높이가 3줄 이내로 제한된다.
- **SC-004**: URL을 입력한 활동의 100%가 카드·상세·API·도구 표현에서 URL을 일관되게 노출하며, 기존 활동의 메모는 100% 보존된다.
- **SC-005**: 320px 폭에서 활동 폼의 시작·종료 입력이 겹치지 않는다(겹침 0건).
- **SC-006**: 가져오기 화면에서 사용자가 애플·구글 영역을 제목만으로 구분한다.
