## Clarifications *(optional, add when drafting raises ambiguity)*

1. **단기 만료 기본값** — 자동 발급 토큰의 기본 만료는 30일. 수기 발급 토큰의 만료 정책(현행: 선택, 무만료 가능)은 바꾸지 않는다.
2. **인증 방식 불변** — 토큰을 단기 JWT로 바꾸지 않는다. 기존 발급 토큰 모델(폐기·목록·만료 가능)을 그대로 쓰고, 자동 발급 경로에 만료를 부여할 뿐이다. 웹 세션 로그인은 손대지 않는다.
3. **토큰 보관** — 발급 토큰은 사용자 기기의 OS 보안 저장소(로컬)에 저장한다. 레포·로그·평문 파일에 남기지 않는다.
4. **헤드리스 device flow 제외** — 브라우저가 없는 환경(SSH·CI)용 별도 인증은 이 피처 범위 밖(후속). 본 피처는 브라우저를 열 수 있는 환경을 전제한다. 그런 환경에선 수기 발급 폴백을 쓴다.
5. **재인증 책임 분담** — 만료 시, 자동 갱신이 가능한 소비자(런타임 통합)는 자동 재인증한다. 그렇지 않은 임의 소비자(스크립트·헬퍼)는 "다시 로그인하라"는 안내를 받고 사용자가 커맨드를 재실행한다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 단일 커맨드로 로그인해 API를 쓴다 (Priority: P1)

API/CLI로 여행 데이터를 다루려는 사용자가 토큰을 웹에서 수기 발급·복사하지 않는다. 단일 커맨드를 실행하면 브라우저에서 Google 로그인이 열리고, 로그인하면 토큰이 발급돼 기기에 자동 저장된다. 이후 임의 API 호출이 바로 동작한다.

**Why this priority**: 수기 발급·복사·저장의 번거로움이 이 피처의 출발점이다. 이 흐름 하나만으로도 "수기 PAT 생략"이라는 핵심 가치를 전달한다.

**Independent Test**: 토큰이 없는 상태에서 로그인 커맨드 1회 실행 → 브라우저 로그인 → 임의 API 호출(GET)이 인증 성공하는지 확인.

**Acceptance Scenarios**:

1. **Given** 저장된 토큰이 없음, **When** 로그인 커맨드 실행 후 브라우저에서 Google 로그인 완료, **Then** 토큰이 기기 보안 저장소에 저장되고 안내가 성공으로 끝난다.
2. **Given** 저장된 유효 토큰이 있음, **When** 임의 API 호출, **Then** 추가 입력 없이 인증되어 응답을 받는다.
3. **Given** 로그인 도중 사용자가 브라우저를 닫거나 일정 시간 응답 없음, **When** 대기 시간 초과, **Then** 실패가 분명히 안내되고 저장된 토큰은 바뀌지 않는다.

---

### User Story 2 - 토큰이 단기 만료되고 만료 시 재인증된다 (Priority: P1)

자동 발급 토큰은 단기 만료를 가진다. 만료되면, 자동 갱신이 가능한 소비자는 스스로 재인증해 끊김 없이 잇고, 그렇지 않은 소비자는 다시 로그인하라는 안내를 받는다. 장수명 토큰을 무기한 보관하지 않는다.

**Why this priority**: "번거로워서 장수명 토큰을 방치"하던 보안 부담을 푸는 것이 이 피처의 또 다른 핵심이다. 자동 발급에 만료가 없으면 문제를 절반만 푼다.

**Independent Test**: 만료된(또는 만료 처리한) 토큰으로 API 호출 → 자동 갱신 소비자는 재인증 후 성공, 그 외는 재로그인 안내 메시지 확인.

**Acceptance Scenarios**:

1. **Given** 자동 발급된 토큰, **When** 발급 직후 토큰 정보를 확인, **Then** 만료 시각이 설정돼 있다(무기한 아님).
2. **Given** 만료된 토큰 + 자동 갱신 가능 소비자, **When** API 호출, **Then** 자동으로 재인증하고 원래 요청이 성공한다.
3. **Given** 만료된 토큰 + 일반 소비자(스크립트), **When** API 호출, **Then** 인증 실패가 "다시 로그인" 안내와 함께 분명히 표시된다.

---

### User Story 3 - 수기 발급을 폴백으로 계속 쓴다 (Priority: P2)

토큰을 미리 세팅해두고 쓰거나 자동 인증이 어려운 환경의 사용자는 기존처럼 토큰을 수기로 발급해 쓸 수 있다.

**Why this priority**: 자동 흐름이 모든 환경을 덮지 못한다(브라우저 없는 환경 등). 기존 사용자의 동작을 깨지 않는 것이 중요하나, 신규 가치보다는 보존이라 P2.

**Independent Test**: 수기 발급 페이지에서 토큰을 만들어 저장 → 임의 API 호출 성공. 자동 흐름과 무관하게 동작.

**Acceptance Scenarios**:

1. **Given** 자동 인증을 쓰지 않음, **When** 수기로 토큰을 발급해 저장, **Then** 임의 API 호출이 인증된다.
2. **Given** 수기 발급 토큰, **When** 만료 정책을 지정(또는 무만료), **Then** 지정대로 동작한다(수기 정책은 이번 변경의 영향을 받지 않는다).

---

### User Story 4 - 기존 인증 경로가 회귀 없이 동작한다 (Priority: P2)

기존 설치 스크립트·런타임 통합·웹 세션 로그인 사용자는 이 변경 이후에도 그대로 로그인·호출할 수 있다.

**Why this priority**: 인증은 모든 기능의 관문이라 회귀가 치명적이다. 신규 기능은 아니지만 반드시 지켜야 한다.

**Independent Test**: 기존 설치 흐름·런타임 통합·웹 로그인을 각각 수행 → 모두 정상 인증.

**Acceptance Scenarios**:

1. **Given** 기존 설치 스크립트, **When** 실행, **Then** 종전과 같이 인증·토큰 저장이 끝난다.
2. **Given** 웹 브라우저 사용자, **When** Google 로그인, **Then** 종전과 같이 세션이 만들어진다.

### Edge Cases

- 로그인 콜백을 받는 로컬 수신 지점이 이미 점유됐을 때 분명한 실패 안내.
- 동시에 두 번 로그인/재인증이 트리거될 때 중복 발급이 폭주하지 않는다(한 번만 진행).
- 콜백 위변조(다른 출처/위조 state) 시 토큰을 저장하지 않는다.
- 만료 직전 토큰으로 진행 중 만료가 걸렸을 때, 자동 갱신 소비자는 한 번의 재인증으로 복구한다.
- 자동 발급과 수기 발급 토큰이 함께 있을 때 어느 것이 쓰이는지 예측 가능해야 한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 사용자는 단일 커맨드 실행으로 브라우저 Google 로그인을 거쳐 토큰을 발급받아 기기에 저장할 수 있다. 임의 API 소비자가 이 진입점을 쓸 수 있어야 한다(설치 스크립트·런타임 통합에 종속되지 않음).
- **FR-002**: 발급된 토큰은 기기의 OS 보안 저장소에 저장되고, 레포·로그·평문 파일에 남지 않는다.
- **FR-003**: 로그인 콜백은 출처와 1회용 식별자(state)를 검증해 위변조 시 토큰을 저장하지 않는다.
- **FR-004**: 자동 발급 토큰에는 단기 만료(기본 30일)가 부여된다.
- **FR-005**: 토큰 만료 시 — 자동 갱신이 가능한 소비자는 재인증 후 원 요청을 잇고, 그 외 소비자는 "다시 로그인" 안내와 함께 인증 실패를 분명히 알린다.
- **FR-006**: 수기 토큰 발급 기능과 그 만료 정책은 이 변경의 영향을 받지 않고 그대로 유지된다(폴백).
- **FR-007**: 기존 설치 스크립트·런타임 통합·웹 세션 로그인은 회귀 없이 동작한다.
- **FR-008**: 동일 사용자에 대해 동시에 다발하는 로그인/재인증은 한 번만 진행되어 토큰이 중복 폭주하지 않는다.
- **FR-009**: 발급 토큰은 출처를 구분할 수 있는 이름과 발급 시각을 가져, 사용자가 목록에서 식별·폐기할 수 있다.

### Key Entities *(include if feature involves data)*

- **발급 토큰(Access Token)**: 사용자가 API 인증에 쓰는 토큰. 기존 모델을 그대로 쓴다 — 이름(출처 구분), 발급/마지막 사용 시각, **만료 시각**, 폐기 가능. 본 피처는 자동 발급 경로가 이 만료 시각을 단기로 채우게 한다. 새 저장 구조(별도 테이블)는 도입하지 않는다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 신규 사용자가 토큰을 웹에서 수기 발급·복사하지 않고, 커맨드 1회 + 브라우저 로그인만으로 API 호출에 성공한다.
- **SC-002**: 자동 발급 토큰의 100%가 만료 시각을 가진다(무기한 0건).
- **SC-003**: 만료된 토큰으로 호출 시, 자동 갱신 소비자는 사용자 추가 조작 없이 복구하고, 그 외 소비자는 재로그인 안내를 받는다.
- **SC-004**: 기존 설치·런타임·웹 로그인 경로의 인증 성공률이 변경 전과 동일하다(회귀 0).
- **SC-005**: 수기 발급 흐름은 변경 전과 동일하게 동작한다.
