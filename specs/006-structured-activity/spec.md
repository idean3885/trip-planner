# Feature Specification: 일정 구조화

**Feature Branch**: `006-structured-activity`
**Created**: 2026-04-15
**Status**: Draft
**Input**: User description: "v2.1 일정 구조화: Day.content 마크다운 블롭을 Activity/Accommodation/Transport 정규화 스키마로 전환. 구조화 폼 UI + 컴포넌트 기반 렌더링 + MCP Activity CRUD 도구 확장."

## User Scenarios & Testing

### User Story 1 — AI 에이전트가 구조화된 활동을 추가한다 (Priority: P1)

사용자가 AI 에이전트에서 "3일차에 벨렘탑 관광 추가해줘, 오전 10시부터 2시간"이라고 말하면, MCP 도구가 시간/장소/카테고리가 분리된 활동 데이터를 생성하고 웹에서 구조화된 카드로 즉시 확인할 수 있다.

**Why this priority**: 프로젝트의 핵심 가치(AI → 웹 반영)가 마크다운 텍스트가 아닌 구조화된 데이터로 동작해야 검색/정렬/필터 등 후속 기능의 토대가 된다.

**Independent Test**: MCP create_activity 호출 → 웹에서 시간/장소/카테고리가 분리된 카드로 표시되는지 확인

**Acceptance Scenarios**:

1. **Given** 여행에 3일차가 존재할 때, **When** MCP로 시간/장소/카테고리를 포함한 활동을 생성하면, **Then** 웹에서 해당 일자에 구조화된 활동 카드가 표시된다
2. **Given** 활동이 존재할 때, **When** MCP로 시간이나 장소를 수정하면, **Then** 웹에서 수정된 내용이 즉시 반영된다
3. **Given** 하루에 여러 활동이 있을 때, **When** 웹에서 일자를 조회하면, **Then** 활동이 시간순으로 정렬되어 표시된다

---

### User Story 2 — 웹에서 구조화 폼으로 활동을 관리한다 (Priority: P2)

사용자가 웹에서 일자 상세 페이지에 들어가면, 기존 마크다운 편집기 대신 시간/장소/카테고리/메모 필드가 분리된 폼으로 활동을 추가/수정/삭제할 수 있다.

**Why this priority**: 웹 UI가 구조화 데이터를 제대로 보여주고 편집할 수 있어야 AI 결과를 사용자가 확인하고 미세 조정할 수 있다.

**Independent Test**: 웹에서 활동 추가 폼 → 시간/장소/카테고리 입력 → 저장 → 카드로 표시 확인

**Acceptance Scenarios**:

1. **Given** 일자 상세 페이지에서, **When** "활동 추가" 버튼을 누르면, **Then** 시간/장소/카테고리/메모 필드가 있는 폼이 나타난다
2. **Given** 활동 폼에 정보를 입력하고, **When** 저장하면, **Then** 활동 카드가 시간순으로 추가된다
3. **Given** 기존 활동 카드에서, **When** 편집 버튼을 누르면, **Then** 기존 정보가 채워진 폼이 나타난다
4. **Given** 기존 활동 카드에서, **When** 삭제 버튼을 누르면, **Then** 확인 후 활동이 제거된다

---

### User Story 3 — 기존 마크다운 일정을 활동으로 변환한다 (Priority: P3)

기존 여행의 Day.content(마크다운)가 있는 경우, 사용자가 "구조화 변환" 기능을 통해 마크다운을 활동 단위로 분리할 수 있다. 변환 전까지는 기존 마크다운이 읽기 전용으로 표시된다.

**Why this priority**: 기존 데이터 호환성이 보장되어야 사용자 이탈이 없다. 단, 신규 활동 생성이 먼저 동작해야 하므로 P3.

**Independent Test**: 마크다운이 있는 기존 일자 → "구조화 변환" 클릭 → AI가 활동 단위로 분리 → 확인 후 저장

**Acceptance Scenarios**:

1. **Given** 마크다운 콘텐츠가 있는 기존 일자에서, **When** 페이지를 열면, **Then** 마크다운이 읽기 전용으로 렌더링된다
2. **Given** 마크다운이 있고 활동이 없는 일자에서, **When** "구조화 변환" 버튼을 누르면, **Then** 마크다운 내용이 활동 단위로 분리되어 미리보기가 표시된다
3. **Given** 변환 미리보기 상태에서, **When** 확인하면, **Then** 활동 데이터가 저장되고 기존 마크다운은 보관된다

---

### User Story 4 — 변경 이력을 추적한다 (Priority: P4)

프로젝트에 CHANGELOG를 도입하여 버전별 변경 사항을 추적한다. README는 v2.1 기능에 맞게 업데이트한다.

**Why this priority**: 버저닝이 지속됨에 따라 변경 이력 추적이 필요. 기능 자체에는 영향 없으므로 P4.

**Independent Test**: CHANGELOG.md에 v2.0.0~v2.1.0 변경 사항이 기록되어 있고, README가 최신 기능을 반영하는지 확인

**Acceptance Scenarios**:

1. **Given** CHANGELOG.md가 존재할 때, **When** 내용을 확인하면, **Then** 버전별 변경 사항이 날짜와 함께 기록되어 있다
2. **Given** v2.1 기능이 완료되었을 때, **When** README를 확인하면, **Then** 구조화 활동 기능과 새 MCP 도구가 반영되어 있다

---

### Edge Cases

- 활동 시간이 겹치는 경우: 경고 표시하되 저장은 허용 (여행 일정 특성상 이동 시간 등 중복 가능)
- 시간 없는 활동: "종일" 또는 시간 미지정으로 처리, 시간순 정렬에서 마지막에 배치
- 마크다운 변환 실패: 원본 마크다운 보존, 수동 활동 추가로 폴백
- 활동이 0개인 일자: 빈 상태 메시지 + 활동 추가 유도
- Day.content와 Activity가 동시에 존재: 활동 목록 우선 표시, 마크다운은 접힌 상태로 보관

## Requirements

### Functional Requirements

- **FR-001**: 시스템은 하루(Day)에 여러 활동(Activity)을 연결할 수 있어야 한다 (1:N 관계)
- **FR-002**: 활동은 시작 시간, 종료 시간, 장소명, 카테고리, 메모 필드를 가져야 한다
- **FR-003**: 활동 카테고리는 관광, 식사, 이동, 숙소, 쇼핑, 기타를 포함해야 한다
- **FR-004**: 활동은 하루 내에서 시간순으로 정렬되어야 한다
- **FR-005**: 활동의 순서를 사용자가 수동으로 변경할 수 있어야 한다
- **FR-006**: MCP 도구로 활동 생성/조회/수정/삭제가 가능해야 한다
- **FR-007**: 기존 Day.content(마크다운)가 있는 일자는 읽기 전용으로 표시되어야 한다
- **FR-008**: 기존 마크다운을 활동 단위로 변환하는 기능을 제공해야 한다
- **FR-009**: 숙소와 이동은 활동의 카테고리로 처리한다 (별도 엔티티가 아닌 카테고리 구분)
- **FR-010**: 활동에 예상 비용(금액, 통화) 필드를 포함해야 한다
- **FR-011**: 활동에 예약 상태(사전 예약 필수/권장/현장 구매/불필요) 필드를 포함해야 한다
- **FR-012**: README를 v2.1 기능에 맞게 업데이트해야 한다
- **FR-013**: CHANGELOG를 도입하여 버전별 변경 이력을 관리해야 한다

### Key Entities

- **Activity**: 하루 내 개별 활동. 시간, 장소, 카테고리, 메모, 비용, 예약 상태를 가진다. Day에 종속(1:N).
- **Day**: 기존 엔티티. content(마크다운, 레거시) + activities(구조화, 신규) 공존. 활동이 있으면 구조화 뷰, 없으면 마크다운 뷰.
- **Category**: 활동 유형 분류. 관광/식사/이동/숙소/쇼핑/기타. 고정 목록.

## Success Criteria

### Measurable Outcomes

- **SC-001**: AI 에이전트가 시간/장소/카테고리를 포함한 활동을 생성하면 웹에서 3초 이내에 구조화된 카드로 확인 가능
- **SC-002**: 웹에서 활동 추가/수정/삭제를 5번의 탭/클릭 이내로 완료 가능
- **SC-003**: 기존 마크다운 일정이 있는 여행에서 데이터 손실 없이 구조화 변환 가능
- **SC-004**: MCP 도구 목록이 기존 14개에서 Activity CRUD 추가로 18개 이상

## Scope

### In Scope

- Activity 데이터 모델 + DB 마이그레이션
- 웹 구조화 폼 UI (추가/수정/삭제)
- 컴포넌트 기반 렌더링 (ActivityCard, 카테고리별 구분)
- MCP Activity CRUD 도구 (create, update, delete, reorder)
- 기존 마크다운 → 활동 변환 기능
- Day.content 레거시 호환 (읽기 전용 표시)
- README v2.1 업데이트
- CHANGELOG.md 도입 (v2.0.0~v2.0.1 소급 + v2.1.0)

### Out of Scope

- 지도 연동 (좌표 기반 장소 표시)
- 실시간 협업 편집 (동시 수정 충돌 처리)
- 활동 템플릿 (자주 쓰는 활동 저장/재사용)
- 예산 자동 집계 (활동 비용 합산 대시보드)
- 사용자 정의 카테고리

## Assumptions

- 숙소/교통을 별도 엔티티로 분리하지 않고 Activity의 카테고리로 처리한다. 필드가 충분히 유사하고(시간, 장소, 비용), 별도 테이블은 복잡도 대비 이점이 적다.
- 마크다운 → 활동 변환은 MCP(AI)를 통해 수행한다. 패턴 매칭 파서를 직접 구현하지 않는다.
- 활동 시간은 현지 시간 기준이다 (기존 Day.date와 동일한 타임존 정책).
- 카테고리는 고정 목록으로 시작하고, 사용자 정의 카테고리는 후속 버전에서 검토한다.

## Dependencies

- 기존 Day 모델, Trip/Member 인증 체계 (v2.0에서 완료)
- DB 마이그레이션 (기존 ORM + DB 인프라)
- MCP 서버 (기존 패키지)
