# Feature Specification: 디자인 시스템 기반 제정 — Tailwind v4 + shadcn/ui + 핸드오프 + 업무 프로세스

**Feature Branch**: `012-shadcn-design-system`
**Created**: 2026-04-19
**Status**: Draft
**Input**: User description: "v2.4.3 마일스톤 통합 스펙. 추적 이슈 #250(Tailwind 3→4), #270(shadcn 도입 + 마이그레이션 + 핸드오프 + 문서). 라이트 모드 단독, 수동 트리거 + 개발자 게이트 협업 모델."

## Clarifications *(confirmed decisions)*

1. **CSS 엔진은 Tailwind v4로 전환** — `@theme` CSS-first 토큰 체계가 디자이너 DTCG 워크플로우·shadcn 변수 체계와 정합. v3를 유지하면 토큰 이중 운영이 불가피해 선결로 묶는다. (추적 이슈 #250)
2. **UI 라이브러리는 shadcn/ui** — Radix 기반 접근성 자동, v0.dev 100% 호환, 코드 vendoring으로 외부 종속 0. v2 AX 방향(팀 기능·디자이너 협업)과 정합. (추적 이슈 #270)
3. **라이트 모드 단독** — 다크 모드는 본 스펙 범위에서 도입하지 않는다. 1인 디자이너 합류 초기 공수 2배 리스크 회피. 사용자 요청 누적 시 별도 마일스톤에서 재검토.
4. **협업은 수동 트리거 + 개발자 게이트** — 자동화 ROI 낮음. 디자이너가 이슈/PR에 핸드오프 첨부 → 개발자가 AI 세션에서 변환 → 개발자가 도메인·접근성·회귀 책임지고 머지. 6개월 운영 후 병목 발생 시점에 자동화 단계 도입.
5. **컴포넌트 마이그레이션은 Phase 1(폼 계열)만** — 복합 컴포넌트(`ActivityCard`, `ActivityList`, `DayEditor`, `MemberList`)는 Phase 2로 분리하여 별도 마일스톤·이슈에서 처리. 본 스펙은 시각 회귀 리스크가 낮은 폼 계열 6종으로 한정.
6. **웹 프레임워크 버전은 현행 유지** — 메이저 업그레이드(#249)는 v2.4.2 트랙에서 별도로 처리. 본 스펙은 현행 버전에서 완결하며 메이저 업그레이드 상호운용은 v2.4.2 머지 후 회귀 검증으로 확인한다.

## Metatag Conventions *(normative, inherited from PR #204)*

본 피처의 tasks.md·plan.md는 네 종 메타태그를 통해 후속 자동 검증과 연결된다:

- `[artifact: <path>|<path>::<symbol>]` — 산출 파일 식별자(drift 감사 기준)
- `[why: <short-tag>]` — 추적 그룹 키(plan↔tasks 커버리지·이슈 합산)
- `[multi-step: N]` — plan bullet이 다단 작업일 때 최소 매핑 태스크 수(N ≥ 2)
- `[migration-type: schema-only | data-migration]` — 마이그레이션 산출물 구분

형식 검증은 `.specify/scripts/bash/validate-metatag-format.sh`가 수행한다. 의미 검증은 각 US의 validator가 담당한다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 디자인 시스템 기반 위에서 신규 UI를 즉시 추가 (Priority: P1)

개발자·AI 에이전트가 "이 프로젝트에 새 버튼/다이얼로그 컴포넌트를 추가해줘"라는 요청을 받았을 때, 토큰·variants·접근성 규칙이 이미 정의된 디자인 시스템 위에서 표준 명령(`npx shadcn@latest add <component>`) 한 줄로 재현 가능하게 컴포넌트를 확보한다. 기존 앱은 시각적으로 동등하게 유지되며, 다크 모드 분기가 제거되고 단일 라이트 테마로 정리된다.

**Why this priority**: 디자인 시스템이 없는 상태에서는 이후 모든 슬라이스(마이그레이션·핸드오프·문서)가 공중에 뜬다. 반대로 이 슬라이스만 완료되어도 "신규 UI를 일관된 규칙으로 즉시 추가 가능"이라는 가장 큰 개발·협업 가치가 성립한다. AI 코딩 에이전트 산출물을 그대로 수용할 수 있는 가장 작은 MVP 표면이다.

**Independent Test**: 마이그레이션·핸드오프·문서 슬라이스가 하나도 완료되지 않은 상태에서도, (1) 앱이 기존과 시각적으로 동등하게 렌더되고 (2) `npx shadcn@latest add button` 등 신규 컴포넌트 추가 명령이 성공하고 (3) 추가된 컴포넌트가 프로젝트 토큰 체계에서 렌더되며 (4) 다크 모드 분기가 코드베이스에서 제거되어 있으면 본 스토리는 완료다.

**Acceptance Scenarios**:

1. **Given** 개발자가 피처 브랜치에서 빈 컴포넌트 파일 하나를 필요로 할 때, **When** 프로젝트 표준 CLI로 신규 컴포넌트를 추가하면, **Then** 해당 컴포넌트가 프로젝트 색상·타이포·간격 토큰에 즉시 결합되어 별도 수정 없이 렌더된다.
2. **Given** 디자인 시스템 전환 전·후의 동일 페이지를 비교할 때, **When** 로그인·여행 목록·여행 상세·설정 등 주요 플로우를 순회하면, **Then** 사용자 체감 시각 변화는 "개선 또는 동등" 범위이며 레이아웃 깨짐·폰트 치환·색상 역전이 관찰되지 않는다.
3. **Given** 이전 코드에 `.dark` 클래스 선택자 또는 `prefers-color-scheme` 분기가 있었을 때, **When** 전환 후 코드베이스를 탐색하면, **Then** 관련 분기가 제거되고 단일 라이트 테마 토큰만 남아 있다.
4. **Given** 빌드·타입체크·테스트 파이프라인이 트리거될 때, **When** 전환 커밋 기준으로 CI를 실행하면, **Then** 모든 스테이지가 전환 전과 동일한 상태(통과)로 완료된다.

### User Story 2 - 앱 사용자가 마이그레이션된 폼에서 개선된 접근성·일관성을 체감 (Priority: P2)

여행 계획을 편집하거나 계정을 관리하는 이용자가 폼 상호작용(활동 생성·초대·삭제·오늘 이동·로그인) 시 키보드 포커스 링, 라벨 연결, 오류 안내, 버튼·입력의 일관된 규격을 체감한다. 시각적으로는 기존과 동등하거나 개선되며, 보조 기술(스크린 리더·키보드 내비)로도 주요 동작이 막히지 않는다.

**Why this priority**: US1만으로는 새 컴포넌트가 추가 가능할 뿐 기존 사용자 경험은 그대로다. 본 스토리가 완료되면 가장 자주 쓰이는 폼 6종이 디자인 시스템 위로 올라오며, 이후 복합 컴포넌트(Phase 2)의 참조 사례가 된다. P1과 독립 PR로 분리 가능.

**Independent Test**: 마이그레이션 대상 폼 6종(`ActivityForm`, `AuthButton`, `DeleteTripButton`, `LeaveTripButton`, `InviteButton`, `TodayButton`)을 실제로 사용하는 플로우에서 (1) 시각 회귀 스크린샷이 기존과 "동등 또는 개선" 판정이고 (2) Tab·Shift+Tab으로 모든 포커스 가능 요소를 순회할 수 있으며 (3) 라벨·입력·오류 메시지가 의미적으로 연결되어 있으면 완료.

**Acceptance Scenarios**:

1. **Given** 사용자가 키보드만 사용해 여행 활동 생성 플로우에 진입할 때, **When** Tab 키로 포커스를 이동하면, **Then** 입력·버튼·취소·저장 순으로 포커스가 이동하며 현재 포커스 위치가 시각적으로 항상 명확하다.
2. **Given** 활동 삭제·여행 탈퇴·초대 생성 등 확인 다이얼로그를 사용자가 열 때, **When** Esc 키를 누르면, **Then** 다이얼로그가 닫히고 포커스가 열기 직전 요소로 복귀한다.
3. **Given** 로그인 버튼·오늘 이동 버튼 등 액션 버튼을 사용자가 클릭할 때, **When** 동일 역할의 두 버튼을 나란히 놓고 비교하면, **Then** 크기·여백·폰트·상태(hover/active/disabled)가 서로 어긋나지 않는다.
4. **Given** 마이그레이션 전·후 스크린샷을 주요 뷰포트(모바일 375px·데스크톱 1280px)에서 촬영할 때, **When** 대응 프레임을 나란히 비교하면, **Then** 사용자 체감 차이가 "개선 또는 동등" 범위이며 의도하지 않은 레이아웃 이동이 없다.

### User Story 3 - 디자이너 산출물을 결정적 절차로 코드에 반영 (Priority: P3)

Figma에서 작업한 디자이너가 토큰 변경(색상·간격·타이포) 또는 신규 컴포넌트 디자인을 개발자에게 넘길 때, 첨부 형식과 처리 절차가 이미 결정되어 있어 개발자는 짐작 없이 재현 가능한 단계로 코드에 반영한다. 개발자는 도메인·접근성·회귀 검증 책임을 게이트로 수행하며 머지 권한을 유지한다.

**Why this priority**: 디자이너가 합류하기 전 제정해두어야 합류 첫 핸드오프부터 파이프라인이 돈다. US1·US2 완료와 독립적으로 "형식·절차 자체"의 완결이 가치이며, 실제 첫 핸드오프는 개발자 단독 dry-run으로도 검증 가능. 자동화 고도화(Figma webhook·CI 자동 PR)는 범위 외로 둔다.

**Independent Test**: (1) 디자이너용 토큰 원본 파일(W3C DTCG 형식) 예시가 저장소에 존재하고 (2) 프로젝트 표준 빌드 스크립트로 해당 원본을 CSS 토큰 변수로 재생성할 수 있으며 (3) GitHub 이슈 생성 UI에 "디자이너 핸드오프" 템플릿이 선택 가능하고 (4) 개발자 단독으로 예시 토큰 변경을 한 번 돌려 결과 PR이 생성되면 완료.

**Acceptance Scenarios**:

1. **Given** 디자이너가 토큰 변경 산출물을 정해진 형식(색상·간격·타이포 포함)으로 저장소에 제출했을 때, **When** 개발자가 프로젝트 표준 토큰 빌드 스크립트를 실행하면, **Then** CSS 토큰 변수 파일이 재생성되고 앱 전반의 색·여백·폰트가 변경에 일관되게 반영된다.
2. **Given** 디자이너·기획자가 신규 UI 요청을 등록하려 할 때, **When** GitHub 이슈 생성 화면을 열면, **Then** "디자이너 핸드오프" 템플릿이 기본 제공 옵션으로 선택 가능하며 필수 필드(Figma URL, 스크린샷, variants/states, 인터랙션, 데이터 바인딩 의도, 토큰 변경 여부)가 안내된다.
3. **Given** 개발자가 핸드오프 이슈를 받았을 때, **When** 저장소 내 명문화된 절차 문서를 따르면, **Then** "도구 준비 → 변환 → 검토 체크포인트 → PR 생성"의 단계가 순서와 책임이 모호함 없이 기술되어 있다.
4. **Given** 필수 필드가 누락된 핸드오프 이슈가 제출될 때, **When** 템플릿이 렌더되면, **Then** 누락 필드는 시각적으로 구분되어 디자이너가 제출 전에 인지한다.

### User Story 4 - 신규 합류자·AI 에이전트가 단일 문서에서 업무 흐름을 파악 (Priority: P4)

새로 합류하는 디자이너·기획자·외부 개발자 또는 AI 에이전트가 "이슈 → 마일스톤 → 브랜치 → 릴리즈 → 디자이너 협업 → 핫픽스"의 전 과정을 하나의 문서에서 순서·책임·예외를 포함해 파악한다. AI 에이전트는 모호한 흐름에서 이 문서를 1차 컨텍스트로 자동 참조한다.

**Why this priority**: 문서 없이도 US1~US3는 기술적으로 작동한다. 그러나 사람·에이전트가 늘어날수록 의사결정이 개별 Claude 세션·대화에 흩어져 유지비가 증가한다. 문서는 체감 가치는 가장 뒤지만 6개월 관점 복리가 크다. US1~3과 완전 독립.

**Independent Test**: (1) 업무 프로세스 단일 문서가 저장소에 존재하고 (2) 명시된 7개 섹션(팀 구성·역할, 이슈 흐름, 버전·릴리즈, 디자이너 협업, AI 에이전트 활용, 마일스톤 운영, 핫픽스) 모두가 채워져 있으며 (3) 디자이너 핸드오프 상세 문서가 해당 문서에서 링크되고 (4) AI 에이전트 1차 컨텍스트(CLAUDE.md)와 저장소 색인(docs/README.md)에서 이 문서로 진입할 수 있으면 완료.

**Acceptance Scenarios**:

1. **Given** 신규 합류자가 저장소 `docs/` 디렉토리를 열었을 때, **When** 색인 문서를 읽으면, **Then** 업무 프로세스·디자이너 핸드오프 두 문서의 위치·용도가 두 줄 이내로 안내된다.
2. **Given** 외부 개발자가 첫 이슈를 할당받았을 때, **When** 업무 프로세스 문서를 처음부터 읽으면, **Then** "이슈 → 브랜치 → PR → 머지 → 릴리즈"의 전체 흐름과 각 단계에서 본인이 할 일이 독립적으로 실행 가능할 만큼 명확하다.
3. **Given** AI 에이전트가 피처 구현 중 릴리즈·마일스톤·핫픽스 등 흐름 관련 모호함을 만났을 때, **When** 1차 컨텍스트(CLAUDE.md)를 참조하면, **Then** 업무 프로세스 문서로 권위 위임이 명시되어 있어 단일 정본을 찾을 수 있다.
4. **Given** 저장소 루트 `README.md`에 접속한 외부 방문자가 있을 때, **When** 상단부를 읽으면, **Then** "1인 BE + 디자이너 + AI 에이전트 하이브리드"의 협업 모델과 업무 프로세스 문서 진입점이 한 문장 이내로 안내된다.

### Edge Cases

- **다크 모드 유지 요청이 접수되는 경우**: 본 스펙 범위에서 수용하지 않는다. 라이트 단독 정책을 근거로 별도 마일스톤·이슈로 등록을 안내한다.
- **기존 커스텀 색상·여백이 새 토큰 체계와 어긋나는 경우**: 토큰 쪽을 정본으로 수렴한다. 해당 커스텀 값은 토큰에 대응 항목을 신설하거나 가장 가까운 기존 토큰에 매핑한다. 임시 인라인 스타일 금지.
- **마이그레이션 중 시각 회귀가 감지되는 경우**: 디자이너 검수용 미리보기 경로에서 전·후 대조가 즉시 가능해야 한다. 회귀가 확인되면 해당 컴포넌트만 별도 브랜치로 분리해 롤백한다.
- **핸드오프 이슈에 필수 필드가 누락된 경우**: 템플릿이 누락 필드를 시각적으로 구분해 디자이너가 제출 전에 인지하도록 한다. 누락 상태 이슈는 개발자가 반려하고 디자이너에게 재작성을 요청한다.
- **Next 16 업그레이드(#249)가 Tailwind v4 전환 이후 머지되는 경우**: 본 스펙은 Next 15 기준으로 완결한다. Next 16 머지 후 회귀 검증은 v2.4.2 트랙 책임이며, 회귀가 발견되면 해당 트랙에서 수정한다.
- **디자인 토큰 원본이 누락된 채 빌드 스크립트가 실행되는 경우**: 토큰 빌드 스크립트는 원본 누락을 빌드 실패로 표시해야 하며, 빈 CSS 변수를 런타임에 허용하지 않는다.
- **WORKFLOW 문서와 기존 CLAUDE.md·DEVELOPMENT.md 내용이 중복·상충하는 경우**: WORKFLOW.md를 단일 정본으로 두고 다른 문서는 권위 위임으로 정리한다. 중복 섹션은 WORKFLOW.md로 이주하거나 요약·링크 형태로 축약한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 프로젝트는 하나의 CSS 토큰 체계(색상·타이포·간격·반경·그림자)를 단일 소스로 노출해야 하며, 앱·신규 컴포넌트·디자이너 산출물이 모두 이 소스를 참조한다.
- **FR-002**: 시스템은 단일 라이트 테마로 운영해야 하며, 다크 모드 분기(`.dark` 클래스, `prefers-color-scheme` 미디어 쿼리 분기)를 코드베이스에서 보유해서는 안 된다.
- **FR-003**: 프로젝트는 신규 UI 구성요소를 표준 명령 한 줄로 추가 가능한 vendored 라이브러리 체계를 갖춰야 하며, 해당 라이브러리는 프로젝트 토큰 체계에 기본 결합된다.
- **FR-004**: 디자인 시스템 도입 전·후의 주요 플로우(로그인·여행 목록·여행 상세·설정) 렌더 결과는 사용자 체감 수준에서 "동등 또는 개선"이어야 하며 의도하지 않은 레이아웃 이동·색상 역전·폰트 치환이 없어야 한다.
- **FR-005**: 마이그레이션 대상 폼 컴포넌트(`ActivityForm`, `AuthButton`, `DeleteTripButton`, `LeaveTripButton`, `InviteButton`, `TodayButton`)는 키보드 내비 가능하고, 포커스 표시가 명확하며, 라벨·입력·오류 메시지가 보조 기술로 연결되어야 한다.
- **FR-006**: 마이그레이션 대상 컴포넌트의 상태(hover/active/disabled/loading)는 디자인 시스템의 공통 상태 규칙을 공유하여 서로 어긋나지 않아야 한다.
- **FR-007**: 저장소는 디자이너가 검수할 수 있는 마이그레이션 대상 컴포넌트 미리보기 경로를 제공해야 하며, 해당 경로는 프로덕션 자산 크기에 영향을 주지 않아야 한다.
- **FR-008**: 디자이너용 토큰 원본(색상·간격·타이포)은 W3C DTCG 호환 형식으로 저장소에 보관되어야 하며, 프로젝트 표준 빌드 스크립트 한 번으로 앱이 참조하는 CSS 토큰 변수를 재생성 가능해야 한다.
- **FR-009**: 디자이너용 토큰 원본이 누락·손상된 상태에서 토큰 빌드를 시도하면 빌드가 실패해야 하며, 런타임에 빈 CSS 변수를 허용해서는 안 된다.
- **FR-010**: 저장소는 GitHub 이슈 생성 UI에서 선택 가능한 "디자이너 핸드오프" 이슈 템플릿을 제공해야 하며, 템플릿은 Figma URL, 스크린샷, variants/states, 인터랙션, 데이터 바인딩 의도, 토큰 변경 여부 필드를 필수로 안내한다.
- **FR-011**: 저장소는 팀 구성·역할, 이슈 흐름, 버전·릴리즈 정책, 디자이너 협업 흐름, AI 에이전트 활용, 마일스톤 운영, 핫픽스 절차의 7개 섹션을 포함하는 업무 프로세스 문서를 단일 파일로 제공해야 한다.
- **FR-012**: 저장소는 디자이너 핸드오프 상세 문서(도구 셋업, 산출물 형식, 처리 절차, 검토 체크포인트)를 별도 파일로 제공하며, 업무 프로세스 문서에서 링크되어야 한다.
- **FR-013**: 저장소의 문서 색인(`docs/README.md`)·AI 에이전트 1차 컨텍스트(`CLAUDE.md`)·루트 `README.md`에는 업무 프로세스 문서로 진입하는 링크가 존재해야 하며, AI 에이전트 1차 컨텍스트에서는 업무 프로세스 문서로의 권위 위임이 명시되어야 한다.
- **FR-014**: 디자이너 핸드오프 파이프라인은 자동 트리거(웹훅·CI 자동 PR 등)를 본 스펙 범위에서 포함하지 않으며, 디자이너 제출 → 개발자 수동 변환 → 개발자 게이트 머지 모델을 따른다.
- **FR-015**: 빌드·타입체크·테스트 파이프라인은 디자인 시스템 전환 전후로 동일한 통과 상태를 유지해야 한다.

### Key Entities *(include if feature involves data)*

- **DesignToken**: 프로젝트의 디자인 변수 원본. 색상·타이포·간격·반경·그림자 카테고리를 가진다. 디자이너가 W3C DTCG 형식으로 편집하며, 빌드 스크립트가 CSS 토큰 변수로 재생성한다. 런타임 저장소 없음.
- **DesignHandoff**: 디자이너가 개발자에게 전달하는 요청 단위. 필드 — Figma URL, 스크린샷, variants/states, 인터랙션, 데이터 바인딩 의도, 토큰 변경 여부. GitHub 이슈 본문으로 저장되며 개별 PR과 1:N 관계.
- **WorkflowDocument**: 업무 프로세스의 단일 정본 문서. 7개 섹션(팀 구성·이슈·릴리즈·디자이너·에이전트·마일스톤·핫픽스) + 디자이너 핸드오프 상세 링크 + CLAUDE.md/루트 README 역참조로 구성.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 디자인 시스템 전환 전·후에 주요 플로우(로그인·여행 목록·여행 상세·설정) 4종의 스크린샷 대조에서 "동등 또는 개선" 판정이 4/4 달성된다.
- **SC-002**: 개발자가 신규 컴포넌트를 프로젝트 표준 명령 한 줄로 추가했을 때, 추가된 컴포넌트가 토큰에 즉시 결합되어 별도 수정 없이 렌더된다(실측 0 수정).
- **SC-003**: 마이그레이션 대상 폼 6종 모두에서 Tab·Shift+Tab으로 모든 포커스 가능 요소를 순회할 수 있으며, 포커스 링이 항상 시각적으로 식별 가능하다(6/6 달성).
- **SC-004**: 디자이너 토큰 원본 예시를 한 번 변경한 뒤 표준 빌드 스크립트를 실행하면, CSS 토큰 변수 파일이 자동 갱신되어 앱 UI에 일관되게 반영된다(수동 편집 0건).
- **SC-005**: GitHub 이슈 생성 UI에서 "디자이너 핸드오프" 템플릿이 기본 옵션 목록에 노출되며, 선택 시 필수 필드 6종이 안내된다.
- **SC-006**: 신규 합류자(개발자 1인 dry-run)가 업무 프로세스 문서만 읽고 이슈 생성 → 브랜치 → PR → 머지의 한 사이클을 외부 질의 없이 완료한다.
- **SC-007**: AI 에이전트 1차 컨텍스트(`CLAUDE.md`)와 저장소 색인(`docs/README.md`), 루트 `README.md` 세 지점 모두에서 업무 프로세스 문서로 1홉 진입할 수 있다.
- **SC-008**: 디자인 시스템 전환 커밋 범위에서 빌드·타입체크·테스트 파이프라인이 전환 전과 동일한 통과 상태를 유지한다(파이프라인 스테이지 실패 0건).
- **SC-009**: 다크 모드 관련 코드(`.dark` 선택자, `prefers-color-scheme` 분기)가 전환 후 코드베이스 전체에서 0건으로 감소한다(정적 검색 기준).
