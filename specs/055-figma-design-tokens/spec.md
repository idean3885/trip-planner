# Feature Specification: Figma 디자인 토큰 전면 정합

**Feature Branch**: `055-figma-design-tokens`
**Created**: 2026-06-03
**Status**: Draft
**Input**: Figma 디자인(Trip Planner)에 등장한 모든 색을 시스템 단일 팔레트로 통일하고, 캘린더 셀/헤더 상태색·타이포·radius·동행 배너를 디자인 정확값으로 재배선. 현재 무채색 shadcn 기본값(oklch grayscale)에서 디자인이 정의한 색 정체성(파랑 포인트 + 여행주말 초록 + 정확 중립 그레이)으로 전환. 라이트 전용 정책 유지.

## Clarifications

1. **색 정본(SSOT)은 디자인 등장 색 전수** — Figma 파일에 실제로 그려진 모든 색(텍스트·배경·테두리·셀 상태)을 시스템 색의 기준으로 삼는다. 무채색 oklch 기본값을 디자인 정확 hex로 교체한다.
2. **다크 전체연도 캘린더·다크/핑크 배경은 스타일 참조용** — 다크 12개월 미니 캘린더와 다크/핑크 배경 스와치는 셀 상태·색·라운드의 참조이며, 새 페이지·새 테마로 구현하지 않는다. 핑크/다크 배경색은 팔레트 토큰으로만 등재(미적용).
3. **라이트 전용 정책 유지** — 다크 모드를 켜지 않는다(기존 정책). `.dark` 블록은 도입하지 않는다.
4. **활동 카테고리 칩은 범위 밖** — 디자인 목업의 활동 카드·카테고리 칩(이동/숙소)은 붙여넣은 현재 앱 스크린샷의 일부로, 신규 디자인 토큰이 아니다. 기존 동작·색 유지.
5. **외형 전환이지 기능 변경 아님** — 데이터 스키마·API·MCP 변경 없음. 색·타이포·radius·캘린더 셀 분기만 바뀐다.
6. **하네스 정합 포함** — 색 정본 위치(`:root`)와 토큰 검증 하네스(audit-tokens·tokens-presence·build-tokens 센티넬)가 신규 색 토큰 체계와 어긋나면 하네스를 이번 변경에 맞춰 정합한다.

## Metatag Conventions *(normative, inherited from PR #204)*

본 피처의 tasks.md·plan.md는 네 종 메타태그로 후속 자동 검증과 연결된다:

- `[artifact: <path>|<path>::<symbol>]` — 산출 파일 식별자(drift 감사 기준)
- `[why: <short-tag>]` — 추적 그룹 키(plan↔tasks 커버리지·이슈 합산)
- `[multi-step: N]` — plan bullet이 다단 작업일 때 최소 매핑 태스크 수(N ≥ 2)
- `[migration-type: schema-only | data-migration]` — 마이그레이션 산출물 구분(본 피처는 마이그레이션 없음)

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 디자인 등장 색 전수 → 시스템 팔레트 통일 (Priority: P1)

디자인에 등장한 모든 색이 시스템 단일 팔레트로 등재되고, 화면 전반의 표면·텍스트·테두리·강조가 그 팔레트(디자인 정확값)로 표시되어야 한다. 무채색 기본값이 디자인 색 정체성으로 일관되게 전환된다.

**Why this priority**: 색 통일이 이 작업의 핵심 가치 — 디자인과 구현의 색 불일치를 0으로 만든다.

**Independent Test**: 주요 화면(여행 상세·목록·설정)을 열어 배경·텍스트·테두리·강조 색이 디자인 팔레트값과 일치하는지, 무채색 잔재가 없는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 디자인에 등장한 색 목록, **When** 시스템 팔레트를 보면, **Then** 모든 색이 빠짐없이 토큰으로 등재되어 있다.
2. **Given** 임의 화면, **When** 배경·텍스트·테두리·강조를 보면, **Then** 디자인 정확값(예: 텍스트 `#121212`, 테두리 `#D9D9D9`, muted 텍스트 `#616161`)으로 표시된다.
3. **Given** 포커스 가능한 요소, **When** 포커스하면, **Then** 포커스 링이 브랜드 블루 `#17A1FA`로 표시된다.

---

### User Story 2 - 캘린더 셀/헤더 상태색 재배선 (Priority: P1)

여행 캘린더의 요일 헤더와 날짜 셀이 디자인의 상태 체계대로 색을 가져야 한다: 토요일=파랑, 일요일=초록, 여행기간 주말=초록 텍스트, 선택=연녹 배경(`#F0FFD7`), 오늘=테두리, 비활성=그레이.

**Why this priority**: 캘린더는 디자인이 가장 정교하게 정의한 표면(Atoms 8변형) — 색 통일의 핵심 적용처.

**Independent Test**: 여행 캘린더를 열어 요일 헤더 색(토=파랑/일=초록/평일=그레이), 여행기간 주말 날짜=초록, 선택 셀=연녹 배경, 오늘=테두리, 기간 밖=그레이가 디자인과 일치하는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 캘린더 요일 헤더, **When** 보면, **Then** 토요일은 파랑, 일요일은 초록, 평일은 그레이다.
2. **Given** 여행기간 안의 주말 날짜, **When** 보면, **Then** 초록 텍스트로 강조된다.
3. **Given** 선택한 날짜, **When** 보면, **Then** 연녹 배경(`#F0FFD7`)으로 표시된다.
4. **Given** 오늘 날짜, **When** 보면, **Then** 테두리 박스로 표시된다.
5. **Given** 여행기간 밖 날짜, **When** 보면, **Then** 그레이로 약하게 표시된다.

---

### User Story 3 - 타이포·radius·동행 배너 정합 (Priority: P2)

본문·헤딩 서체가 Inter 변수로 연결되고, 모서리 라운드가 디자인 기준(8px)으로 정렬되며, 동행 배너가 디자인 색(`#F0FFD7` 배경 / `#2F2F2F` 텍스트)으로 표시되어야 한다.

**Why this priority**: 색 통일 이후 형태·서체의 잔여 불일치를 닫는 마감 작업.

**Independent Test**: 헤딩·본문의 서체가 Inter인지, 카드·버튼 모서리가 디자인 라운드인지, 동행 배너 색이 디자인과 일치하는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 헤딩·본문, **When** 서체를 보면, **Then** Inter가 적용된다.
2. **Given** 카드·버튼, **When** 모서리를 보면, **Then** 디자인 라운드(8px 기준)로 표시된다.
3. **Given** 동행 배너, **When** 보면, **Then** 연녹 배경에 `#2F2F2F` 텍스트로 표시된다.

---

### User Story 4 - 토큰 하네스 정합 (Priority: P2)

신규 색 토큰 체계가 토큰 검증 하네스(존재 검사·정본 감사·빌드 센티넬)와 어긋나지 않고, 색 정본 위치가 명확해야 한다.

**Why this priority**: 정본·검증이 신규 색과 어긋나면 회귀·드리프트를 못 잡는다. 통일을 지속가능하게 만드는 토대.

**Independent Test**: 토큰 빌드·감사·존재 테스트를 돌려 신규 색 토큰이 통과하고, 정본(`design/tokens.json` 설명 또는 `:root`)과 산출물(`@theme`/`:root`)이 일치하는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 신규 색 토큰, **When** 토큰 존재 테스트를 돌리면, **Then** 통과한다.
2. **Given** 정본과 산출물, **When** 감사 스크립트를 돌리면, **Then** 불일치 0으로 통과한다.

---

### Edge Cases

- 무채색 잔재(하드코딩 oklch/회색)가 화면에 남으면 안 된다 — 전부 토큰 경유.
- 라이트 전용 정책 — 다크 토글이 생기면 안 된다.
- 캘린더 선택+오늘이 겹치는 셀 → 선택 우선 표현이 일관돼야 한다.
- 기존 활동 카테고리 칩 색은 변하지 않는다(범위 밖).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 디자인에 등장한 모든 색(white·gray 5단계·black·blue 2종·green 3종·pink)이 **프리미티브 팔레트 토큰**으로 빠짐없이 등재되어야 한다.
- **FR-002**: shadcn 시맨틱 토큰(background·foreground·card·popover·primary·secondary·muted·accent·border·input·ring 등)이 디자인 **정확 hex**로 매핑되어야 한다(무채색 oklch 잔재 제거).
- **FR-003**: 캘린더 요일 헤더·날짜 셀이 디자인 상태 체계(토=파랑/일=초록/여행주말=초록/선택=`#F0FFD7`/오늘=테두리/비활성=그레이)대로 **신규 시맨틱 토큰**을 통해 표시되어야 한다.
- **FR-004**: 본문·헤딩 서체가 **Inter** 변수로 연결되어야 한다.
- **FR-005**: 모서리 라운드 기준이 디자인(8px 기준 + 캘린더 셀 4px)으로 정렬되어야 한다.
- **FR-006**: 동행 배너가 `#F0FFD7` 배경 / `#2F2F2F` 텍스트로 표시되어야 한다.
- **FR-007**: 토큰 검증 하네스(존재 테스트·정본 감사·빌드 센티넬)가 신규 색 토큰 체계와 정합되어 통과해야 한다.
- **FR-008**: 라이트 전용 정책을 유지한다(`.dark` 블록·다크 토글 도입 금지).
- **FR-009**: 본 피처는 데이터 스키마·API·MCP·활동 카테고리 칩을 변경하지 않는다(범위 제외).

### Key Entities *(include if feature involves data)*

해당 없음 — 영속 데이터 스키마 변경 없음. UI 토큰·표현 전용.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 디자인에 등장한 색이 100% 토큰으로 등재되고, 화면의 무채색 잔재가 0이다.
- **SC-002**: 캘린더 헤더·셀의 상태색이 디자인 8변형 정의와 100% 일치한다.
- **SC-003**: 헤딩·본문 서체가 Inter로, 동행 배너가 디자인 색으로 표시된다.
- **SC-004**: 토큰 하네스(존재·감사·빌드·lint·typecheck·vitest 전체)가 통과하고 커버리지 100%를 유지한다.
- **SC-005**: 기존 기능(활동·동기화·예약 상태 등)의 동작이 변하지 않는다(외형만 전환).
