**Feature Branch**: `064-landing-access-flow`
**Created**: 2026-07-14
**Status**: Draft

## Clarifications *(optional, add when drafting raises ambiguity)*

1. **대문(`/`)과 여행 목록(`/trips`)의 역할 분리** — 대문은 "제품이 무엇을 하는지" 보여주는 소개 화면이고, 여행 목록은 "내 여행을 다루는" 작업 화면이다. 로그인 여부와 무관하게 대문은 볼 수 있어야 하며, 여행 목록은 로그인 사용자 전용을 유지한다.
2. **"최초 진입은 대문" 범위** — 도메인 루트(`/`)에 진입했을 때 대문을 보여준다는 의미다. 로그인 사용자가 여행 목록(`/trips`)을 직접 열면 그대로 목록을 보여주며, 대문으로 강제 이동시키지 않는다.
3. **로그인 후 목적지** — 로그인 화면을 거쳐 인증되면 여행 목록으로 이동한다. 목적지가 지정되지 않은 경우에도 대문이 아니라 여행 목록으로 고정한다.
4. **대문의 로그인 상태별 진입 유도** — 대문의 주 행동 유도는 로그인 여부에 따라 달라진다. 로그아웃 상태는 "시작하기(로그인)", 로그인 상태는 "여행 목록으로"를 제공한다.

## Metatag Conventions *(normative, inherited from PR #204)*

본 피처의 tasks.md·plan.md는 네 종 메타태그를 통해 후속 자동 검증과 연결된다:

- `[artifact: <path>|<path>::<symbol>]` — 산출 파일 식별자(drift 감사 기준)
- `[why: <short-tag>]` — 추적 그룹 키(plan↔tasks 커버리지·이슈 합산)
- `[multi-step: N]` — plan bullet이 다단 작업일 때 최소 매핑 태스크 수(N ≥ 2)
- `[migration-type: schema-only | data-migration]` — 마이그레이션 산출물 구분

본 피처는 데이터 스키마 변경이 없어 `migration-type`은 사용하지 않는다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 로그인 사용자도 대문을 본다 (Priority: P1)

로그인한 사용자가 도메인 루트에 진입하거나 상단 로고("우리의 여행")를 누르면 대문을 볼 수 있다. 현재는 로그인 상태면 대문이 즉시 여행 목록으로 리다이렉트되어, 대문을 열 수 없고 로고를 눌러도 목록으로 되돌아온다.

**Why this priority**: 대문 접근 불가와 "로고를 눌러도 목록으로 되돌아옴"이 같은 원인(로그인 시 대문→목록 강제 이동)에서 비롯된다. 이 하나를 풀면 요청의 핵심이 해결되며, 대문의 활용도가 살아난다.

**Independent Test**: 로그인 상태에서 루트로 이동하면 대문이 표시되는지, 여행 목록에서 로고를 누르면 대문으로 이동하는지로 단독 검증된다.

**Acceptance Scenarios**:

1. **Given** 로그인한 사용자, **When** 도메인 루트에 진입, **Then** 여행 목록으로 리다이렉트되지 않고 대문이 표시된다.
2. **Given** 로그인한 사용자가 여행 목록 화면에 있음, **When** 상단 로고("우리의 여행")를 누름, **Then** 대문으로 이동한다.
3. **Given** 로그아웃한 사용자, **When** 도메인 루트에 진입, **Then** 이전과 동일하게 대문이 표시된다.

---

### User Story 2 - 로그인 후 여행 목록으로 이동 (Priority: P1)

사용자가 로그인 화면에서 인증을 마치면 여행 목록으로 이동한다. 특정 페이지로 가기 위해 로그인한 경우 그 페이지로 이동하고, 그렇지 않으면 여행 목록으로 이동한다.

**Why this priority**: US1로 대문이 더 이상 목록으로 리다이렉트하지 않게 되면, 목적지가 지정되지 않은 로그인은 대문에 머무를 위험이 있다. 로그인 직후 사용자의 목표는 자기 여행을 다루는 것이므로 목적지를 여행 목록으로 명시해 플로우를 정립한다.

**Independent Test**: 목적지 없이 로그인 화면에서 인증을 마쳤을 때 여행 목록에 도착하는지로 단독 검증된다.

**Acceptance Scenarios**:

1. **Given** 목적지가 지정되지 않은 로그인, **When** 인증 완료, **Then** 여행 목록으로 이동한다.
2. **Given** 보호된 페이지 접근으로 인해 로그인 화면으로 유도된 경우, **When** 인증 완료, **Then** 원래 접근하려던 페이지로 이동한다.
3. **Given** 이미 로그인한 사용자, **When** 로그인 화면에 진입, **Then** 여행 목록으로 이동한다.

---

### User Story 3 - 로그인 사용자용 대문 진입 유도 (Priority: P2)

이미 로그인한 사용자가 대문을 볼 때, 여행 목록으로 자연스럽게 넘어갈 수 있는 안내(주 행동 유도)를 제공한다. 로그아웃 상태의 "시작하기(로그인)" 유도와 구분된다.

**Why this priority**: US1로 로그인 사용자도 대문에 머무를 수 있게 되면, 대문에서 작업 화면(여행 목록)으로 가는 명확한 다음 걸음이 필요하다. 없으면 로그인 사용자가 대문에서 길을 잃는다.

**Independent Test**: 로그인 상태로 대문을 열었을 때 "여행 목록으로" 유도가 보이고 그것을 누르면 여행 목록으로 이동하는지로 단독 검증된다.

**Acceptance Scenarios**:

1. **Given** 로그인한 사용자가 대문에 있음, **When** 주 행동 유도를 확인, **Then** "여행 목록으로" 성격의 안내가 표시된다.
2. **Given** 로그아웃한 사용자가 대문에 있음, **When** 주 행동 유도를 확인, **Then** "시작하기(로그인)" 성격의 안내가 표시된다.

---

### User Story 4 - 랜딩 중복 "할 수 있다" 섹션 통합 (Priority: P3)

대문에 "할 수 있다"는 취지의 섹션이 둘("이런 걸 할 수 있어요", "할 수 있는 것") 중복되어 있다. 하나로 통합하고 문구를 정리해 같은 말을 두 번 하지 않게 한다.

**Why this priority**: 내용 손실 없이 중복만 제거하는 다듬기다. 접근성·플로우(P1)가 우선이며, 이는 대문 품질을 높이는 후속 정리다.

**Independent Test**: 대문에서 "할 수 있다" 취지의 최상위 섹션이 하나만 남고, 통합 후에도 소개하려던 핵심 가치가 빠지지 않았는지로 단독 검증된다.

**Acceptance Scenarios**:

1. **Given** 대문 방문자, **When** 페이지를 훑음, **Then** "할 수 있다" 취지의 최상위 섹션이 하나만 보인다.
2. **Given** 통합 전 두 섹션이 전하던 핵심 가치, **When** 통합 후 대문 확인, **Then** 핵심 가치가 누락 없이 유지된다.

---

### Edge Cases

- 세션이 만료된 채로 여행 목록 등 보호 페이지에 접근하면? → 기존과 동일하게 로그인 화면으로 유도하고, 인증 후 원래 페이지로 복귀한다(US2-2).
- 로그인 사용자가 대문에서 아무 유도도 누르지 않고 이탈하면? → 대문은 소개 화면이므로 문제 없다. 로고·상단 내비로 언제든 여행 목록에 접근할 수 있다.
- 로그아웃 직후 사용자는? → 대문으로 돌아오며, 주 행동 유도는 "시작하기(로그인)"로 전환된다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 로그인 여부와 무관하게 도메인 루트에서 대문을 표시해야 한다. (로그인 상태의 루트→여행 목록 강제 리다이렉트 제거)
- **FR-002**: 상단 로고("우리의 여행")는 로그인 여부와 무관하게 대문으로 이동해야 한다.
- **FR-003**: 로그인 인증 완료 시, 목적지가 지정된 경우 그 페이지로, 지정되지 않은 경우 여행 목록으로 이동해야 한다.
- **FR-004**: 이미 로그인한 사용자가 로그인 화면에 진입하면 여행 목록으로 이동해야 한다.
- **FR-005**: 여행 목록(`/trips`)은 로그인 사용자 전용 접근을 유지해야 한다. 비로그인 접근은 로그인 화면으로 유도한다.
- **FR-006**: 대문의 주 행동 유도는 로그인 상태에서 "여행 목록으로", 로그아웃 상태에서 "시작하기(로그인)"를 제공해야 한다.
- **FR-007**: 대문의 "할 수 있다" 취지 섹션은 하나로 통합되어야 하며, 통합 전 두 섹션이 전하던 핵심 가치는 누락되지 않아야 한다.
- **FR-008**: 대문은 로그인 사용자가 직접 연 여행 목록(`/trips`)을 대문으로 되돌리지 않아야 한다. ("최초 진입은 대문"은 루트 진입에 한정)

### Key Entities

- 데이터 스키마 변경 없음. 본 피처는 라우팅·표시·카피 수준이며, 로그인 상태(세션 유무)만 참조한다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 로그인 사용자가 루트 진입 또는 로고 클릭 시 100% 대문에 도달한다(여행 목록으로 되돌아오지 않는다).
- **SC-002**: 목적지 없이 로그인한 사용자는 100% 여행 목록에 도착한다.
- **SC-003**: 대문에서 "할 수 있다" 취지의 최상위 섹션은 정확히 1개다.
- **SC-004**: 로그인 사용자가 대문에서 1번의 조작으로 여행 목록에 도달할 수 있다(로고 또는 주 행동 유도).
- **SC-005**: 통합 전 두 섹션이 소개하던 핵심 가치 항목이 통합 후에도 모두 확인된다(누락 0건).

## Assumptions

- 웹/모바일 단일 반응형 디자인을 유지한다(뷰포트별 분기 설계 없음).
- 색은 `globals.css`의 `:root` 팔레트 정본을 따른다.
- 대문 구성(가치 소개·데모·CTA 등)의 전체 구조는 유지하며, 본 피처는 접근 조건·이동 목적지·중복 섹션·상태별 CTA에 한정한다.
- 통합할 두 섹션의 최종 카피·구성은 UX 전문가 서브에이전트 피드백을 반영해 plan 단계에서 확정한다.
