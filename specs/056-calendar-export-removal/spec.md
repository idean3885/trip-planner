# Feature Specification: 외부 캘린더 내보내기 제품 노출 제거 (SSOT 단방향 정립)

**Feature Branch**: `056-calendar-export-removal`
**Created**: 2026-06-04
**Status**: Draft
**Input**: User description: "여행 → 외부 캘린더 내보내기(자동 반영/export) 기능을 제품 흐름에서 제거(스펙 아웃). trip-planner → 외부 캘린더는 단방향 읽기(가져오기)만 유지. SSOT를 trip-planner DB로 일원화. 코드 경로는 보존하되 제품 표면(UI/API/MCP/자동 sync)에서 분리. 이슈 #761, 마일스톤 #57."

## Clarifications *(optional, add when drafting raises ambiguity)*

초안 작성 중 드러난 모호 지점을 결정으로 봉합한다. 각 항목은 WHAT/WHY 수준이며, 구체 구현 수단은 plan 단계에서 결정한다.

1. **정본 방향 결정** — 여행 데이터의 단일 정본은 trip-planner DB다. 외부 캘린더(구글/애플)는 trip-planner가 읽어오는 대상일 뿐, trip-planner가 쓰는 대상이 아니다. WHY: 양쪽이 동시에 정본 역할을 하면서 외부 캘린더에 잘못되거나 중복된 항목이 누적됐고, 외부 캘린더는 제어 불가 영역이라 누적 오염을 trip-planner에서 정리할 수 없다.
2. **코드 보존, 표면만 분리** — 내보내기/자동 반영의 핵심 코드 경로는 삭제하지 않고 제품 표면(사용자 진입점·공개 동작)에서만 비노출·비활성화한다. WHY: 향후 정합성 문제가 해소되면 재도입할 여지를 남긴다. 미사용 데이터 모델·테이블의 물리적 정리는 본 피처 범위 밖이며 별도 후속에서 다룬다.
3. **기존에 내보낸 항목은 방치 + 안내** — 이미 외부 캘린더에 쓰여진 항목은 trip-planner가 더 이상 생성·수정·삭제하지 않고 그대로 둔다. 외부 캘린더는 제어 불가 영역이므로 사용자가 직접 정리한다. 사용자에게 "지금부터 trip-planner는 외부 캘린더에 쓰지 않는다"는 사실과 정리 방법을 안내한다. WHY: 동 시간대 다중 활동을 같은 활동으로 단정할 수 없어 자동 중복 정리가 안전하지 않다.
4. **가져오기 인증·접근은 유지** — 외부 캘린더 → trip-planner 가져오기(import)에 필요한 외부 캘린더 읽기 접근(인증·자격 증명)은 유지한다. 단 "여행용 전용 공유 캘린더 생성"과 "멤버 공유(권한 부여)"처럼 trip-planner가 외부 캘린더를 쓰기 목적으로 다루는 연동 표면은 제거 대상에 포함한다. WHY: 가져오기는 읽기 전용이므로 단방향 정본 원칙과 충돌하지 않는다.
5. **쓰기/동기화 공개 인터페이스는 410 Gone으로 명시 폐지** — 외부 캘린더 쓰기·동기화·연결·멤버 공유에 해당하는 공개 엔드포인트는 제거하지 않고 폐지(410 Gone) 응답으로 남긴다. WHY: 기존 소비자에게 "더 이상 제공되지 않음"을 명확히 전달하고, 레거시 폐지 엔드포인트(`gcal/sync`)가 이미 쓰는 패턴과 일관된다. 이는 API 계약 변경(breaking 성격)이므로 버전 결정에 반영한다. trip MCP는 외부 캘린더 쓰기 도구를 보유하지 않아 별도 폐지 대상이 없다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 활동을 바꿔도 외부 캘린더가 오염되지 않는다 (Priority: P1)

여행 작성자가 여행의 일정(활동)을 추가·수정·삭제하거나 가져온 초안을 확정해도, trip-planner가 외부 캘린더로 자동으로 항목을 쓰지 않는다. 그 결과 외부 캘린더에 잘못되거나 중복된 항목이 새로 누적되지 않는다.

**Why this priority**: 오염의 직접 원인이 활동 변경 시 자동 반영(자동 sync)이다. 이 경로를 끊는 것이 문제 해결의 핵심이며, 이것만 적용해도 신규 오염 누적이 멈춘다.

**Independent Test**: 외부 캘린더가 연동된 상태에서 활동을 추가·수정·삭제하고, 외부 캘린더에 신규/변경/삭제 반영이 발생하지 않음을 확인한다.

**Acceptance Scenarios**:

1. **Given** 외부 캘린더가 연동된 여행, **When** 활동을 추가한다, **Then** 외부 캘린더에 새 항목이 만들어지지 않는다.
2. **Given** 외부 캘린더가 연동된 여행, **When** 활동을 수정하거나 삭제한다, **Then** 외부 캘린더의 기존 항목이 변경·삭제되지 않는다.
3. **Given** 가져온 초안이 있는 여행, **When** 초안을 활동으로 확정한다, **Then** 확정된 활동이 외부 캘린더로 다시 쓰여지지 않는다.

---

### User Story 2 - 외부 캘린더 연동은 "가져오기 전용"으로 보인다 (Priority: P1)

여행 작성자가 여행 상세에서 외부 캘린더 연동 영역을 볼 때, "내보내기/동기화/반영" 같은 쓰기 진입점이 더 이상 노출되지 않는다. 연동은 "외부 캘린더에서 가져오기" 단방향으로만 제시되고, 가져오기 전용임을 안내하는 문구가 보인다.

**Why this priority**: 사용자가 여전히 "동기화" 진입점을 보면 외부 캘린더로 쓰여질 것이라 기대하게 되어 정본 혼란이 재발한다. 표면을 가져오기 전용으로 정리해야 정본 방향이 사용자에게도 일관되게 전달된다.

**Independent Test**: 여행 상세의 캘린더 연동 영역에서 내보내기/동기화/반영 버튼·메뉴가 보이지 않고, 가져오기 진입점과 단방향 안내만 보이는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 여행 상세 화면, **When** 캘린더 연동 영역을 연다, **Then** "내보내기/동기화/다시 반영하기" 등 쓰기 동작 진입점이 보이지 않는다.
2. **Given** 캘린더 연동 영역, **When** 화면을 본다, **Then** "외부 캘린더에서 가져오기"는 보이고, 연동이 가져오기(읽기) 전용임을 알리는 안내 문구가 보인다.

---

### User Story 3 - 가져오기는 그대로 동작한다 (Priority: P2)

여행 작성자가 외부 캘린더에서 일정을 가져오는 흐름(연결된 외부 캘린더 선택 → 가져오기 → 초안 검토 → 확정)은 본 변경 이후에도 이전과 동일하게 동작한다.

**Why this priority**: 가져오기는 유지 대상이다. 내보내기 제거 과정에서 가져오기가 함께 깨지면 사용자 가치가 손상되므로 회귀가 없어야 한다.

**Independent Test**: 외부 캘린더에서 가져오기를 실행해 초안이 생성되고, 초안을 활동으로 확정하는 전체 흐름이 정상 동작하는지 확인한다(단, US1에 따라 확정 후 외부로의 재반영은 없다).

**Acceptance Scenarios**:

1. **Given** 읽기 접근이 가능한 외부 캘린더, **When** 가져오기를 실행한다, **Then** 외부 일정이 초안으로 생성된다.
2. **Given** 가져온 초안, **When** 초안을 활동으로 확정한다, **Then** 활동이 trip-planner에 저장된다.

---

### User Story 4 - 외부 인터페이스에서 쓰기 동작이 사라진다 (Priority: P2)

프로그램적 소비자(API·MCP 도구 사용자)가 외부 캘린더로 쓰기/동기화를 요청하는 동작을 더 이상 사용할 수 없다. 해당 요청은 명확하게 "더 이상 제공되지 않음"으로 응답하거나, 공개 인터페이스 목록에서 노출되지 않는다.

**Why this priority**: 웹 UI만 가리면 API·MCP를 직접 쓰는 경로로 오염이 재발할 수 있다. 외부 인터페이스에서도 쓰기 경로를 일관되게 닫아야 정본 방향이 보장된다.

**Independent Test**: 외부 캘린더 쓰기/동기화 관련 공개 동작을 호출했을 때 더 이상 수행되지 않음(명시적 폐지 응답 또는 미노출)을 확인한다.

**Acceptance Scenarios**:

1. **Given** 외부 캘린더 쓰기/동기화 공개 동작, **When** 이를 호출한다, **Then** 동작이 수행되지 않고 폐지되었음이 명확히 전달된다.
2. **Given** 공개 인터페이스(문서·도구 목록), **When** 외부 캘린더 연동 항목을 본다, **Then** 가져오기 관련 항목만 노출되고 쓰기/동기화 항목은 노출되지 않는다.

---

### User Story 5 - 이미 내보낸 항목은 안내와 함께 사용자가 정리한다 (Priority: P3)

과거에 외부 캘린더로 내보내진 항목이 있는 사용자가, trip-planner가 더 이상 그 항목들을 관리하지 않는다는 사실과 직접 정리하는 방법을 안내받는다.

**Why this priority**: 데이터 손실·오작동은 아니지만, 기존 항목을 방치하면 사용자가 "왜 옛 항목이 남아 있나"라고 혼란을 겪는다. 안내로 기대를 정렬한다. 신규 오염 차단(P1)보다는 후순위다.

**Independent Test**: 과거 내보내기 이력이 있는 상태에서 연동 영역의 안내 문구가 "trip-planner는 더 이상 외부 캘린더에 쓰지 않으며, 기존 항목은 외부 캘린더에서 직접 정리"임을 전달하는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 과거 내보내기 이력이 있는 여행, **When** 캘린더 연동 영역을 본다, **Then** trip-planner가 더 이상 외부 캘린더를 쓰지 않는다는 안내와 정리 방법이 제시된다.
2. **Given** 기존에 내보내진 외부 캘린더 항목, **When** trip-planner에서 활동을 삭제한다, **Then** 외부 캘린더의 해당 항목은 자동 삭제되지 않고 그대로 남는다.

---

### Edge Cases

- 활동을 일괄 삭제하거나 여러 날을 한 번에 삭제할 때도 외부 캘린더로의 자동 반영이 발생하지 않아야 한다.
- 가져오기를 위해 외부 캘린더 읽기 접근(인증)이 필요한 경우, 쓰기 표면을 제거하더라도 읽기 접근은 유지되어 가져오기가 계속 동작해야 한다.
- 멤버가 여러 명인 공유 여행에서, 멤버 추가·역할 변경·탈퇴가 더 이상 외부 캘린더 권한을 자동으로 바꾸지 않아야 한다(쓰기 표면 제거의 일부).
- 이미 연동(공유 캘린더 생성)이 끝난 여행에서, 연동 상태 표시가 가져오기 전용 의미와 모순되지 않아야 한다.
- 과거 내보내기 매핑이 남아 있어도, 그 매핑을 근거로 한 자동 쓰기가 트리거되지 않아야 한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 활동의 추가·수정·삭제(단건·일괄) 시 외부 캘린더로 자동 반영을 수행하지 않아야 한다.
- **FR-002**: 시스템은 가져온 초안을 활동으로 확정할 때 외부 캘린더로 자동 반영을 수행하지 않아야 한다.
- **FR-003**: 여행 작성자는 여행 상세에서 외부 캘린더로의 내보내기/동기화/반영 진입점을 볼 수 없어야 한다.
- **FR-004**: 시스템은 외부 캘린더 연동을 "가져오기(읽기) 전용"으로 제시하고, 단방향임을 알리는 안내 문구를 제공해야 한다.
- **FR-005**: 여행 작성자는 외부 캘린더에서 일정을 가져오는 흐름(가져오기 → 초안 → 확정)을 본 변경 이후에도 동일하게 사용할 수 있어야 한다.
- **FR-006**: 시스템은 외부 캘린더 쓰기·동기화·연결(전용 캘린더 생성)·멤버 공유(권한 부여)에 해당하는 공개 동작을 제품 표면에서 더 이상 제공하지 않아야 한다.
- **FR-007**: 외부 캘린더 쓰기/동기화/연결/멤버 공유 관련 공개 엔드포인트 호출은 폐지(410 Gone) 응답으로 더 이상 수행되지 않음을 명확히 전달해야 한다.
- **FR-008**: 시스템은 가져오기에 필요한 외부 캘린더 읽기 접근(인증·자격 증명)을 유지해야 한다.
- **FR-009**: 시스템은 이미 외부 캘린더에 내보내진 항목을 생성·수정·삭제하지 않고 그대로 두어야 한다.
- **FR-010**: 시스템은 사용자에게 trip-planner가 더 이상 외부 캘린더에 쓰지 않는다는 사실과, 기존 항목을 외부 캘린더에서 직접 정리하는 방법을 안내해야 한다.
- **FR-011**: 내보내기/자동 반영의 핵심 코드 경로는 삭제하지 않고 제품 표면에서 비노출·비활성화 상태로 보존되어야 한다(향후 재도입 여지). 미사용 데이터 모델의 물리적 정리는 본 피처 범위에서 제외한다.
- **FR-012**: 멤버 추가·역할 변경·탈퇴·오너 이관 등 멤버십 변화가 더 이상 외부 캘린더 권한을 자동으로 변경하지 않아야 한다.

### Key Entities *(include if feature involves data)*

- **여행(Trip)**: 단일 정본. 모든 일정·활동의 정본 보관처. 본 변경 이후 외부 캘린더는 이 정본을 보조 복제하지 않는다.
- **활동(Activity)**: 여행 내 일정 항목. 변경되어도 외부 캘린더로 전파되지 않는다.
- **가져오기 초안(ActivityDraft) / 가져오기 실행 기록(ImportRun)**: 외부 → 내부 단방향 가져오기 산출물. 유지 대상.
- **외부 캘린더 연동 링크·이벤트 매핑**: 과거 내보내기 추적용 연결·매핑 데이터. 본 피처에서 신규 생성·갱신을 멈추되 물리적 삭제는 하지 않는다(코드 보존 전제). 자동 쓰기의 근거로 사용되지 않는다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 활동 추가·수정·삭제(단건·일괄) 및 초안 확정 후 외부 캘린더에 발생하는 자동 변경은 0건이다.
- **SC-002**: 여행 상세의 캘린더 연동 영역에 노출되는 외부 캘린더 쓰기/동기화 진입점은 0개다.
- **SC-003**: 외부 캘린더에서 가져오기 → 초안 → 확정 전체 흐름의 성공률이 본 변경 전과 동일하게 유지된다(회귀 0건).
- **SC-004**: 외부 캘린더 쓰기/동기화 공개 동작 호출 시 쓰기가 실행되는 경우가 0건이다.
- **SC-005**: 캘린더 연동 영역에서 사용자가 "연동이 가져오기 전용"임을 안내 문구로 확인할 수 있다(안내 노출 100%).
