# Feature Specification: 레거시 per-user 캘린더 모델 contract 정리

**Feature Branch**: `022-gcal-legacy-contract`
**Created**: 2026-04-23
**Status**: Released (v2.10.0 — 2026-04-23, 매핑 expand. 레거시 테이블 DROP은 후속 contract)
**Input**: User description: "v2.9.0 expand-and-contract 패턴의 C 단계 — 레거시 per-user GCalLink / GCalEventMapping 테이블·코드·API를 제거하고 이벤트 매핑을 공유 캘린더 기반 신규 모델로 이관."

## Background

v2.9.0(spec 019)에서 외부 캘린더 연동을 per-user에서 여행당 1개 공유 모델로 재설계하면서 **expand-and-contract** 패턴을 채택했다.

- **Expand** (v2.9.0 · 완료): 공유 모델 신규 도입, 레거시 모델 병존, 백필 마이그레이션으로 주인 전용 캘린더를 승격.
- **Dual read** (v2.9.1 · 완료): spec 020에서 상태 조회의 레거시 폴백을 제거. 레거시 조회 경로가 실질적으로 죽음.
- **Contract** (본 피처 · v2.10.0): 레거시 테이블·라우트·코드 자체를 제거.

현재 남은 부채:

- 레거시 테이블 2종이 DB에 잔존(`gcal_links`, `gcal_event_mappings`).
- 코드 다수 지점이 여전히 레거시 모델을 참조:
  - v2 공유 캘린더 sync 경로가 이벤트 매핑을 위해 "bridge" 용도로 per-user 모델을 동적 재사용.
  - v1 레거시 API 라우트 3종(status/link/sync)이 제거되지 않고 남아 있어 내부 혼선 원천.
  - 이벤트 매핑 라이브러리(`src/lib/gcal/sync.ts`)가 per-user 매핑 테이블을 직접 조작.
- 데이터 모델에 두 레거시 모델 정의 유지.

본 피처는 "v2.9.0 전환의 미완 뒷정리"이며, 외부 사용자 경험에는 의도적으로 무영향(공유 캘린더 기능 동작 유지)을 목표로 한다.

## Clarifications

1. **용어 정합** — 본 스펙은 [`docs/glossary.md`](../../docs/glossary.md) 정본(주인/호스트/게스트, 여행, 공유 캘린더) 용어를 따른다.
2. **레거시의 정의** — 본 스펙에서 "레거시"는 여행 단위 공유 모델(`TripCalendarLink`) 이전의 per-user 개인 캘린더 연결 모델과 그 이벤트 매핑 테이블을 가리킨다. 공유 모델 자체·API 동작은 본 피처의 수정 대상이 **아니다**.
3. **승격되지 않은 잔재 사용자의 범위** — 백필 대상에서 제외된 per-user 레코드의 사용자는 spec 020·021 전개 과정에서 이미 "공유 캘린더 재연결 필요" 또는 "외부 OAuth 미등록" 상태로 UI에서 안내받았다. 본 피처는 그 사용자에게 **추가 알림을 강제하지 않되**, 릴리즈 노트에 "레거시 데이터 정리" 사실을 포함한다.

### Session 2026-04-23

- Q1 (이벤트 매핑 모델 경로) → **A: 신규 테이블 도입 + 데이터 복사 후 구 테이블 잔존**. 이유: 무중단 배포 요구를 충족하려면 구 테이블·구 모델이 배포 플랫폼 전환의 과도 구간 동안 살아 있어야 하고, 신규 테이블 병존이 가장 자연스럽다. FK 이동(B)은 기존 NOT NULL FK를 단일 단계로 바꾸기 어려워 expand·contract 분리와 맞지 않는다.
- Q2 (레거시 API 종료 정책) → **B: 410 Gone 응답으로 한 릴리즈 유지**. 이유: 구 배포 인스턴스가 새 배포 Ready 직전까지 여전히 라우트를 호출할 수 있다. 즉시 삭제는 500 유발 가능. 410 Gone은 "의도된 종료"로 로깅되고 외부 잔존 호출자를 안전하게 추적한다.
- Q3 (사전 공지) → **A: 릴리즈 노트만**. 이유: 영향 사용자는 spec 020/021 전개에서 이미 UI 안내를 받았다. 중복 공지는 노이즈.
- **릴리즈 분할 결정** — 본 피처는 **v2.10.0 expand 매핑 단계**만 수행한다. 실제 레거시 테이블 drop과 라우트 파일 제거는 후속 피처(v2.11.0+)에서 "완전 contract"로 분리한다. 배포 플랫폼의 과도 구간에서 구 인스턴스가 사라진 테이블·라우트를 참조하지 않도록 보장.

## User Scenarios & Testing *(mandatory)*

### 용어

- **레거시 모델**: 여행 단위 공유 모델 도입 이전의 per-user 개인 캘린더 연결·이벤트 매핑 데이터 구조.
- **정본 모델**: 여행 단위 공유 모델(`TripCalendarLink`) + 그 하위 이벤트 매핑.
- **Contract 단계**: expand-and-contract 패턴의 마지막 단계 — 레거시를 완전히 철거하는 릴리즈.

### User Story 1 — 기존 공유 캘린더 사용자가 동작 변화를 체감하지 않는다 (Priority: P1)

v2.10.0 배포 이후, **이미 공유 캘린더를 연결해 사용 중인 주인·호스트·게스트**는 본 피처로 인해 어떤 동작 변화나 오류도 체감하지 않는다. 공유 캘린더 연결·해제·동행자 구독·sync가 그대로 작동한다.

**Why this priority**: 정본 동작 유지가 본 피처의 존재 조건. 회귀가 발생하면 피처 전체가 무가치.

**Independent Test**: v2.9.2 시점에 공유 캘린더를 정상 연결한 여행을 배포 후 재방문 → 캘린더 섹션 UI 변화 없음 + sync 실행 → 기존 이벤트 매핑이 유지되어 신규 이벤트 생성·업데이트가 정상.

**Acceptance Scenarios**:

1. **Given** 공유 캘린더 연결 상태 여행, **When** 주인이 "다시 반영하기" 실행, **Then** 기존 이벤트가 중복 생성 없이 업데이트되고 이벤트 매핑 일관성 유지.
2. **Given** 공유 캘린더 추가 상태 동행자, **When** 페이지 재방문, **Then** 캘린더 섹션이 기존과 동일하게 "추가됨" 상태 노출.
3. **Given** 공유 캘린더 연결 해제 플로우, **When** 주인이 해제 실행, **Then** 동행자 권한 회수·이벤트 매핑 정리가 기존과 동일하게 수행.

---

### User Story 2 — 레거시 데이터만 남아 있던 여행의 정리 (Priority: P2)

v2.9.0 백필에서 제외돼 per-user 레거시 레코드만 남은 여행은 본 피처로 **DB에서 완전 정리**된다. 해당 여행·사용자는 spec 020·021에서 이미 "공유 캘린더 미연결" / "외부 OAuth 미등록" 상태 안내를 받은 상태이며, 본 피처에서 데이터가 사라져도 UI 상태가 **변하지 않는다**(이미 미연결 상태로 노출 중).

**Why this priority**: 스토리지·마이그레이션 관점에서 완전한 정리가 contract의 목표. 사용자 UI에는 이미 영향이 반영된 상태이므로 본 피처 단독으로 추가 UX 전환이 발생하지 않는다.

**Independent Test**: 백필 누락 샘플(예: 오너가 PRIMARY로 연결했던 여행) 케이스를 준비 → v2.10.0 마이그레이션 실행 → 레거시 레코드 0건 + 여행 페이지 UI가 "공유 캘린더 연결" 상태 그대로 유지.

**Acceptance Scenarios**:

1. **Given** 백필 누락 여행, **When** v2.10.0 마이그레이션 실행, **Then** 해당 여행 관련 레거시 레코드가 전부 제거되고 공유 모델 레코드가 있으면 그대로 유지.
2. **Given** 레거시 데이터만 있던 여행의 사용자가 UI에 진입, **When** 페이지 로드, **Then** 기존 "공유 캘린더 연결 필요" 안내가 그대로 노출(spec 020).

---

### User Story 3 — 내부 코드·API 일관성 회복 (Priority: P2)

개발 관점에서, v2.10.0 배포 이후 **코드베이스에 레거시 모델 참조가 0건**이고, 레거시 API 라우트도 호출 가능 상태가 아니다. 공유 모델만 유일한 경로.

**Why this priority**: 향후 피처 개발 시 레거시 병존 부담 제거. Cross-domain integrity 원칙 강화.

**Independent Test**: 리포에서 레거시 모델 이름 전수 검색 → 테스트·문서·주석 외에는 0건. 레거시 라우트 호출 시 존재하지 않거나 합의된 종료 응답.

**Acceptance Scenarios**:

1. **Given** v2.10.0 병합된 코드베이스, **When** 레거시 per-user 모델 참조를 전수 검색, **Then** 활성 코드 경로에 0건.
2. **Given** 레거시 API 라우트를 직접 호출, **When** 요청 도달, **Then** 라우트가 존재하지 않거나 합의된 종료 응답을 반환한다(Q2 결정에 따라).

---

### Edge Cases

- **마이그레이션 중 공유 모델 sync가 동시 실행**: 마이그레이션은 `배포 파이프라인의 스키마 마이그레이션` 배포 파이프라인 시점에 수행되며, 배포 직후 한 번의 빌드 인스턴스에서 처리된다. 진행 중 sync 호출은 매핑 신규 모델로 쓰이므로 데이터 불일치 없음.
- **외부 계정에 남은 개인 trip-planner 캘린더**: 본 피처는 외부 계정에 물리적으로 존재하는 캘린더를 **건드리지 않는다**. 사용자가 본인 외부 계정에서 직접 정리해야 한다(spec 019 원칙 유지).
- **이벤트 매핑이 0건인 여행**: 기존에 sync를 한 번도 돌리지 않은 공유 캘린더 여행은 매핑 이관 대상이 없음. 마이그레이션이 no-op.
- **하위 호환 클라이언트가 레거시 API를 호출**: 내부 조사로 호출자 0건임을 확인한 전제이며, 외부 호출이 감지되면 즉시 롤백 또는 라우트 부활 옵션을 대기.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 여행 단위 공유 캘린더의 이벤트 매핑을 **공유 모델에 직접 귀속**한다. 본 피처 이후 이벤트 매핑은 per-user 레코드에 의존하지 않는다.
- **FR-002**: 시스템은 기존 공유 캘린더 여행의 **이벤트 매핑을 손실 없이** 신규 귀속 구조로 복사한다(Clarification Q1 — 신규 테이블 도입).
- **FR-003**: 시스템은 v2.10.0 시점에 레거시 테이블(`gcal_links`·`gcal_event_mappings`)과 레거시 데이터 모델을 **남겨둔다**. 실제 drop은 후속 릴리즈(v2.11.0+)에서 "완전 contract" 단계로 분리(Clarification — 무중단 배포 보장).
- **FR-004**: 시스템은 v2.10.0 이후 활성 코드 경로에서 레거시 모델 **쓰기·읽기를 중단**한다. 코드에 직접 참조가 남아 있어도 실행 경로에 포함되지 않아야 한다(bridge 로직 제거 포함).
- **FR-005**: 시스템은 레거시 API 라우트(`/api/trips/<id>/gcal/{link,sync,status}`)를 **410 Gone 응답**으로 교체한다. 파일 자체는 남겨 배포 플랫폼 과도 구간의 구 인스턴스 호출을 안전하게 처리한다. 실제 파일 삭제는 후속 릴리즈(Clarification Q2).
- **FR-006**: 본 피처로 인해 v2.9.2 시점 공유 캘린더의 연결·해제·sync·동행자 구독 동작이 **변경되지 않아야** 한다(US1 회귀 0).
- **FR-007**: 본 피처의 마이그레이션 SQL은 `[migration-type]` 헤더를 명시한다(신규 테이블 생성 + 데이터 복사 → `data-migration`).
- **FR-008**: 사전 공지는 **릴리즈 노트로 충분**하다. 앱 내 토스트·배너·추가 README 공지는 수행하지 않는다(Clarification Q3 — 영향 사용자는 spec 020/021에서 이미 안내).
- **FR-009**: 후속 릴리즈(v2.11.0+)에서 레거시 테이블·API 파일·데이터 모델을 완전히 제거할 **선행 조건**(관찰 기간 1주+ 무장애, 잔존 호출 0건 확인)을 릴리즈 노트에 명시한다.
- **FR-010**: 본 피처의 v2.10.0 빌드가 구 배포 인스턴스와 일정 시간 병존하더라도, 양쪽 모두 **사용자 에러 0건**이어야 한다(배포 과도 구간 안전 조건).

### Key Entities

- **이벤트 매핑(정본)**: 공유 캘린더 기반의 새 귀속 구조. 본 피처에서 형태가 확정된다(Q1에 따라).
- **레거시 per-user 캘린더 연결(제거 대상)**: 본 피처 종료 시 DB에 존재하지 않는다.
- **레거시 이벤트 매핑(제거 대상)**: 본 피처 종료 시 DB에 존재하지 않는다. 기존 데이터는 FR-002에 따라 이관.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: v2.10.0 배포 후, 공유 캘린더 정상 경로(연결·해제·sync·구독)의 자동 테스트 회귀 **0건**.
- **SC-002**: v2.10.0 활성 코드 실행 경로에 레거시 매핑 테이블 쓰기·읽기 호출이 **0건**(bridge 로직 제거 완료). 모델 정의·파일 존재는 허용.
- **SC-003**: 이벤트 매핑 복사 후, 배포 직전 `gcal_event_mappings` 레코드 수 = 배포 직후 `trip_calendar_event_mappings` 레코드 수(손실 0).
- **SC-004**: 레거시 API 라우트 호출 시 **410 Gone** 응답을 돌려주며, 요청이 서버 에러(5xx)로 이어지지 않는다.
- **SC-005**: v2.10.0 배포 과도 구간(새·구 인스턴스 병존)에서 사용자에게 노출되는 5xx 에러율이 **평소 기준선과 동일**(회귀 0).
- **SC-006**: v2.10.0 배포 후 1주 관찰 구간 동안 `gcal_event_mappings` 테이블에 신규 INSERT가 **0건**(DB 쿼리로 확인). 후속 contract 릴리즈의 전제 조건.

## Assumptions

- 외부 공개 호출자가 레거시 API를 쓰지 않는다(내부 조사 완료 전제). 만약 감지되면 별도 롤백 경로를 둔다.
- v2.9.0 백필 이후 공유 모델로 승격된 여행의 이벤트 매핑은 모두 공유 캘린더 ID와 같은 `calendar_id`를 사용해 왔다(spec 019 bridge 로직 관찰). 이관은 단순 소유 외래키 재작성 또는 데이터 복사로 충분.
- 배포 파이프라인이 `배포 파이프라인의 스키마 마이그레이션`만 허용하므로 개발자 로컬 임의 `migrate dev`는 금지된다(CLAUDE.md 정본).
- Contract 릴리즈 시점은 expand(v2.9.0) 배포 이후 충분한 관찰 기간(1주+) 경과 — 2026-04-22~2026-04-23 기준 충족.

## Non-Goals

- 새 provider 연동(#345 별도).
- 공유 캘린더 UI·권한 로직 변경(spec 019/020 범위).
- 외부 계정의 실제 캘린더 삭제(사용자 소유 데이터는 건드리지 않는다).
- 복수 뱃지 UI(spec 023 별도).
- 이벤트 매핑 스키마 근본 재설계(소유 외래키 정리 선에서 멈춘다).
