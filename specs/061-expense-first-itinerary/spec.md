# Feature Specification: 지출 중심 일정 UX 정비

**Feature Branch**: `061-expense-first-itinerary`  
**Created**: 2026-06-23  
**Status**: Draft  
**Input**: User description: "지출 중심 일정 UX 정비 — 모바일 간소화 추가 + 예약상태 제거 + 사전/현장 지출 + 금액 합산."

## Clarifications *(optional, add when drafting raises ambiguity)*

1. **빠른 기록 우선** — 여행 중 일정 추가는 "한 줄 지출 기록"이 1순위다. 시간·세부는 옵셔널이며 강요하지 않는다(옵셔널 강요가 오히려 기록을 막았다). 세부가 필요한 사전 플래닝은 "확장"으로 처리한다.
2. **예약상태 제거** — 예약 "여부"(필수/권장/현장/불필요/예약완료)는 과한 입력이라 제거한다. 예약에서 정작 중요한 정보는 제목·내용·시간이며 이는 기존 필드로 충분히 표현된다. 기존 데이터의 예약상태 값은 정리(이전/폐기)한다.
3. **사전/현장은 "지출 시점"** — 신규 구분은 결제가 사전(여행 전 결제·예약)인지 현장(가서 지불)인지의 2값이다. 예약 "여부"가 아니라 돈을 "언제 쓰는가"다. 현장 지출 금액은 여행이 끝나야 확정된다.
4. **맥락 기반 디폴트** — 여행 시작 전 등록은 사전 결제를 디폴트로, 여행 기간 중(오늘이 여행 일자 범위 내)의 모바일 추가는 현장 결제를 디폴트로 한다. 여행 중 모바일은 주간 달력을 기본 뷰로 한다. 모두 사용자가 수동 변경 가능하다.
5. **가격·통화·시간 모델 승계** — "가격"은 기존 비용·통화 체계를 따른다(통화 필드 유지). 시간은 부동 시간(floating time) 모델 그대로다. "내용"은 자유 메모다.
6. **카테고리 유지** — 활동 분류(관광/식사/이동/숙소/쇼핑/기타)는 그대로 둔다. 이번에 걷어내는 과잉은 예약상태이지 카테고리가 아니다.

## Metatag Conventions *(normative, inherited from PR #204)*

본 피처의 tasks.md·plan.md는 네 종 메타태그를 통해 후속 자동 검증과 연결된다:

- `[artifact: <path>|<path>::<symbol>]` — 산출 파일 식별자(drift 감사 기준)
- `[why: <short-tag>]` — 추적 그룹 키(plan↔tasks 커버리지·이슈 합산)
- `[multi-step: N]` — plan bullet이 다단 작업일 때 최소 매핑 태스크 수(N ≥ 2)
- `[migration-type: schema-only | data-migration]` — 마이그레이션 산출물 구분

형식 검증은 `.specify/scripts/bash/validate-metatag-format.sh`가 수행한다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 여행 중 한 줄로 지출을 빠르게 남긴다 (Priority: P1)

여행 중 무언가를 사 먹거나 사면(맥주·길거리 음식 등), 모바일에서 일정을 추가할 때 **제목·가격·내용** 세 가지만 적고 바로 저장한다. 시간·장소 등은 요구되지 않는다. 적을 게 더 있으면 "확장"을 눌러 한 화면에서 전체를 입력한다.

**Why this priority**: 이 피처의 출발점이자 핵심 가치다. 옵셔널 강요를 걷어내 "지출을 그 자리에서 즉시 남기는" 행동이 실제로 일어나게 한다.

**Independent Test**: 모바일에서 일정 추가 → 제목·가격·내용만 입력 → 저장 성공(시간 미입력 허용). "확장"을 누르면 전체 필드가 한 화면에 나타나는지 확인.

**Acceptance Scenarios**:

1. **Given** 모바일 일정 추가 화면, **When** 제목·가격·내용만 입력하고 저장, **Then** 시간 입력 없이도 활동이 생성된다.
2. **Given** 간소화 추가 화면, **When** "확장"을 누름, **Then** 같은 화면에서 전체 필드(시간·장소·카테고리·사전/현장 등)를 한 번에 입력할 수 있다.
3. **Given** 가격을 입력, **When** 저장, **Then** 그 금액이 활동에 기록되어 합산에 반영된다.

---

### User Story 2 - 예약상태 입력을 걷어낸다 (Priority: P1)

활동 추가·편집에서 예약 "여부"(필수/권장/현장/불필요/예약완료)를 더 이상 요구하지 않는다. 예약 정보가 필요하면 제목·내용·시간으로 적는다.

**Why this priority**: 과한 옵셔널 입력이 기록을 방해한 주원인이다. 제거가 곧 간소화의 핵심이며 US1과 함께 가야 일관된다.

**Independent Test**: 추가·편집·상세 어디에도 예약상태 입력/표시가 없는지 확인. 기존에 예약상태가 있던 활동도 오류 없이 조회·편집된다.

**Acceptance Scenarios**:

1. **Given** 활동 추가/편집/상세 화면, **When** 화면을 본다, **Then** 예약상태 입력 요소가 없다.
2. **Given** 기존에 예약상태 값이 있던 활동, **When** 변경 후 조회·편집, **Then** 회귀·오류 없이 동작한다(예약상태 값은 정리됨).

---

### User Story 3 - 사전 지출과 현장 지출을 구분한다 (Priority: P1)

각 활동(지출)이 **사전 결제**인지 **현장 결제**인지 구분한다. 여행 전 계획 등록은 사전이 디폴트, 여행 중 모바일 추가는 현장이 디폴트이며 사용자가 바꿀 수 있다. 여행 중 모바일은 주간 달력을 기본으로 본다.

**Why this priority**: 합산을 "이미 쓴 돈 vs 앞으로 쓸 돈"으로 의미 있게 나누는 기준이며, 디폴트가 맥락에 맞아야 입력 부담이 0에 가깝다.

**Independent Test**: 여행 시작 전 추가 → 디폴트 사전. 여행 기간 중 모바일 추가 → 디폴트 현장 + 주간 뷰. 두 경우 모두 수동 변경 가능.

**Acceptance Scenarios**:

1. **Given** 오늘이 여행 시작 이전, **When** 활동을 추가, **Then** 지출 시점 디폴트가 사전이다.
2. **Given** 오늘이 여행 기간 내 + 모바일, **When** 일정 화면을 열고 활동을 추가, **Then** 기본 뷰가 주간 달력이고 지출 시점 디폴트가 현장이다.
3. **Given** 어느 경우든, **When** 사용자가 사전/현장을 토글, **Then** 선택대로 저장된다.

---

### User Story 4 - 총액과 일별 금액을 한눈에 본다 (Priority: P2)

여행 화면에서 **여행 총액**과 **일별 금액**을 보고, 사전/현장 **소계**로 이미 쓴 돈과 계획된 돈을 구분해 파악한다.

**Why this priority**: 지출 기록의 보상이자 목적. 합산이 보여야 기록 동기가 유지된다. P1 흐름 위에 얹는 가치라 P2.

**Independent Test**: 여러 날에 금액 있는 활동을 두고 → 여행 총액·각 일자 합계가 정확한지, 사전/현장 소계가 분리되는지 확인.

**Acceptance Scenarios**:

1. **Given** 금액이 있는 활동들, **When** 여행 화면을 본다, **Then** 여행 총액과 선택/표시 일자의 일별 금액이 보인다.
2. **Given** 사전·현장이 섞인 활동들, **When** 합산을 본다, **Then** 사전 소계와 현장 소계가 구분되어 보인다.

### Edge Cases

- 가격을 비워 저장하면(금액 없는 일정), 합산에 0으로 기여하고 정상 저장된다.
- 통화가 섞인 경우, 합산은 통화별로 구분해 표기한다(임의 환산하지 않는다).
- 여행 기간이 설정되지 않은 여행에서는 "여행 중" 판정이 불가하므로 사전을 디폴트로 둔다.
- 데스크탑/큰 화면에서는 간소화 화면이 강제되지 않고 기존 전체 입력 흐름을 쓸 수 있다(모바일 우선이되 데스크탑 회귀 없음).
- 기존 예약상태 값만 있고 다른 정보가 빈약한 활동도 제거 이후 조회·편집에 문제없다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 모바일 일정 추가의 기본 화면은 **제목·가격·내용** 3입력만 노출하고, 시간 등 다른 입력을 요구하지 않아야 한다.
- **FR-002**: 간소화 화면은 **"확장"** 수단을 제공해, 누르면 같은 흐름에서 전체 필드를 한 번에 입력할 수 있어야 한다.
- **FR-003**: 시간 미입력으로도 활동을 저장할 수 있어야 한다(시간은 선택).
- **FR-004**: 시스템은 활동 추가·편집·상세에서 **예약상태 입력·표시를 제거**해야 한다.
- **FR-005**: 기존 활동의 예약상태 데이터를 정리(이전·폐기)하고, 이후 조회·편집이 회귀 없이 동작해야 한다.
- **FR-006**: 활동은 **지출 시점(사전/현장)** 구분을 가져야 한다.
- **FR-007**: 여행 시작 이전 등록은 지출 시점 디폴트가 **사전**, 여행 기간 중(오늘∈여행 일자) 모바일 추가는 디폴트가 **현장**이어야 하며, 사용자가 수동 변경할 수 있어야 한다.
- **FR-008**: 여행 기간 중 모바일에서 일정 화면 기본 뷰는 **주간 달력**이어야 한다.
- **FR-009**: 시스템은 **여행 총액**과 **일별 금액**을 표시해야 한다.
- **FR-010**: 합산은 **사전/현장 소계**를 구분해 보여줘야 하며, 통화가 섞이면 통화별로 표기해야 한다(임의 환산 금지).
- **FR-011**: 카테고리(관광/식사/이동/숙소/쇼핑/기타)는 유지되어야 한다.
- **FR-012**: 데스크탑/기존 조회 흐름은 회귀 없이 동작해야 한다(모바일 우선이되 기존 깨짐 없음).

### Key Entities *(include if feature involves data)*

- **활동(Activity)**: 일정/지출 한 건. 제목·내용(메모)·가격(비용+통화)·카테고리·**지출 시점(사전/현장)**·선택적 시간을 가진다. 기존 **예약상태 속성은 제거**된다.
- **여행(Trip)·일자(Day)**: 합산 단위. 일자별 금액과 여행 총액(및 사전/현장 소계)을 집계한다.
- **지출 시점**: 활동의 사전/현장 2값 구분.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 모바일에서 새 지출 한 건을 **제목·가격·내용만으로 3탭 이내·시간 입력 0회**로 저장할 수 있다.
- **SC-002**: 활동 추가/편집/상세 어디에도 예약상태 입력이 **0건** 남지 않는다.
- **SC-003**: 여행 전 등록은 사전, 여행 중 모바일 추가는 현장이 **별도 설정 없이** 디폴트로 잡힌다(맥락 정확도 100%).
- **SC-004**: 여행 총액·일별 금액·사전/현장 소계가 입력값과 정확히 일치한다(통화별 구분 포함).
- **SC-005**: 기존 활동(예약상태 보유분 포함)이 변경 이후 회귀 없이 조회·편집된다(0 회귀).
