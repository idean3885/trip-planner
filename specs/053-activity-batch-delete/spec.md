# Feature Specification: 활동·일자 다건 삭제 API

**Feature Branch**: `053-activity-batch-delete`
**Created**: 2026-06-03
**Status**: Draft
**Input**: 여러 활동(또는 일자)을 지울 때 단건 삭제를 N번 호출해야 하는 왕복을 줄인다 — 단건 삭제는 유지하고 다건(배치) 삭제를 추가.

## Clarifications

1. **단건 삭제 유지 + 배치는 추가(additive)** — 단건 삭제(`{id}` 단위)는 URI가 단일 리소스를 가리키는 정석·멱등 연산이라 그대로 둔다. 배치는 왕복 절감용 보조 수단으로 추가한다(다건만으로 통일하지 않음). 근거: 단일 리소스 URI·멱등·발견성·에이전트 안전성. 업계 관례(별도 배치 메서드 병존)와도 일치.
2. **부분 성공 허용** — 요청한 식별자 중 일부가 없거나 권한 밖이면 그 항목만 건너뛰고, 삭제된 식별자와 건너뛴 식별자를 결과로 함께 반환한다(전체 요청 실패로 처리하지 않음).
3. **외부 캘린더 반영** — 활동 배치 삭제는 외부 캘린더 자동 반영(spec 049)을 **요청당 한 번** 수행한다(삭제 건마다 반복하지 않음). 일자 배치 삭제도 활동을 함께 제거하므로 동일하게 한 번 반영한다.
4. **범위 제외** — MCP 도구 변경은 범위 밖(차선). BE dry-run/시뮬레이션은 도입하지 않는다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 활동 다건 삭제 (Priority: P1)

API 소비자가 여러 활동을 **한 번의 요청**으로 삭제할 수 있어야 한다. 지금은 활동마다 단건 삭제를 따로 호출해야 해 왕복이 많다.

**Why this priority**: 여러 활동을 정리하는 작업(일정 대량 정비)이 호출 수에 비례해 느려지고 실패 지점이 늘어난다. 가장 잦은 대량 작업이라 최우선.

**Independent Test**: 한 여행의 활동 식별자 여러 개를 한 요청으로 삭제하고, 응답에 삭제된 식별자가 모이며 실제로 사라졌는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 활동 식별자 여러 개, **When** 한 요청으로 배치 삭제하면, **Then** 해당 활동들이 모두 삭제되고 삭제된 식별자 목록이 반환된다.
2. **Given** 일부는 존재하고 일부는 없는 식별자 목록, **When** 배치 삭제하면, **Then** 존재한 것만 삭제되고 없는 식별자는 건너뜀 목록으로 함께 반환된다(전체 실패 아님).
3. **Given** 다른 여행에 속한 활동 식별자, **When** 배치 삭제하면, **Then** 그 식별자는 삭제되지 않고 건너뜀으로 처리된다(여행 경계 보호).
4. **Given** 배치 삭제 완료, **When** 응답 후, **Then** 외부 캘린더 자동 반영이 요청당 한 번 수행된다.

---

### User Story 2 - 일자 다건 삭제 (Priority: P2)

API 소비자가 여러 일자를 **한 번의 요청**으로 삭제할 수 있어야 한다. 일자 삭제는 그 안의 활동을 함께 제거한다.

**Why this priority**: 여행 일정을 크게 갈아엎을 때 일자 단위 대량 삭제가 필요하다(이번 일정 전면 교체에서 일자 15개를 개별 삭제). 활동 배치 다음으로 잦다.

**Independent Test**: 일자 식별자 여러 개를 한 요청으로 삭제하고, 해당 일자와 그 활동이 사라졌는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 일자 식별자 여러 개, **When** 한 요청으로 배치 삭제하면, **Then** 해당 일자와 그 안의 활동이 모두 삭제되고 삭제된 식별자가 반환된다.
2. **Given** 일부 없는 일자 식별자 포함, **When** 배치 삭제하면, **Then** 존재한 것만 삭제되고 나머지는 건너뜀으로 반환된다.

---

### Edge Cases

- 빈 식별자 목록 → 잘못된 요청(거부).
- 모든 식별자가 없거나 다른 여행 소속 → 삭제 0건, 건너뜀 목록만 반환(오류 아님).
- 편집 권한 없는 사용자 → 거부.
- 같은 식별자 중복 입력 → 한 번만 삭제 처리.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 한 요청으로 **여러 활동을 삭제**하게 해야 하며, 삭제된 식별자와 건너뛴 식별자를 결과로 반환해야 한다.
- **FR-002**: 시스템은 한 요청으로 **여러 일자를 삭제**하게 해야 하며(그 안의 활동 포함), 삭제·건너뜀 식별자를 반환해야 한다.
- **FR-003**: 배치 삭제는 **여행 경계로 보호**되어야 한다 — 다른 여행에 속하거나 존재하지 않는 식별자는 삭제하지 않고 건너뜀으로 처리한다.
- **FR-004**: 배치 삭제는 **부분 성공**을 허용해야 한다 — 일부 식별자가 무효여도 나머지는 삭제하고 전체 요청을 실패시키지 않는다.
- **FR-005**: 단건 삭제는 **그대로 유지**되어야 한다(배치로 대체·제거하지 않는다).
- **FR-006**: 빈 식별자 목록은 거부하고, 편집 권한이 없으면 거부해야 한다.
- **FR-007**: 활동을 제거하는 배치 삭제는 외부 캘린더 자동 반영을 **요청당 한 번** 수행해야 한다.
- **FR-008**: 배치 삭제 경로는 **API 문서에 노출**되어야 한다(요청·응답 형식·예시).
- **FR-009**: 본 피처는 MCP 도구 변경과 BE dry-run/시뮬레이션을 포함하지 않는다(범위 제외).

### Key Entities *(include if feature involves data)*

- **활동(Activity)**: 삭제 대상. 식별자로 지정한다. 여행 경계 안에서만 삭제된다.
- **일자(Day)**: 삭제 대상. 삭제 시 그 안의 활동을 함께 제거한다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 활동 N개를 삭제하는 데 필요한 요청 수가 N에서 **1**로 준다.
- **SC-002**: 일자 N개를 삭제하는 데 필요한 요청 수가 N에서 **1**로 준다.
- **SC-003**: 무효 식별자가 섞여도 유효한 식별자는 모두 삭제되고, 결과가 삭제·건너뜀으로 구분돼 반환된다.
- **SC-004**: 단건 삭제 경로는 변경 후에도 그대로 동작한다(회귀 없음).
- **SC-005**: 배치·단건 삭제 경로가 모두 API 문서에 노출된다.
