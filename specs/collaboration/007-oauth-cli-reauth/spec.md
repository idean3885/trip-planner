# Feature Specification: OAuth CLI 인증 + MCP 런타임 재인증

**Feature Branch**: `007-oauth-cli-reauth`  
**Created**: 2026-04-16  
**Status**: Draft  
**Input**: GitHub 이슈 #128, #129 (마일스톤 v2.2: 원클릭 인증)

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 설치 시 브라우저 원클릭 인증 (Priority: P1)

신규 사용자가 install.sh를 실행하면, 터미널에서 토큰을 수동 복사-붙여넣기하지 않고 브라우저에서 Google 로그인 1회만으로 API 토큰이 자동 발급·저장된다.

**Why this priority**: 현재 PAT 설정은 "웹사이트 접속 → 로그인 → 토큰 생성 → 복사 → 터미널 붙여넣기" 5단계가 필요하다. 이를 "브라우저 로그인 1회"로 줄이면 비개발자 온보딩 마찰이 크게 감소한다.

**Independent Test**: install.sh를 PAT 없는 상태에서 실행하고, 브라우저 로그인만으로 로컬 보안 저장소에 토큰이 저장되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 로컬 보안 저장소에 API 토큰이 없는 상태에서, **When** install.sh를 실행하면, **Then** 브라우저가 자동으로 열리고 로그인 페이지가 표시된다.
2. **Given** 브라우저에서 로그인을 완료하면, **When** 서버가 토큰을 생성하고 localhost로 리다이렉트하면, **Then** install.sh가 토큰을 수신하여 로컬 보안 저장소에 자동 저장하고, 브라우저에는 "인증 완료" 메시지가 표시된다.
3. **Given** 브라우저 인증이 120초 내에 완료되지 않으면, **When** 타임아웃이 발생하면, **Then** 기존 수동 토큰 입력 모드로 폴백한다.
4. **Given** 로컬 보안 저장소에 이미 유효한 토큰이 있으면, **When** install.sh를 실행하면, **Then** 인증 단계를 건너뛴다.

---

### User Story 2 - MCP 런타임 자동 재인증 (Priority: P2)

AI 에이전트 환경에서 여행 관리 도구를 사용하는 중 토큰이 만료되면, 사용자에게 브라우저 로그인을 요청하고 새 토큰으로 자동 교체하여 작업을 이어간다.

**Why this priority**: 토큰 만료 시 현재는 "에러 메시지 → 사용자가 웹사이트에서 수동 토큰 재발급 → install.sh 재실행"이라는 번거로운 과정이 필요하다. 자동 재인증으로 작업 중단을 최소화한다.

**Independent Test**: 만료된 토큰으로 MCP 도구를 호출하고, 브라우저 재인증 후 원래 요청이 자동 재시도되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** MCP 도구 호출 시 서버가 인증 만료(401)를 응답하면, **When** 시스템이 브라우저를 열어 재인증을 요청하면, **Then** 사용자가 로그인 완료 후 새 토큰이 로컬 보안 저장소에 갱신되고 원래 요청이 자동으로 재시도된다.
2. **Given** 재인증이 120초 내에 완료되지 않으면, **When** 타임아웃이 발생하면, **Then** 사용자에게 수동 토큰 생성을 안내하는 에러 메시지가 반환된다.
3. **Given** 여러 MCP 도구 호출이 동시에 401을 받으면, **When** 재인증이 시작되면, **Then** 브라우저는 1회만 열리고 모든 대기 중인 요청이 새 토큰으로 재시도된다.

---

### User Story 3 - PAT 미설정 시 초기 인증 (Priority: P3)

install.sh를 건너뛰고 MCP 서버만 설정한 사용자가 처음 도구를 호출하면, 토큰이 없더라도 브라우저 인증으로 자동 발급받을 수 있다.

**Why this priority**: install.sh 없이 수동 설정한 사용자도 동일한 편의를 누릴 수 있어야 한다. P1/P2의 인프라를 재사용하므로 추가 작업이 적다.

**Independent Test**: 로컬 보안 저장소와 환경변수 모두 PAT가 없는 상태에서 MCP 도구를 호출하고, 브라우저 인증으로 토큰이 자동 발급되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 로컬 보안 저장소와 환경변수에 PAT가 없는 상태에서, **When** MCP 도구를 처음 호출하면, **Then** 브라우저가 열려 인증을 요청하고, 완료 후 원래 요청이 정상 처리된다.
2. **Given** 브라우저를 열 수 없는 환경(headless 서버 등)에서, **When** 인증이 실패하면, **Then** 수동 토큰 설정을 안내하는 명확한 에러 메시지가 반환된다.

---

### Edge Cases

- 사용자가 브라우저에서 로그인을 완료했지만 localhost 콜백 수신 전에 터미널을 종료하면 어떻게 되는가? → 생성된 토큰은 DB에 남지만 로컬에 저장되지 않는다. 다음 실행 시 새 토큰이 발급된다.
- 동일 사용자가 브라우저 인증을 반복 실행하면 어떻게 되는가? → 매번 새 토큰이 발급된다. 이전 토큰은 여전히 유효하며, 사용자가 웹 설정에서 관리할 수 있다.
- 인증 콜백의 state가 일치하지 않으면 어떻게 되는가? → CSRF 방어로 콜백이 거부되고, 수동 폴백으로 전환된다.
- localhost 포트가 이미 사용 중이면 어떻게 되는가? → 랜덤 포트(0 바인딩)를 사용하므로 OS가 빈 포트를 할당한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 CLI/MCP 클라이언트가 브라우저 인증을 통해 API 토큰을 자동 발급받을 수 있는 인증 엔드포인트를 제공해야 한다.
- **FR-002**: 인증 엔드포인트는 CSRF 방어를 위한 state 파라미터를 검증해야 한다 (최소 16자).
- **FR-003**: 인증 엔드포인트는 콜백 포트 번호를 검증해야 한다 (1024~65535 범위).
- **FR-004**: 인증 완료 시 토큰은 localhost 콜백으로만 전달되어야 한다 (외부 URL 리다이렉트 불가).
- **FR-005**: install.sh는 브라우저 인증 실패 시 기존 수동 토큰 입력 방식으로 폴백해야 한다.
- **FR-006**: MCP 클라이언트는 API 인증 만료(401) 시 자동으로 브라우저 재인증을 시도해야 한다.
- **FR-007**: MCP 재인증 성공 시 로컬 보안 저장소의 토큰을 갱신하고, 원래 API 요청을 1회 자동 재시도해야 한다.
- **FR-008**: 동시에 여러 요청이 401을 받더라도 브라우저 인증은 1회만 실행되어야 한다.
- **FR-009**: 브라우저 인증의 타임아웃은 120초이며, 초과 시 에러 메시지를 반환해야 한다.
- **FR-010**: 자동 발급된 토큰은 기존 수동 생성 토큰과 동일한 형식 및 권한을 가져야 한다.

### Key Entities

- **Personal Access Token (PAT)**: 사용자 인증용 API 토큰. `tp_` 접두사 + 랜덤 hex. DB에는 SHA256 해시만 저장. CLI/MCP 자동 발급 시 출처 구분을 위한 이름(예: "CLI (자동 로그인)")이 부여됨.
- **인증 콜백 세션**: state(CSRF 토큰) + port(localhost 콜백 포트)로 구성된 일회성 인증 요청. 서버 측 상태를 저장하지 않고 URL 파라미터로만 관리됨.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 신규 사용자가 install.sh 실행 시 토큰 설정을 2분 이내, 클릭 2회 이내(브라우저 열기 + Google 로그인)로 완료할 수 있다. (기존: 5단계, 약 3-5분)
- **SC-002**: MCP 토큰 만료 시 사용자 개입 후 30초 이내에 원래 작업이 자동 재개된다. (기존: 수동 재발급 필요, 5분 이상)
- **SC-003**: 브라우저 인증 실패 시 100% 수동 폴백이 제공되어, 인증 불가 상황이 발생하지 않는다.
- **SC-004**: 동시 401 발생 시에도 브라우저가 1회만 열려, 사용자 혼란이 없다.

## Assumptions

- 대상 환경은 데스크톱 OS로, 시스템 브라우저 열기 명령과 로컬 보안 저장소가 사용 가능하다.
- MCP 서버는 GUI 접근 가능한 데스크톱 환경에서 실행되므로, 브라우저 열기가 가능하다.
- 자동 발급 토큰은 만료 없음(null)으로 생성한다. 사용자가 웹 설정에서 개별 관리할 수 있다.
- 기존 PAT 모델과 인증 로직은 변경 없이 재사용한다.
