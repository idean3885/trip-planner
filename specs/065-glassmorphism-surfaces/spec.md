# Feature Specification: 글래스모피즘 표면 디자인 적용 (크롬+컨테이너)

**Feature Branch**: `065-glassmorphism-surfaces`
**Created**: 2026-07-18
**Status**: Draft
**Input**: User description: "글래스모피즘 디자인 전면 적용. 지금까지 잡아둔 톤과 이미지 유지, 표면만 글래스로."

## Clarifications

1. 적용 범위 — 크롬(헤더·푸터)·오버레이(다이얼로그·팝오버)·컨테이너 카드에만 글래스를 적용한다. 캘린더 셀과 본문 텍스트 표면, 스크롤 목록의 콘텐츠 카드(활동 카드·여행 목록 행)는 성능·가독성을 위해 불투명을 유지한다.
2. 톤 유지 — 기존 팔레트(아이보리+모노크롬+블루 #17A1FA)·브랜드 이미지·캔버스 그라데이션은 그대로 둔다. 색을 바꾸지 않고 표면 처리(반투명·블러·미세 테두리)만 더한다.
3. 색 정본 — 글래스 값은 하드코딩하지 않고 `globals.css :root`의 신규 토큰으로만 표현한다(색상 가드 준수).
4. 폴백 — `backdrop-filter` 미지원 환경에서는 블러 대신 불투명도를 높여 텍스트 가독성을 보존한다.

## Metatag Conventions *(normative, inherited from PR #204)*

본 피처의 tasks.md·plan.md는 네 종 메타태그를 통해 후속 자동 검증과 연결된다:

- `[artifact: <path>|<path>::<symbol>]` — 산출 파일 식별자(drift 감사 기준)
- `[why: <short-tag>]` — 추적 그룹 키(plan↔tasks 커버리지)
- `[multi-step: N]` — plan bullet이 다단 작업일 때 최소 매핑 태스크 수(N ≥ 2)
- `[migration-type: schema-only | data-migration]` — 마이그레이션 산출물 구분(본 피처 해당 없음)

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 크롬·오버레이가 유리 표면으로 보인다 (Priority: P1)

사용자가 앱을 쓰는 동안 헤더·푸터·다이얼로그·드롭다운·셀렉트 표면이 반투명 유리처럼 보이고, 뒤의 파스텔 캔버스가 은은하게 배어 나온다. 색·문구·브랜드 이미지는 그대로다.

**Why this priority**: 크롬·오버레이는 모든 화면에 공통으로 노출되는 표면이라, 여기만 바꿔도 앱 전반이 글래스 디자인으로 읽힌다. 개수가 적고 스크롤 중 겹쳐 쌓이지 않아 성능 위험이 낮다.

**Independent Test**: 다이얼로그·드롭다운·셀렉트를 열고 헤더·푸터를 렌더했을 때 각 표면이 글래스 유틸 클래스를 갖고, 글래스 토큰이 `:root`에 정의되어 있는지로 검증한다.

**Acceptance Scenarios**:

1. **Given** 임의 화면, **When** 헤더·푸터가 렌더되면, **Then** 두 표면이 반투명 배경 + 블러 + 미세 테두리를 가진다.
2. **Given** 다이얼로그/드롭다운/셀렉트, **When** 열리면, **Then** 오버레이 표면이 반투명 + 블러이면서 임의 콘텐츠 위에서도 본문 텍스트 대비 4.5:1 이상을 유지한다.
3. **Given** `backdrop-filter` 미지원 브라우저, **When** 글래스 표면이 렌더되면, **Then** 블러 없이 불투명도를 높인 폴백 배경으로 텍스트 가독성이 유지된다.

---

### User Story 2 - 컨테이너 카드가 유리 패널로 보인다 (Priority: P2)

설정·캘린더 연동·멤버·빈 상태 안내 등 단일 인스턴스 섹션 패널(컨테이너 카드)이 유리 패널로 보인다. 스크롤 목록의 콘텐츠 카드는 불투명을 유지한다.

**Why this priority**: 섹션 패널은 화면당 소수만 있어 성능 위험이 낮고, 유리 패널이 캔버스 위에 떠 보이는 효과가 크다. 반복되는 콘텐츠 카드까지 유리로 만들면 스크롤 중 블러가 겹쳐 성능이 무너지므로 제외한다.

**Independent Test**: `Card`에 글래스 변형을 켠 컨테이너 인스턴스는 글래스 유틸 클래스를 갖고, 글래스 변형을 끈 콘텐츠 카드(활동 카드·목록 행)는 기존 불투명 배경을 유지하는지로 검증한다.

**Acceptance Scenarios**:

1. **Given** 컨테이너 카드(설정·캘린더 패널·멤버·빈 상태), **When** 렌더되면, **Then** 유리 표면(반투명+블러)으로 보인다.
2. **Given** 활동 카드·여행 목록 행, **When** 렌더되면, **Then** 기존 불투명 흰 배경을 유지한다.

---

### Edge Cases

- 오버레이(다이얼로그·드롭다운)가 어두운/복잡한 콘텐츠 위에 열릴 때 — 오버레이는 카드보다 높은 불투명도 토큰을 써서 대비를 보존한다.
- 저사양 기기의 긴 스크롤 — 글래스 표면을 크롬·오버레이·소수 컨테이너로 한정해 동시 블러 개수를 억제한다. 스크롤 다중 콘텐츠 카드는 불투명 유지.
- `prefers-reduced-transparency`/미지원 브라우저 — `@supports` 폴백으로 불투명 배경 제공.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 글래스 표면 값(반투명 배경·블러 반경·테두리·폴백 배경)을 `globals.css :root` 토큰으로만 정의해야 한다(하드코딩 색 금지).
- **FR-002**: 시스템은 재사용 가능한 글래스 유틸(표면용·오버레이용)을 제공하고, 이를 통해서만 표면에 글래스를 적용해야 한다.
- **FR-003**: 헤더·푸터 표면은 글래스(반투명+블러+미세 테두리)로 렌더되어야 한다.
- **FR-004**: 다이얼로그·드롭다운·셀렉트 오버레이 표면은 글래스로 렌더되어야 하며, 카드보다 높은 불투명도를 사용해 임의 콘텐츠 위 대비를 보존해야 한다.
- **FR-005**: `Card`는 글래스 변형을 opt-in으로 제공하고 기본값은 기존 불투명 배경이어야 한다.
- **FR-006**: 단일 인스턴스 컨테이너 카드(설정·캘린더 연동·멤버·빈 상태)에는 글래스 변형을 적용해야 한다.
- **FR-007**: 스크롤 목록의 콘텐츠 카드(활동 카드·여행 목록 행·활동 리스트)와 캘린더 셀·래퍼는 불투명을 유지해야 한다.
- **FR-008**: `backdrop-filter` 미지원 시 `@supports` 폴백으로 불투명도를 높인 배경을 제공해 텍스트 가독성을 보존해야 한다.
- **FR-009**: 글래스 표면 위 본문·보조 텍스트는 WCAG AA(4.5:1) 대비를 유지해야 한다.
- **FR-010**: 기존 팔레트·브랜드 이미지·캔버스 그라데이션·문구는 변경하지 않는다.

### Key Entities

해당 없음 — 영속 데이터 스키마 변경 없음. UI 표면 처리 전용.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 크롬·오버레이 표면(헤더·푸터·다이얼로그·드롭다운·셀렉트) 5종 전부가 글래스 유틸 클래스를 통해 렌더된다.
- **SC-002**: 글래스 표면 위 본문 텍스트 대비가 4.5:1 이상(폴백 배경 기준 포함).
- **SC-003**: 활동 카드·여행 목록·캘린더 셀은 표면 처리 이전과 동일한 불투명 배경을 유지한다(회귀 없음).
- **SC-004**: `backdrop-filter` 미지원 환경에서도 모든 글래스 표면이 불투명 폴백으로 판독 가능하다.
- **SC-005**: 하드코딩 색 도입 0건(색상 가드 통과) — 글래스 값은 전부 `:root` 토큰 경유.
