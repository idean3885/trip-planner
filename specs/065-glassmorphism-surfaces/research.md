# Research: 글래스모피즘 표면 디자인 (크롬+컨테이너)

**Feature**: `065-glassmorphism-surfaces`

## 결정 사항

### 1. 글래스 적용 대상 = 크롬 + 오버레이 + 소수 컨테이너

- **결정**: 헤더·푸터·다이얼로그·드롭다운·셀렉트 + 단일 인스턴스 컨테이너 카드에만 글래스. 스크롤 목록의 콘텐츠 카드·캘린더 셀은 불투명 유지.
- **근거**: `backdrop-filter`는 GPU 부담이 크다. 스크롤 중 다수 표면이 동시에 블러되면 모바일 프레임이 무너진다. 크롬·오버레이는 개수가 적고 겹쳐 쌓이지 않아 안전하다. 이 앱은 여행 상세 모바일 스크롤 회귀 이력이 있어(주간 달력 sticky 등) 콘텐츠 카드 블러를 배제한다.
- **대안**: 전 카드 글래스 → 스크롤 성능 위험으로 기각.

### 2. 두 단계 불투명도 토큰

- **결정**: 캔버스 위 표면은 `--glass-bg`(더 투명), 임의 콘텐츠 위 오버레이는 `--glass-overlay`(더 불투명).
- **근거**: 오버레이는 어둡거나 복잡한 콘텐츠 위에 열릴 수 있어, 표면보다 불투명도를 높여야 본문 텍스트 4.5:1 대비가 유지된다.

### 3. `@supports` 폴백

- **결정**: `@supports not ((backdrop-filter) or (-webkit-backdrop-filter))`로 폴백 배경(`*-fallback`, 불투명도 상향)을 제공.
- **근거**: 블러 미지원 시 반투명만 남으면 텍스트 대비가 무너진다. 폴백은 거의 불투명한 배경으로 가독성을 보존한다.

### 4. 헤더 sticky 미승격

- **결정**: 헤더는 표면 처리만. sticky 승격은 범위 밖.
- **근거**: 여행 상세 모바일이 `sticky top-0 z-20` 주간 달력을 쓴다(`TripDetailLayout.tsx`). 전역 헤더를 sticky로 올리면 offset이 충돌한다. 재설계는 별도 이슈.

## 미해결/후속

- 컨테이너 카드 목록의 완전 열거는 콜사이트 판단이 필요 — 본 피처는 명확한 단일 인스턴스 패널부터 적용하고, 추가 후보는 후속에서 검토.
