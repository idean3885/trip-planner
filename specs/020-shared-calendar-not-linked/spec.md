# Feature Specification: 공유 캘린더 미연결 상태의 역할별 UI

**Feature Branch**: `020-shared-calendar-not-linked`
**Created**: 2026-04-22
**Status**: Draft
**Input**: User description: "공유 캘린더 미연결 상태의 역할별 UI — 주인이 아직 v2.9.0 공유 캘린더를 연결하지 않은 여행에 호스트·게스트가 진입했을 때, 역할별로 명확한 상태 문구와 액션을 제공한다. 증상으로 발견된 status 폴백 → v2 엔드포인트 404 케이스를 정상 시나리오로 재정의."

## Background

v2.9.0(공유 플로우 재설계) 릴리즈 후 신혼여행 여행(tripId=5)에서 발견된 사용자 혼란 시나리오:

- **주인(jamittlee)** 가 아직 공유 캘린더 연결을 시작하지 않은 상태.
- **호스트(idean3885)** 가 여행 페이지에 진입 → 캘린더 섹션에 "내 구글 캘린더에 추가" / "다시 반영하기" 버튼이 노출됨.
- 두 버튼 모두 클릭 시 `POST /api/v2/trips/5/calendar/{subscribe,sync}` 가 **404 `not_linked`** 로 응답.

근본 원인은 "주인이 공유 캘린더를 아직 연결하지 않았다"는 **정상 초기 상태**가 v2.9.0 스펙(spec 019)에서 호스트·게스트 관점의 UI로 다뤄지지 않았던 데에 있다. 상태 조회 응답이 본인의 레거시 per-user 링크를 근거로 "연결됨"으로 신호한 것은 스펙 공백을 임시 메운 부작용이었다.

본 피처의 목적은 이 시나리오를 **"마이그레이션/레거시 버그"가 아닌 정상 시나리오**로 재정의하고, 역할별로 명확한 UI·상태 전달·CTA를 규정하여 스펙 공백을 메우는 것이다.

## Clarifications

1. **용어** — 본 피처가 다루는 상태의 공식 표기는 **"공유 캘린더 미연결 상태"**(약식 "미연결 상태"). 이유: "공석"은 자리비움 뉘앙스라 주인/멤버 부재로 오독 여지. 실제는 캘린더 자체가 아직 만들어지지 않은 상태.
2. **주인 식별자 노출** — 미연결 안내 문구에서 주인 이메일·표시명을 **별도로 재노출하지 않는다**. 여행 페이지 동행자 목록에 이미 주인 식별자가 노출되므로 안내 문구는 "동행자 목록에서 주인에게 요청하세요" 수준으로 간접 지시한다. 이유: 프로젝트 차원의 이메일 노출 정책이 아직 미수립된 상태에서 새 노출 면을 만들지 않는다. 동행자 목록 노출 정책이 명시화되면 재검토.
3. **호스트·게스트의 미연결 탈출 보조 액션** — 제공하지 않는다. 이유: 현 시점 앱 내 알림 채널(카카오 알림톡/이메일/Dooray 등) 부재. 알림 채널이 도입되면 "주인에게 연결 요청 전송"을 후속 피처로 재검토(Future note).
4. **역할 용어 정본** — 본 스펙의 역할 호칭은 [`docs/glossary.md`](../../docs/glossary.md)의 정본(주인/호스트/게스트)을 따른다. 주인은 호스트 권한을 **포함하는 상위 역할**이다(데이터 모델은 상호 배타). 동행자 목록에서 주인을 "주인 + 호스트" 복수 뱃지로 표시할지는 본 피처 범위 밖(spec 021 예정).

### Session 2026-04-22

- Q: 미연결 상태에서 호스트·게스트 UI의 가시성 수준은? → A: 주인의 "공유 캘린더 연결" 트리거 버튼과 **동일한 위치·크기의 버튼**을 호스트·게스트에게도 노출. 클릭 시 열리는 다이얼로그 내부는 **미연결 상태 안내문**과 **"닫기" 버튼 하나만** 포함하며, 공유 캘린더 조작 버튼(추가/다시 반영하기/해제)은 다이얼로그에 포함하지 않는다.
- Q: SC-001의 측정 기준은? → A: **서버 4xx 로그**를 정본으로 사용한다. `POST /api/v2/trips/*/calendar/{subscribe,sync}`의 404 응답 비율을 운영 서버 로그에서 관찰한다. 별도 클라이언트 telemetry는 구축하지 않는다. (로그 조회 수단 등 운영 도구는 plan.md에서 결정.)
- Q: 해제 후 미연결 vs 생성 직후 미연결의 UI는? → A: **완전히 동일**하게 처리한다. "미연결 판정 = `TripCalendarLink` 부재" 원칙을 충실히 따르고, 해제 직후 피드백은 기존 해제 토스트가 담당한다. 별도 세션 플래그·한시적 차별화는 두지 않는다.

## Metatag Conventions *(normative, inherited from PR #204)*

본 피처의 tasks.md·plan.md는 네 종 메타태그를 통해 후속 자동 검증과 연결된다:

- `[artifact: <path>|<path>::<symbol>]` — 산출 파일 식별자(drift 감사 기준)
- `[why: <short-tag>]` — 추적 그룹 키(plan↔tasks 커버리지·이슈 합산)
- `[multi-step: N]` — plan bullet이 다단 작업일 때 최소 매핑 태스크 수(N ≥ 2)
- `[migration-type: schema-only | data-migration]` — 마이그레이션 산출물 구분

형식 검증은 `.specify/scripts/bash/validate-metatag-format.sh`가 수행한다. 의미 검증은 각 US의 validator가 담당한다.

## User Scenarios & Testing *(mandatory)*

### 용어

- **미연결 상태**: 해당 여행에 대해 주인이 아직 공유 캘린더(`TripCalendarLink`)를 연결하지 않은 정상 초기 상태. 여행 생성 직후, 혹은 해제 후 재연결 전 구간.
- **연결됨 상태**: `TripCalendarLink`가 존재하는 상태. 본 피처의 대칭 상태로 참조만 한다.

### User Story 1 — 미연결 여행에 호스트 진입 (Priority: P1)

호스트가 미연결 상태 여행 페이지에 들어왔을 때, 의미 없는 버튼이나 오류 알림 없이 **"주인이 공유 캘린더를 연결하면 이용 가능"** 이라는 상태를 명확히 전달받고, 앱 내 일정 조회·편집은 정상적으로 수행할 수 있다.

**Why this priority**: 현재 프로덕션(trip.idean.me)에서 실제로 오류가 발생 중이며 사용자 혼란이 가장 큰 경로. 본 피처가 실질적으로 해결해야 하는 최우선 증상.

**Independent Test**: 미연결 상태 여행 한 건과 호스트 계정을 준비 → 호스트로 여행 페이지 진입 → 캘린더 섹션에 대기 안내만 노출되고 조작 불가한 상태임을 확인 + 캘린더 관련 오류 알림이 발생하지 않음을 확인.

**Acceptance Scenarios**:

1. **Given** 미연결 상태 여행에 호스트가 진입, **When** 여행 페이지 로드, **Then** 캘린더 섹션에 대기 안내(의미: "주인이 공유 캘린더를 아직 연결하지 않았습니다")가 노출되고 추가/재반영/해제 등 액션 버튼은 보이지 않는다.
2. **Given** 미연결 상태 여행에 호스트가 진입, **When** 페이지 내 어떤 조작을 하더라도, **Then** 캘린더 관련 오류 알림은 발생하지 않는다.
3. **Given** 미연결 상태 여행에 호스트가 진입, **When** 여행의 일정(Day/Activity)을 편집, **Then** 캘린더 미연결 상태와 무관하게 편집은 정상 동작한다.

---

### User Story 2 — 미연결 여행에 주인 진입 (Priority: P1)

주인이 미연결 상태 여행 페이지에 들어왔을 때, 공유 캘린더를 연결하는 **단일 CTA**가 명확히 노출되고, 클릭 → 외부 동의 완료 후 미연결이 해소된다.

**Why this priority**: 미연결을 탈출할 수 있는 유일한 주체는 주인. 주인의 연결 동선이 명확해야 호스트·게스트의 대기가 끝난다. US1과 동시에 갖춰야 피처가 의미를 갖는다.

**Independent Test**: 미연결 상태 여행에 주인이 진입 → "공유 캘린더 연결" CTA 노출 → 클릭 → 연결이 완료되어 여행의 `TripCalendarLink`가 생성되고 현재 멤버에게 권한이 자동 부여됨을 확인.

**Acceptance Scenarios**:

1. **Given** 미연결 상태 여행에 주인 진입, **When** 페이지 로드, **Then** 캘린더 섹션에 단일 **"공유 캘린더 연결"** CTA만 노출된다(레거시 "업그레이드" 분기 없음).
2. **Given** 주인이 CTA 클릭 + 외부 동의 완료, **When** 연결이 성공, **Then** 여행에 공유 캘린더가 생성되고 현재 호스트·게스트에게 권한이 부여된 상태로 전환된다.

---

### User Story 3 — 미연결 여행에 게스트 진입 (Priority: P2)

게스트가 미연결 상태 여행 페이지에 들어왔을 때, 호스트와 **동일한 대기 안내**를 보이되 본인이 취할 수 있는 액션 문구 없이 간결하게 표시된다.

**Why this priority**: 게스트는 본래 캘린더 쓰기 권한이 없어 관여도가 낮으나, 상태 UI 일관성이 신뢰감을 결정. P1 구현에 부록처럼 달린 정합성 항목이라 P2.

**Independent Test**: 게스트 계정으로 미연결 상태 여행 진입 → 캘린더 섹션이 호스트와 의미상 동일한 대기 안내를 보이되, "다시 반영" 등 편집 뉘앙스 표현이 포함되지 않음을 확인.

**Acceptance Scenarios**:

1. **Given** 미연결 상태 여행에 게스트 진입, **When** 페이지 로드, **Then** 캘린더 섹션은 대기 안내만 노출하며 게스트가 조작 가능한 액션은 없다.

---

### User Story 4 — 연결 해소 후 상태 자연 전환 (Priority: P2)

주인이 공유 캘린더를 연결해 미연결이 해소된 뒤, 호스트·게스트가 여행 페이지를 **재방문하거나 새로고침**하면 자동으로 "내 구글 캘린더에 추가"가 가능한 상태로 전환된다(실시간 push는 요구하지 않음).

**Why this priority**: 상태 전환의 연속성. 없으면 호스트·게스트가 수동으로 "다시 로그인" 같은 해법을 찾게 된다.

**Independent Test**: 미연결 상태에서 주인 연결 완료 후, 호스트 계정으로 여행 페이지 새로고침 → 캘린더 섹션이 연결됨 UI로 바뀌고 "내 구글 캘린더에 추가" CTA가 활성화됨을 확인.

**Acceptance Scenarios**:

1. **Given** 미연결 상태가 주인 연결로 해소됨, **When** 호스트 또는 게스트가 여행 페이지를 다시 열거나 새로고침, **Then** 캘린더 섹션이 연결됨 상태로 전환되어 역할에 맞는 액션이 노출된다.

---

### Edge Cases

- **호스트·게스트가 v2.8.0 시점에 남긴 per-user 연결**: 본 피처에서는 UI상 미연결 상태와 **동일하게 취급**한다. 개인 per-user 링크는 v2.9.0 이후 정본이 아니므로 미연결 UI 문구·동작에 반영되지 않는다. (데이터 정리는 본 피처 범위 밖, 별도 건.)
- **주인 연결 시도가 실패하여 일시적 미연결 유지**: 주인에게는 기존 오류 토스트로 알림. 호스트·게스트는 여전히 미연결 UI로 본다.
- **주인 외부 권한 REVOKED 상태**: 이전에 연결됐던 이력이 있어 `TripCalendarLink`가 존재하므로 미연결이 아님. 별도 플로우(권한 회수/재연결)에서 처리되며 본 피처 범위 밖.
- **주인 role 이관 직후 미연결 회귀**: 신 주인 기준 `TripCalendarLink`가 없는 경우 본 피처의 미연결 UI가 그대로 적용된다(spec 019의 주인 이관 규정과 합치).
- **게스트가 주인 이메일을 여전히 여행 멤버 목록(동행자)에서 볼 수 있음**: 멤버 목록 노출은 기존 피처. 본 피처의 캘린더 섹션 안내에서는 주인 식별자를 재노출하지 않는다(Clarification 2).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 미연결 상태 여행에서 호스트·게스트 UI의 캘린더 섹션에 **주인의 "공유 캘린더 연결" 트리거와 동일한 위치·크기의 버튼**을 노출한다. 다만 이 버튼은 미연결 안내 전용이며, 공유 캘린더 조작(추가/다시 반영하기/해제)에 해당하는 어떤 액션도 발생시키지 않는다(Clarification Session 2026-04-22).
- **FR-002**: 시스템은 미연결 상태 여행에서 호스트·게스트가 FR-001의 버튼을 클릭하면 열리는 다이얼로그에 **미연결 상태 안내문**과 **"닫기" 버튼 하나만** 노출한다. 공유 캘린더 조작 버튼(추가/다시 반영하기/해제)은 포함하지 않는다.
- **FR-003**: 시스템은 미연결 상태 여행에서 **주인에게 단일 CTA("공유 캘린더 연결")** 를 노출하며, 대체 분기(업그레이드/레거시 안내 등)를 제시하지 않는다.
- **FR-004**: 시스템은 미연결 상태 여행에서 호스트·게스트의 어떤 조작으로도 **공유 캘린더 조작 엔드포인트 호출이 발생하지 않도록** 보장해야 한다.
- **FR-005**: 시스템의 캘린더 상태 조회 응답은 **미연결 상태를 명확히 식별 가능한 형태**로 전달해야 한다(구체 스키마는 plan에서 결정). 호스트·게스트의 per-user 레거시 링크 존재 여부는 미연결 판정에 영향을 주지 않는다.
- **FR-006**: 미연결이 주인의 연결로 해소된 후, 호스트·게스트가 **여행 페이지를 재방문하거나 새로고침**하면 UI가 연결됨 상태로 자동 전환되어야 한다(실시간 push 요구 없음).
- **FR-007**: 미연결 상태 UI 문구는 **역할별로 구분**되어야 한다 — 주인은 연결 CTA를 동반한 문구, 호스트·게스트는 행동 가능성을 암시하지 않는 대기 문구.
- **FR-008**: 호스트·게스트 미연결 안내 문구에는 **주인의 이메일·표시명을 재노출하지 않는다**. 대신 동행자 목록을 통해 주인에게 요청하도록 간접 지시한다(Clarification 2).
- **FR-009**: 호스트·게스트에게 **미연결 탈출 보조 액션(알림 전송 등)을 제공하지 않는다**. 앱 내 알림 채널이 도입되면 별도 피처로 재검토한다(Clarification 3).
- **FR-010**: 미연결 UI는 앱 내 일정(Day/Activity) 편집·조회 동작과 **상호 독립적**이어야 한다(캘린더 섹션 이슈가 일정 화면을 막지 않는다).

### Key Entities

- **공유 캘린더 연결(TripCalendarLink)**: 여행당 최대 1개. 본 피처는 "부재 = 미연결" 판정에 참조만 하고 신규 속성 추가는 없다.
- **캘린더 상태 응답**: 사용자 관점에서 "이 여행의 캘린더 상태가 내 화면에서 어떻게 보여야 하는가"의 원천. 미연결을 식별 가능한 형태로 표현해야 한다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 배포 후 1주간 운영 서버 로그에서 `POST /api/v2/trips/*/calendar/{subscribe,sync}`의 **404 응답 비율 = 0%** (Clarification Session 2026-04-22 — 서버 로그 정본).
- **SC-002**: 미연결 상태 여행에 진입한 호스트·게스트의 **100%가 "주인 미연결"이라는 상태를 안내 문구만으로 이해**한다(5인 정성 테스트에서 5/5 응답).
- **SC-003**: 주인이 "공유 캘린더 연결" CTA를 누른 뒤 **한 번의 호스트 새로고침만으로** UI가 연결됨 상태로 전환되는 것을 시연 가능.
- **SC-004**: 미연결 상태에서 세 역할(주인·호스트·게스트)의 캘린더 섹션 UI가 **시각적으로 서로 다른 영역·문구**를 갖는다(스크린샷 3종 비교로 검증).
- **SC-005**: 미연결 상태 UI가 존재하는 동안에도 앱 내 일정 편집·조회 완료율은 **연결됨 여행과 동일**(회귀 0).

## Assumptions

- v2.9.0 공유 캘린더 모델(`TripCalendarLink`)은 확정. 본 피처는 조회·렌더링만 다루며 신규 엔티티 도입은 없다.
- "미연결 판정"은 `TripCalendarLink` 부재로 결정한다(v2.9.0 스펙과 합치).
- 본 피처는 v1 레거시 API(`/api/trips/[id]/gcal/*`) 경로의 동작을 변경하지 않는다. 레거시 정리는 expand-and-contract의 contract 릴리즈에서 별도 처리한다.
- v2.8.0 per-user `GCalLink`가 남아있는 호스트·게스트는 미연결 UI에서 동일하게 취급되며, 자신의 남은 per-user 링크 정리는 별도 건으로 다룬다(본 피처 범위 밖).
- 실시간 상태 전환(주인 연결 즉시 호스트 화면이 자동 갱신)은 요구하지 않는다. 새로고침/재방문 기준으로 충분하다.
- 여행 페이지 동행자 목록은 현재 관행대로 주인 이메일·표시명을 노출하며, 본 피처는 이 노출 정책을 변경하지 않는다.

## Non-Goals

- v2.8.0 → v2.9.0 데이터 마이그레이션 보강(별도 건).
- v1 레거시 API 동작 변경(expand-and-contract의 contract 단계에서 별도 처리).
- 주인 역할 이관 플로우 보강(spec 019 범위).
- 외부 권한 REVOKED 후 재연결 플로우 보강(별도 플로우).
- 미연결 상태를 실시간 push 채널로 즉시 해소하는 기술 도입(새로고침 기반으로 충분).
- 동행자/주인 이메일 노출 정책 재수립(별도 정책 수립 필요).
